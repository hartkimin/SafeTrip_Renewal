## §1. 목적 및 적용 범위

### 1.1 목적

본 문서는 SafeTrip Flutter 앱(`safetrip-mobile`)의 프로젝트 디렉토리 구조, 파일 네이밍 규칙, 코드 아키텍처 패턴, 상태 관리 방식, 라우팅 구조, 환경 설정, 테스트 전략을 비즈니스 원칙 v5.1 기준으로 정의한다.

개발팀 신규 멤버가 본 문서 단독으로 프로젝트 구조를 파악할 수 있도록 하며, 코드 기여 시 일관된 표준을 준수할 수 있도록 한다.

### 1.2 적용 범위

| 대상 | 포함 여부 |
|------|----------|
| `safetrip-mobile/` (Flutter 앱) | ✅ 본 문서의 주 대상 |
| `safetrip-server-api/` (Node.js 백엔드) | △ 참조 (연동 관계만) |
| `scripts/` (개발 도구) | △ 참조 (ngrok, 테스트 스크립트) |
| Firebase 프로젝트 설정 | △ 환경 설정 섹션에서 참조 |

### 1.3 상위 원칙 참조

- **비즈니스 원칙 v5.1** (`Master_docs/01_T1_SafeTrip_비즈니스_원칙_v5.1.md`): 역할 체계, 과금 모델, AI 기능, 프라이버시 등급 정의
- **아키텍처 구조 원칙 (#08)**: 시스템 아키텍처, Firebase 구성, 인프라 레이아웃
- **DB 설계 v3.4 (#07)**: PostgreSQL 54개 테이블 정의 (SSoT)

## §2. 아키텍처 개요

### 2.1 채택 패턴: Service-Manager-Screen

SafeTrip Flutter 앱은 **Service-Manager-Screen** 패턴을 채택하고 있다. Clean Architecture(Domain/Data/Presentation 3계층 완전 분리)는 현재 미채택 상태이며, v5.1 신규 기능(과금·AI) 모듈 추가 시점에 부분 도입을 검토한다.

> **주의**: 지침 문서(#34 작성 프롬프트)에는 Clean Architecture 적용이 제안되어 있으나, 실제 현재 코드베이스는 Service-Manager-Screen 패턴으로 구현되어 있다. 본 문서는 **실제 구조**를 기준으로 작성하며, 향후 마이그레이션 방향은 §9 구현 우선순위에서 다룬다. 현재 `core/`, `features/`, `shared/` 디렉토리 스캐폴드가 생성되어 있으며, `features/trip/providers/`에 일부 프로바이더가 구현되어 점진적 마이그레이션이 진행 중이다.

### 2.2 레이어 구조

```
┌─────────────────────────────────────────────────────┐
│                   screens/ (UI Layer)                │
│  StatefulWidget + setState — 화면별 로컬 상태 관리    │
│  GoRouter 라우팅 — 선언적 화면 전환                   │
└───────────────────┬─────────────────────────────────┘
                    │ 의존 (단방향)
┌───────────────────▼─────────────────────────────────┐
│       services/ + managers/ (Business Layer)         │
│  services/  — 비즈니스 로직, REST API, Firebase 호출 │
│  managers/  — 복잡한 상태 관리 (위치, 마커, 세션)    │
└───────────────────┬─────────────────────────────────┘
                    │ 의존 (단방향)
┌───────────────────▼─────────────────────────────────┐
│              models/ (Data Layer)                    │
│  데이터 모델 (DTO + Entity 혼용)                     │
│  sqflite 로컬 DB 캐시                                │
└─────────────────────────────────────────────────────┘
```

### 2.3 레이어 간 의존성 규칙

| 규칙 | 내용 |
|------|------|
| `screens/` → `services/` | 허용 (직접 호출) |
| `screens/` → `managers/` | 허용 (직접 호출) |
| `screens/` → `models/` | 허용 (데이터 표시용) |
| `services/` → `models/` | 허용 |
| `services/` → `screens/` | 금지 (역방향 의존 금지) |
| `managers/` → `services/` | 허용 |
| `widgets/` → `models/` | 허용 (표시용 데이터만) |
| `widgets/` → `services/` | 원칙적 금지 (부모 Screen을 통해 전달) |

## §6. 상태 관리

### 6.1 패턴 개요

SafeTrip 앱은 외부 상태 관리 라이브러리(Riverpod, Bloc, GetX) 없이 Flutter 기본 `StatefulWidget` + `ChangeNotifier`를 사용한다.

**경량 패턴 채택 이유**: 현재 앱 규모에서 외부 라이브러리 도입 없이도 충분히 관리 가능하며, 팀 내 Flutter 숙련도 격차를 최소화하기 위해 기본 패턴을 유지한다.

### 6.2 전역 상태

| 클래스 | 위치 | 관리 대상 |
|-------|------|---------|
| `AuthNotifier` | `router/auth_notifier.dart` | Firebase Auth 인증 상태 (로그인/로그아웃) |
| `SessionManager` | `managers/session_manager.dart` | 현재 여행 세션 정보 (trip_id, 역할, 멤버 목록) |
| `UserModeManager` | `managers/user_mode_manager.dart` | 현재 사용자 역할 모드 (캡틴/크루장/크루/가디언) |
| `MarkerManager` | `managers/marker_manager.dart` | 지도 마커 전체 목록 |
| `FirebaseLocationManager` | `managers/firebase_location_manager.dart` | RTDB 실시간 위치 스트림 |

### 6.3 로컬 상태

각 `Screen`의 `State` 클래스 내 `setState`로 관리하며, 해당 화면에서만 사용되는 데이터(로딩 상태, 폼 입력값, 탭 인덱스 등)가 해당된다.

### 6.4 상태 관리 규칙

| 규칙 | 내용 |
|------|------|
| 전역 상태 추가 기준 | 2개 이상의 화면에서 공유되는 데이터 |
| 로컬 상태 사용 기준 | 단일 화면 내에서만 사용되는 임시 데이터 |
| ChangeNotifier 상속 | 전역 상태 클래스만 적용 |
| notifyListeners() 호출 | 상태 변경 완료 후 1회 호출 |

## §8. 환경 설정

### 8.1 환경 변수 관리

`flutter_dotenv` 패키지를 사용하여 `.env` 파일로 환경 변수를 관리한다.

| 환경 파일 | 용도 | 적용 방법 |
|---------|------|---------|
| `.env` | 기본 환경 (prod) | `flutter run` |
| `.env.local` | 로컬 개발 (에뮬레이터) | 수동 교체 또는 플레이버 |

**주요 환경 변수 예시:**

```
API_SERVER_URL=http://10.0.2.2:3001
USE_FIREBASE_EMULATOR=true
FIREBASE_EMULATOR_HOST=10.0.2.2
```

### 8.2 Firebase 에뮬레이터 설정

`config/firebase_emulator_config.dart`에서 Firebase 에뮬레이터 연결을 관리한다.

- Firebase Auth, Realtime Database, Storage를 로컬 에뮬레이터로 연결
- `USE_FIREBASE_EMULATOR=true` 환경 변수 확인 후 에뮬레이터 연결 활성화
- 에뮬레이터 호스트: 안드로이드 에뮬레이터 기준 `10.0.2.2` (localhost 대리 주소)
- Firebase Project ID: `safetrip-urock`
- RTDB URL: `safetrip-urock-default-rtdb.firebaseio.com`

### 8.3 dev/prod 환경 분리 현황

| 항목 | dev 환경 | prod 환경 |
|------|---------|---------|
| API 서버 | `http://10.0.2.2:3001` (로컬) 또는 ngrok | 실제 서버 URL |
| Firebase Auth | 에뮬레이터 (포트 9099) | 실제 Firebase |
| Firebase RTDB | 에뮬레이터 (포트 9000) | 실제 Firebase |
| FCM | 미지원 (에뮬레이터) | 실제 FCM |

### 8.4 빌드 플레이버

현재 빌드 플레이버(flavor)는 **미구현** 상태이다. 환경 분리는 `.env` 파일 수동 교체 방식으로 운영 중이다. 플레이버 도입은 §9 우선순위 P2로 계획되어 있다.

## §13. 에러 및 엣지케이스 처리

### 13.1 환경 설정 에러

| 상황 | 처리 방식 |
|------|---------|
| `.env` 파일 누락 | `main.dart`에서 `dotenv.load()` 실패 → 앱 시작 차단 + 에러 로그 |
| 필수 환경 변수 누락 | `ApiService` 초기화 시 assert 검증 → 빠른 실패(fast fail) |
| Firebase 초기화 실패 | `main.dart` try-catch → 재시도 불가 에러 화면 표시 |

### 13.2 라우팅 에러

| 상황 | 처리 방식 |
|------|---------|
| 미등록 경로 접근 | GoRouter `errorBuilder` → 404 에러 화면 표시 |
| 인증 없이 보호 경로 접근 | `AuthNotifier` redirect → `/auth/phone` 리다이렉트 |
| 딥링크 잘못된 코드 | `/trip/join` 화면에서 코드 검증 실패 → 에러 토스트 표시 |

### 13.3 Firebase 에러

| 상황 | 처리 방식 |
|------|---------|
| RTDB 연결 실패 | 위치 공유 일시 중단 + 재연결 시도 (exponential backoff) |
| FCM 수신 실패 | `notification_service.dart` 내 에러 캐치 → 앱 크래시 방지 |
| Firebase Auth 세션 만료 | `AuthNotifier`가 상태 변경 감지 → `/auth/phone` 리다이렉트 |

### 13.4 위치 추적 에러

| 상황 | 처리 방식 |
|------|---------|
| GPS 권한 거부 | `screen_permission.dart`에서 권한 재요청 유도 |
| GPS 신호 없음 | 마지막 알려진 위치 사용 + 사용자 알림 |
| 배경 위치 추적 중단 | `flutter_background_geolocation` 재시작 시도 |
