## 1. 문서 목적 및 적용 범위

### 1.1 목적

본 문서는 SafeTrip 앱 실행 시 최초로 표시되는 스플래시 화면의 설계 원칙을 정의한다. 스플래시 화면은 브랜드 인식 확보와 함께 백그라운드 초기화 작업을 수행하고, 사용자 상태(신규/복귀/딥링크)에 따라 다음 진입 화면을 결정하는 라우팅 허브 역할을 한다.

비즈니스 원칙 v5.1 §06(회원가입 및 온보딩 원칙), 특히 §06.4(앱 재실행 시 흐름)와 온보딩 UX 시나리오(#14)의 분기 로직을 UI 레벨에서 구현하는 규칙을 명세한다.

### 1.2 적용 범위

| 범위 | 포함 여부 |
|------|:--------:|
| 앱 콜드 스타트 시 스플래시 표시 | ✅ |
| 배경 초기화 작업 (토큰 갱신, 버전 확인 등) | ✅ |
| 신규 유저 분기 (웰컴화면으로 전환) | ✅ |
| 복귀 유저 분기 (메인 화면으로 전환) | ✅ |
| 딥링크 진입 분기 (딥링크 대상 화면으로 전환) | ✅ |
| 강제 업데이트 다이얼로그 표시 | ✅ |
| 로딩 인디케이터 단계별 표시 | ✅ |
| 접근성 (스크린 리더, 다크 모드) | ✅ |

## 2. 스플래시 화면 설계 철학

스플래시 화면은 4가지 핵심 원칙을 동시에 충족해야 한다.

### 2.1 브랜드 인식

- SafeTrip 로고와 슬로건을 화면 중앙에 배치하여 브랜드 인식을 확보한다.
- 슬로건: "함께 떠나고, 안전하게 돌아오다" (단말 언어 설정에 따라 다국어 표시)
- 로고는 밝은 테마·어두운 테마(다크 모드) 모두 가독성을 확보하는 배리언트를 각각 적용한다.
- 배경: SafeTrip 브랜드 컬러 (프라이머리 컬러, 다크 모드에서는 다크 배경으로 자동 전환)

### 2.2 최소 대기 시간

- 사용자 대기 시간을 최소화한다: 초기화가 완료되는 즉시 전환한다 (단, 최소 표시 시간 1.0초 보장).
- 불필요한 인위적 지연(sleep) 금지 — 초기화 완료 이벤트 기반으로 전환한다.

### 2.3 배경 초기화 완료 후 전환

- 스플래시 표시 중 5가지 배경 초기화 작업(§5 참조)을 병렬로 수행한다.
- 모든 필수 초기화 완료 또는 최대 대기 시간(3.0초) 도달 중 먼저 발생한 조건에서 전환한다.
- 초기화 실패 항목은 오프라인 모드 또는 에러 화면으로 분기한다.

### 2.4 분기 결정

- 스플래시 종료 시점에 다음 3가지 경로 중 하나로 정확히 분기한다:
  - **신규 유저**: 웰컴화면(#29)
  - **복귀 유저**: 메인 화면 (또는 딥링크 대상)
  - **딥링크 진입**: 딥링크 대상 화면
- 분기 결정 로직은 단말 로컬 상태(저장된 인증 토큰 유무) 기반으로 동작하며, 서버 호출 없이 판단한다.

## 3. 스플래시 표시 시간 규칙

| 구분 | 시간 | 설명 |
|------|------|------|
| **최소 표시 시간** | 1.0초 | 브랜드 인식 확보를 위한 최소 보장 시간. 초기화가 1.0초 이전에 완료되어도 전환하지 않는다. |
| **최대 표시 시간** | 3.0초 | 초기화 완료를 기다리는 최대 대기 시간. 3.0초 초과 시 재시도 UI를 표시한다. |
| **정상 전환 조건** | 1.0초 이후 + 초기화 완료 | 두 조건을 모두 충족한 직후 다음 화면으로 전환한다. |

```
타임라인:
  0s ─────── 1s ─────────────────── 3s
  │          │                      │
  [로고만]    [최소 보장]             [최대 대기]
             │                      │
             └─ 초기화 완료 시 즉시 → 전환
                                    └─ 3초 초과 시 → 재시도 UI 표시
```

## 4. 분기 로직 — 3가지 진입 경로

스플래시 종료 시점에 단말 로컬 상태를 판단하여 다음 3가지 경로 중 하나로 분기한다. 분기 판단은 서버 네트워크 요청 없이 단말에서 수행한다.

### 4.1 경로 A — 신규 유저

**조건**: 로컬 저장소에 인증 토큰이 없음 (최초 설치 또는 로그아웃 상태)

```
스플래시
  ↓
웰컴화면 (#29) — 가치 슬라이드 (3~4장)
  ↓
목적 선택: "여행 만들기" / "초대코드 입력" / "먼저 둘러보기" / "가디언으로 참여"
  ↓
전화번호 인증 → 온보딩 흐름 (#14)
```

- 딥링크 파라미터가 있는 경우: 웰컴화면(#29)에 딥링크 파라미터를 전달하여 자동 처리 (§6 참조)

### 4.2 경로 B — 복귀 유저 (로그인됨)

**조건**: 로컬 저장소에 유효한 인증 토큰이 있음

```
스플래시
  ↓ (Firebase 토큰 갱신 성공)
메인 화면
  └─ 진행 중(active) 여행이 있으면: 해당 여행 화면으로 직접 이동
  └─ 진행 중 여행이 없으면: 탐색 모드 메인 화면 (§06.3 비즈니스 원칙)
```

- 메인 화면 자동 배정 우선순위 (비즈니스 원칙 §06.2):
  1. `active` 상태 여행 (현재 날짜 기준 가장 가까운 순)
  2. `planning` 상태 여행 (시작일이 현재에 가장 가까운 순)
  3. 동일 우선순위 여행이 복수인 경우 시작일이 빠른 여행 선택

### 4.3 경로 C — 딥링크 진입

**조건**: 앱이 딥링크(URI 스킴 또는 유니버설 링크)를 통해 실행됨

```
스플래시 (딥링크 파라미터 파싱 및 메모리 보존)
  ↓
  ├─ 로그인 상태: 딥링크 대상 화면으로 직접 이동
  └─ 비로그인 상태: 인증 흐름 진행 → 완료 후 딥링크 대상으로 자동 이동
```

| 딥링크 패턴 | 대상 화면 | 처리 규칙 |
|------------|---------|---------|
| `safetrip://invite/{code}` | 웰컴화면(#29) 또는 초대코드 입력 화면 | 초대코드 자동 입력 (초대코드 원칙 #23) |
| `safetrip://trip/{tripId}` | 해당 여행 메인 화면 | 로그인 및 여행 참여 확인 필요 |
| `safetrip://guardian/{linkId}` | 가디언 초대 수락 화면 | 로그인 필요, 가디언 링크 유효성 확인 |

## 5. 배경 초기화 작업 목록

스플래시 표시 시작과 동시에 다음 5가지 작업을 병렬로 수행한다. 각 작업의 실패 처리는 §11에서 정의한다.

| 순서 | 작업 | 종류 | 실패 시 처리 |
|------|------|------|------------|
| 1 | **Firebase 인증 토큰 갱신** (`silentRefresh`) | 필수 | 토큰 삭제 후 재로그인 유도 (경로 A로 분기) |
| 2 | **FCM 토큰 유효성 확인 및 갱신** | 선택 | 실패 무시 (알림 기능 비정상 동작 허용, 이후 재시도) |
| 3 | **앱 버전 확인** (서버 최소·권장 버전과 비교) | 필수 | critical: 업데이트 강제 차단 / optional: 스킵 가능 (§7 참조) |
| 4 | **로컬 캐시 무결성 체크** (corruption 검사) | 선택 | 손상된 캐시 초기화 후 계속 진행 |
| 5 | **딥링크 파라미터 파싱 및 메모리 보존** | 조건부 | 딥링크 진입 시에만 수행 (§6 참조) |

**초기화 완료 판단 기준**: 작업 1(Firebase 토큰 갱신)과 작업 3(앱 버전 확인)이 완료되면 필수 초기화 완료로 간주한다. 작업 2·4는 비동기로 계속 실행한다.

**병렬 실행 구조**:
```
스플래시 시작
  ├─ [작업 1] Firebase silentRefresh  ─────────────────┐
  ├─ [작업 2] FCM 토큰 갱신            ─ 비동기 처리     │
  ├─ [작업 3] 앱 버전 확인  ───────────────────────────┤ 필수 완료 → 전환 판단
  ├─ [작업 4] 로컬 캐시 체크           ─ 비동기 처리     │
  └─ [작업 5] 딥링크 파라미터 파싱  ──────────────────┘
```

## 6. 딥링크 보존 규칙

딥링크 파라미터는 사용자 경험의 연속성을 위해 인증 흐름을 거치는 동안에도 보존한다.

### 6.1 파싱 및 보존

- 스플래시 화면 초기화 시점에 딥링크 파라미터를 파싱하여 앱 메모리(세션 범위)에 저장한다.
- 파싱 대상: URI 스킴(`safetrip://`) 및 유니버설 링크(`https://safetrip.app/...`)
- 보존 범위: 세션 유지 동안 (앱 강제 종료 시까지)

### 6.2 로그인 필요 시 처리

```
딥링크 진입
  ↓
스플래시에서 딥링크 파라미터 파싱 → 메모리 보존
  ↓
비로그인 상태: 전화번호 인증 화면으로 이동
  ↓
인증 완료
  ↓
보존된 딥링크 파라미터 복원 → 딥링크 대상 화면으로 자동 이동
```

### 6.3 초대코드 딥링크 처리

- 딥링크 패턴: `safetrip://invite/{code}`
- 신규 유저: 웰컴화면(#29)으로 이동 시 초대코드를 자동 입력 상태로 전달
- 복귀 유저(로그인): 초대코드 입력 화면으로 직접 이동 후 코드 자동 입력
- 초대코드 유효성 검증은 초대코드 원칙(#23)의 규칙을 따른다

### 6.4 보존 만료

- 딥링크 파라미터는 다음 조건에서 폐기한다:
  - 딥링크 대상 화면으로 성공적으로 이동한 후
  - 사용자가 수동으로 다른 화면 흐름을 시작한 후 (예: "건너뛰기" 선택)
  - 앱을 강제 종료한 후

## 7. 강제 업데이트 분기

앱 버전 확인(배경 초기화 작업 3)에서 서버의 최소·권장 버전과 현재 앱 버전을 비교한다.

### 7.1 critical — 필수 업데이트

**조건**: 서버 최소 버전(`min_version`) > 현재 앱 버전

```
스플래시
  ↓
업데이트 필수 다이얼로그 표시
  - 제목: "업데이트가 필요합니다"
  - 내용: "안전한 여행을 위해 최신 버전으로 업데이트해 주세요."
  - 버튼: "지금 업데이트" (유일한 버튼, 닫기 불가)
  ↓
"지금 업데이트" 탭
  ↓
앱 스토어 이동 (iOS: App Store / Android: Google Play)
  ↓
앱 차단 (스토어에서 돌아와도 동일 다이얼로그 재표시)
```

- 앱 차단 상태에서는 어떤 화면으로도 진입 불가
- 업데이트 확인 주기: 앱 실행 시마다 (콜드 스타트 매회)

### 7.2 optional — 권장 업데이트

**조건**: 서버 권장 버전(`recommended_version`) > 현재 앱 버전 (단, 최소 버전 이상)

```
스플래시 정상 진행
  ↓
다음 화면 진입 후 업데이트 권장 배너 또는 알림 표시
  - 내용: "새 버전이 있습니다. 지금 업데이트하시겠습니까?"
  - 버튼: "업데이트" / "나중에"
  ↓
"나중에" 선택 시: 정상 앱 사용 계속 (차단 없음)
```

### 7.3 버전 확인 실패 처리

- 네트워크 오류로 버전 확인 불가: 마지막으로 성공한 버전 정보 캐시 사용 → 캐시도 없으면 업데이트 확인 생략하고 정상 진행
- 서버 오류(5xx): 업데이트 확인 생략하고 정상 진행 (사용자 차단 금지)

## 8. 로딩 인디케이터 규칙

스플래시 표시 시간에 따라 로딩 인디케이터를 단계적으로 표시하여 사용자에게 진행 상태를 전달한다.

| 단계 | 시간 범위 | 표시 요소 | 비고 |
|------|---------|---------|------|
| **1단계** | 0 ~ 1.0초 | SafeTrip 브랜드 로고 + 슬로건만 표시 | 프로그레스 인디케이터 없음. 빠른 실행 시 이 단계에서 바로 전환됨 |
| **2단계** | 1.0 ~ 3.0초 | 로고 + 슬로건 + 미세 프로그레스 바 (하단) | 초기화가 진행 중임을 시각적으로 표시. 프로그레스 바는 애니메이션 (indeterminate) |
| **3단계** | 3.0초 초과 | 로고 + "연결 중..." 텍스트 + 재시도 버튼 | 초기화 미완료 시 사용자에게 재시도 옵션 제공 |

### 8.1 단계 1 (0 ~ 1.0초) 상세

- 화면 중앙: SafeTrip 로고 (애니메이션: 페이드인 0.3초)
- 화면 하단: 슬로건 텍스트 (로고 페이드인 완료 후 0.2초 딜레이로 표시)
- 프로그레스 바: 표시하지 않음

### 8.2 단계 2 (1.0 ~ 3.0초) 상세

- 기존 로고·슬로건 유지
- 하단 미세 프로그레스 바: 높이 2dp, 좌우 여백 40dp, indeterminate 애니메이션
- 프로그레스 바 출현: 페이드인 0.2초

### 8.3 단계 3 (3.0초 초과) 상세

- 기존 로고 유지, 슬로건 숨김
- "연결 중..." 텍스트: 로고 하단 24dp, 보조 텍스트 색상
- 재시도 버튼: "다시 시도" 텍스트, 아웃라인 버튼 스타일
- 재시도 탭 시: 배경 초기화 작업 전체 재실행, 타임라인 0초부터 재시작

## 9. 역할별 접근 권한 매트릭스

**스플래시 화면은 역할 판별 전 화면이므로 역할별 분기가 존재하지 않는다.**

| 역할 | 스플래시 표시 | 배경 초기화 | 분기 로직 |
|------|:-----------:|:---------:|:--------:|
| captain | ✅ (역할 판별 전, 동일) | ✅ | ✅ (동일) |
| crew_chief | ✅ (역할 판별 전, 동일) | ✅ | ✅ (동일) |
| crew | ✅ (역할 판별 전, 동일) | ✅ | ✅ (동일) |
| guardian | ✅ (역할 판별 전, 동일) | ✅ | ✅ (동일) |
| 미분류 (신규) | ✅ (역할 판별 전, 동일) | ✅ | ✅ (동일) |

**역할별 최적화 (복귀 유저 전용)**: 복귀 유저의 경우, 로컬 저장소에 캐시된 역할 정보를 활용하여 메인 화면 진입 후 역할에 맞는 UI를 즉시 표시한다. 스플래시 자체의 동작은 역할 무관하게 동일하다.

## 10. 프라이버시 등급별 동작 차이

**스플래시 화면은 프라이버시 등급 판별 전 화면이므로 등급별 동작 차이가 없다.**

| 프라이버시 등급 | 스플래시 동작 |
|--------------|:----------:|
| safety_first (안전최우선) | 해당 없음 (등급 판별 전) |
| standard (표준) | 해당 없음 (등급 판별 전) |
| privacy_first (프라이버시우선) | 해당 없음 (등급 판별 전) |

프라이버시 등급 기반 동작 차이는 메인 화면 진입 후부터 적용된다. 위치 공유, 마커 표시 등 등급별 차이는 지도탭 원칙(#17) 및 위치 공유 원칙을 참조한다.

## 11. 에러 및 엣지케이스 처리

### 11.1 에러 유형별 처리 전략

| 에러 유형 | 감지 시점 | 처리 방법 | 사용자 노출 |
|---------|---------|---------|----------|
| **네트워크 오류** | 배경 초기화 중 | 오프라인 모드 진입 안내 후 분기 진행 | "오프라인 상태입니다. 일부 기능이 제한됩니다." 토스트 |
| **서버 점검 중 (503)** | 앱 버전 확인 또는 토큰 갱신 중 | 점검 공지 화면으로 전환 | 점검 시작·종료 예정 시각 및 공지 표시 |
| **Firebase 인증 만료** | Firebase silentRefresh 실패 | 로컬 토큰 삭제 → 경로 A(신규 유저 흐름) 분기 | "다시 로그인이 필요합니다." 안내 (웰컴화면 진입 시) |
| **앱 버전 확인 타임아웃** | 3초 초과 | 버전 확인 생략, 정상 분기 진행 | 없음 (사용자 인지 불필요) |
| **로컬 캐시 손상** | 캐시 무결성 체크 중 | 손상 캐시 초기화 후 계속 진행 | "캐시를 초기화했습니다." (선택적 토스트) |
| **딥링크 파싱 오류** | 딥링크 파라미터 파싱 중 | 딥링크 무시, 경로 A 또는 B로 정상 분기 | 없음 |

### 11.2 에러 분기 결정 트리

```
배경 초기화 실행
  ├─ Firebase 토큰 갱신 성공? ──YES─→ 복귀 유저 흐름 진행
  │                           └─NO─→ 토큰 만료? ─YES→ 로컬 삭제 → 신규 유저 경로
  │                                               └─NO (네트워크 오류)
  │                                                    → 오프라인 모드 알림 → 경로 판단 계속
  ├─ 앱 버전 확인 성공?
  │   ├─ critical 업데이트 필요? → 강제 업데이트 다이얼로그 (§7.1)
  │   ├─ optional 업데이트 있음? → 정상 진행, 이후 권장 알림 (§7.2)
  │   └─ 최신 버전: 정상 진행
  └─ 서버 503? → 점검 공지 화면 전환
```

### 11.3 재시도 메커니즘

- 재시도 버튼 표시 조건: 초기화 시작으로부터 3.0초 경과 후 미완료 상태
- 재시도 실행: 배경 초기화 전체(작업 1~5)를 처음부터 다시 수행
- 재시도 횟수 제한 없음 (사용자가 원하는 만큼 재시도 가능)
- 연속 실패 시 안내: 3회 연속 실패 후 "네트워크 연결을 확인해 주세요." 추가 안내 표시

## 12. DB 스키마

**해당 없음**

스플래시 화면은 서버 DB에 직접 접근하지 않는다. 배경 초기화 작업 중 Firebase Authentication(토큰 갱신) 및 서버 버전 확인 API는 사용하지만, 이는 스플래시 화면의 DB 스키마가 아닌 각 시스템의 API 인터페이스에 해당한다.

관련 서버 API:
- `GET /api/v1/version/check` — 앱 버전 확인 (서버 최소·권장 버전 반환)
- Firebase Auth `currentUser.getIdToken(true)` — 토큰 갱신 (Firebase SDK)

## 13. 오프라인 대응

### 13.1 오프라인 진입 정책

스플래시 화면에서 네트워크가 없는 경우, 가능한 범위 내에서 초기화를 완료하고 제한된 기능으로 앱을 진행시킨다. 사용자를 오프라인 상태에서 완전히 차단하지 않는다.

| 초기화 작업 | 오프라인 처리 |
|----------|------------|
| Firebase 토큰 갱신 | 로컬 캐시 토큰 사용 (만료 전이면 계속 사용 가능) |
| FCM 토큰 갱신 | 건너뜀 (네트워크 복구 후 자동 재시도) |
| 앱 버전 확인 | 마지막 성공 캐시 사용 / 캐시 없으면 건너뜀 |
| 로컬 캐시 무결성 체크 | 정상 수행 (로컬 작업이므로 네트워크 불필요) |
| 딥링크 파라미터 파싱 | 정상 수행 (로컬 작업) |

### 13.2 오프라인 상태 사용자 안내

- 화면 하단 배너: "오프라인 상태입니다. 일부 기능이 제한됩니다." (앰버 색상)
- 배너는 다음 화면(웰컴 또는 메인)에도 유지된다
- 네트워크 복구 시 배너 자동 소멸 및 백그라운드 초기화 재시도

### 13.3 오프라인 분기 결정

- 로컬 토큰 유효: 복귀 유저 경로(경로 B) 진행 (오프라인 모드 알림 포함)
- 로컬 토큰 없음: 신규 유저 경로(경로 A) 진행 (로그인 시 네트워크 필요 안내)
