### 1.2 위치 데이터 유형 분류

SafeTrip에서 수집하는 위치 데이터를 5가지 유형으로 분류한다. 각 유형은 수집 목적, 빈도, 보존 기간, 접근 권한이 다르다.

| 유형 | 설명 | 수집 빈도 | 저장소 | 보존 기간 |
|------|------|----------|--------|----------|
| **L1: 실시간 위치** | 지도 마커 표시용 현재 위치 | 30초~10분 | Firebase RTDB + PostgreSQL | 여행 종료 + 90일 |
| **L2: 이동 경로 로그** | 과거 이동 경로 재생용 상세 위치 | 30초~5분 | PostgreSQL (TB_LOCATION) ⚠️v3.1 | 여행 종료 + 90일 |
| **L3: Heartbeat 위치** | 생존 신호 포함 위치 | 3~10분 | PostgreSQL (TB_HEARTBEAT) | 여행 종료 + 90일 |
| **L4: SOS/안전 위치** | SOS 발송, 지오펜스 이벤트 시점 위치 | 이벤트 발생 시 | PostgreSQL (TB_SOS_EVENT, TB_EVENT_LOG) | 여행 종료 + 3년 |
| **L5: 체류 지점** | 10분 이상 체류한 장소 집계 | 5분 배치 분석 | PostgreSQL (TB_STAY_POINT) | 여행 종료 + 90일 |
| **L6: 가디언 스냅샷** ⭐ | 표준 등급 비공유 시간대 30분 간격 스냅샷 (가디언 전용) | 30분 (비공유 시간) | PostgreSQL (TB_GUARDIAN_SNAPSHOT) | 여행 종료 + 30일 |

### 1.3 핵심 원칙

| # | 원칙 | 설명 |
|---|------|------|
| D1 | **목적 명확성** | 수집하는 모든 위치 데이터에는 명시적 수집 목적이 존재해야 한다. 목적 외 활용은 금지한다. |
| D2 | **최소 수집** | 서비스 제공에 필요한 최소한의 위치 데이터만 수집한다. 프라이버시 등급에 따라 수집 빈도를 차등 적용한다. |
| D3 | **안전 우선** | SOS 관련 위치 데이터는 프라이버시 설정과 무관하게 수집·보존한다. 안전은 프라이버시보다 상위 가치이다. |
| D4 | **투명성** | 수집되는 데이터 항목, 빈도, 보존 기간, 접근 권한을 사용자에게 명확히 고지한다. |
| D5 | **확실한 삭제** | 보존 기간 만료 시 3단계 파기 절차를 통해 복구 불가능한 수준으로 삭제한다. |
| D6 | **접근 통제** | 위치 데이터 접근은 역할·등급 기반으로 통제하며, 모든 접근을 기록한다. |

## 2. 수집 단계: 데이터 수집 정책

### 2.1 수집 빈도 매트릭스

수집 빈도는 **프라이버시 등급**, **멤버 상태**, **배터리 수준**의 3축으로 결정된다.

#### 프라이버시 등급별 기본 수집 빈도

| 수집 항목 | 안전 최우선 | 표준 | 프라이버시 우선 |
|----------|:---------:|:----:|:-------------:|
| 실시간 위치 (이동 중) | 30초 | 1분 | 5분 |
| 실시간 위치 (정지 중) | 2분 | 5분 | 10분 |
| Heartbeat | 3분 | 5분 | 10분 |
| 지오펜스 판정 | 30초 | 1분 | 2분 |

#### 배터리 수준별 보정

| 배터리 수준 | 보정 계수 | 적용 결과 (표준 등급, 이동 중) |
|:---------:|:--------:|:---------------------------:|
| 20% 이상 | ×1 (기본) | 1분 |
| 10~20% | ×2 | 2분 |
| 5~10% | ×5 (+ 고정밀 GPS 활성화) | 5분 |
| 5% 미만 | Heartbeat 1분 최대 빈도 전환 | 최종 위치 확보 우선 |

#### 스케줄 시간대별 보정

| 시간대 | 보정 | 설명 |
|--------|------|------|
| 스케줄 ON (활동 시간) | ×1 (기본) | 일정이 진행되는 시간대 |
| 스케줄 OFF (자유 시간) | ×2 | Heartbeat 주기도 2배 완화 |
| 심야 (00:00~06:00) + 숙소 지오펜스 내 | ×3 | 배터리 절약, 불필요한 수집 최소화 |

### 2.2 수집 항목 상세

각 위치 데이터 레코드에 포함되는 필드:

```
위치 데이터 레코드 구조:
{
  user_id:         VARCHAR(128),    // 수집 대상자 ID
  group_id:        UUID,            // 소속 그룹 ID
  trip_id:         UUID,            // 소속 여행 ID
  latitude:        DOUBLE,          // 위도 (소수점 6자리)
  longitude:       DOUBLE,          // 경도 (소수점 6자리)
  accuracy:        REAL,            // GPS 정확도 (미터)
  speed:           REAL,            // 순간 속도 (m/s)
  bearing:         REAL,            // 이동 방향 (도)
  altitude:        REAL,            // 고도 (미터)
  battery_level:   SMALLINT,        // 배터리 잔량 (%)
  network_type:    VARCHAR(10),     // wifi/4g/5g/none
  is_sharing:      BOOLEAN,         // 위치 공유 활성 상태 여부
  motion_state:    VARCHAR(20),     // moving/stationary/unknown
  provider:        VARCHAR(10),     // gps/network/fused
  recorded_at:     TIMESTAMPTZ,     // 기록 시각 (클라이언트 시계)
  server_received_at: TIMESTAMPTZ   // 서버 수신 시각
}
```

### 2.3 수집 시작/종료 조건

```
수집 시작 조건:
├── 여행 상태가 `active`로 전환된 시점
├── AND 사용자가 위치 수집 동의를 완료 (tb_user_consent.location_collect = true)
├── AND 기기 위치 권한 "항상 허용" 상태
└── AND 여행 그룹의 멤버 상태가 active

수집 종료 조건:
├── 여행 상태가 `completed` 또는 `deleted`로 전환
├── OR 사용자가 위치 수집 동의 철회
├── OR 그룹에서 제거됨
├── OR 계정 삭제 요청
└── 예외: SOS 발동 중에는 동의 철회·그룹 제거와 무관하게 수집 지속
          (SOS 해소 시점까지, 최대 24시간)
```

### 2.4 동의 기반 수집 체계

위치정보법 제15조 및 개인정보보호법 제15조에 따라, 위치 데이터 수집은 명시적 동의에 기반한다.

```
동의 획득 시점 및 항목:

1. 온보딩 약관 동의 (screen_5_terms)
   ├── [필수] 위치기반서비스 이용약관 동의
   ├── [필수] 개인정보처리방침 동의
   └── 저장: tb_user_consent (location_service, privacy_policy)

2. 여행 참가 시 (trip join)
   ├── [필수] 위치 공유 동의 (여행 기간 중)
   ├── [선택] 이동기록 저장 동의 (과거 경로 보존)
   └── 저장: tb_user_consent (location_share, location_log)

3. SOS 관련 (온보딩 시 일괄)
   ├── [필수] SOS 시 위치 강제 수집 동의
   ├── [선택] Heartbeat 모니터링 동의 (안전 최우선 등급은 필수)
   └── 저장: tb_user_consent (sos_location, heartbeat)

4. 미성년자 추가 동의 (14세 미만)
   ├── [필수] 법정대리인(가디언) 동의
   └── 저장: tb_user_consent (minor_guardian_consent, consent_guardian_id)
```

## 3. 전송 단계: 클라이언트→서버 데이터 전송

### 3.1 이중 전송 경로

SafeTrip은 실시간성과 영속성을 동시에 달성하기 위해 이중 전송 경로를 사용한다.

```
┌─────────────┐
│  클라이언트   │
│  (Flutter)   │
└──────┬───┬──┘
       │   │
       │   │  경로 A: 실시간 (WebSocket)
       │   └──────────────────────────────┐
       │                                   │
       │  경로 B: 이력 저장 (HTTP/REST)     │
       │                                   │
       ▼                                   ▼
┌──────────────┐                   ┌──────────────────┐
│  NestJS API │                   │  Firebase RTDB   │
│  (port 3001) │                   │  /locations/     │
│              │                   │  {groupId}/      │
│  TB_LOCATION │   10분 배치 동기   │  {userId}        │
│  (v3.1~)     │◄══════════════════│                  │
│  (PostgreSQL)│                   │  (최신 1건만 보존) │
└──────────────┘                   └──────────────────┘
```

#### 경로 A: Firebase RTDB (실시간)

- **목적**: 지도 위 멤버 마커 실시간 업데이트
- **전송 방식**: WebSocket (Firebase SDK 자동 관리)
- **데이터**: 최신 위치 1건만 덮어쓰기 (이력 미보존)
- **구조**: `/locations/{groupId}/{userId}` → `{ lat, lng, accuracy, timestamp, battery, motion }`
- **보안**: Firebase RTDB Security Rules (자신의 데이터만 쓰기 가능)

#### 경로 B: NestJS API (이력)

- **목적**: 이동 경로 로그 영속 저장
- **전송 방식**: HTTP POST (Dio 클라이언트, 배치 업로드)
- **배치 규칙**: 로컬에 5건 누적 또는 5분 경과 시 배치 전송
- **인증**: Bearer {Firebase ID Token}
- **엔드포인트**: `POST /api/v1/locations/batch`

### 3.2 전송 암호화

| 구간 | 암호화 방식 | 비고 |
|------|-----------|------|
| 클라이언트 → Firebase RTDB | TLS 1.3 (Firebase SDK 내장) | Google 인프라 |
| 클라이언트 → NestJS API | TLS 1.3 (HTTPS) | ngrok 개발 환경은 HTTP |
| NestJS API → PostgreSQL | TLS 1.2+ (pg 라이브러리 ssl 옵션) | 로컬 개발은 비암호화 |
| Firebase RTDB → PostgreSQL | 내부 네트워크 (배치 동기화) | 서버 → 서버, 인증된 서비스 계정 |

### 3.3 오프라인 전송 큐

```
오프라인 상태 감지
    │
    ▼
로컬 SQLite 큐에 위치 데이터 저장
    {
      queue_id: AUTO_INCREMENT,
      data: JSON(위치 레코드),
      priority: SOS(1) > HEARTBEAT(2) > LOCATION(3),
      queued_at: TIMESTAMP,
      retry_count: 0,
      status: "pending"
    }
    │
    ▼ 네트워크 복구 감지
    │
    ├── SOS 큐 → 즉시 전송 (우선순위 최상)
    ├── Heartbeat 큐 → 5초 후 전송
    └── Location 큐 → 10초 후 배치 전송 (최대 50건/요청)
    │
    ├── 전송 성공 → 큐에서 삭제
    └── 전송 실패 → retry_count++ → 재시도 (최대 10회, 지수 백오프)
        └── 10회 실패 → "전송 실패" 로컬 로그 + 사용자 알림

큐 보존 기간: 72시간 (이후 자동 삭제)
큐 최대 크기: 10,000건 (초과 시 오래된 LOCATION 우선 삭제, SOS/HEARTBEAT 보존)
```

## 4. 저장 단계: 데이터 저장 구조 및 암호화

### 4.1 저장소별 데이터 분류

| 저장소 | 저장 데이터 | 보존 방식 | 접근 제어 |
|--------|-----------|----------|----------|
| **Firebase RTDB** | 최신 실시간 위치 (1건/유저) | 덮어쓰기 (이력 없음) | Security Rules (그룹 기반) |
| **PostgreSQL** | 위치 이력, 체류, Heartbeat, SOS | 시계열 테이블 (파티셔닝) | 앱 레벨 RBAC + Row Level Security |
| **클라이언트 SQLite** | 오프라인 큐, 로컬 캐시 (7일) | 암호화 DB (SQLCipher) | 앱 샌드박스 |
| **클라이언트 메모리** | 현재 세션 위치 (표시용) | 앱 종료 시 소멸 | 프로세스 격리 |

### 4.2 PostgreSQL 위경도 암호화

위경도 데이터는 민감한 개인정보이므로, 저장 시 컬럼 레벨 암호화를 적용한다.

```
암호화 전략: Application-Level Encryption (ALE)

[클라이언트] → 평문 위경도 → [NestJS API] → AES-256-GCM 암호화 → [PostgreSQL]

키 관리:
  - 암호화 키: 환경변수 LOCATION_ENCRYPTION_KEY (256bit)
  - 키 로테이션: 90일 주기
  - 이전 키 보존: 2세대 (현재 + 이전 1개)
  - 키 저장: AWS KMS 또는 GCP Secret Manager

암호화 대상 컬럼:
  - TB_LOCATION: latitude, longitude  (⚠️ v3.1: TB_LOCATION_LOG → TB_LOCATION 변경)
  - TB_STAY_POINT: center_lat, center_lng
  - TB_HEARTBEAT: location_lat, location_lng
  - TB_SOS_EVENT: location_lat, location_lng
  - TB_POWER_EVENT: location_lat, location_lng
  - TB_GUARDIAN_SNAPSHOT: location_lat, location_lng  ⭐ v3.4 신규

비암호화 컬럼 (인덱싱/쿼리 필요):
  - recorded_at, user_id, group_id (필터링용)
  - accuracy, speed (통계 분석용)
  - geohash (근접 검색용, 정밀도 6 = ~1.2km 해상도)
```

### 4.3 Firebase RTDB 데이터 구조

```json
{
  "locations": {
    "{groupId}": {
      "{userId}": {
        "lat": 35.6762,
```

Firebase RTDB Security Rules:

```
    "locations": {
      "$groupId": {
        "$userId": {

}
```

### 4.4 테이블 파티셔닝 전략

위치 로그는 시간 경과에 따라 대량으로 축적되므로, 월별 파티셔닝을 적용한다.

> ⚠️ **v3.1 실제 구현 반영**: 원칙 문서에서 정의한 `TB_LOCATION_LOG`는 v3.1부터 `TB_LOCATION`으로 대체되었다.
> 실제 `TB_LOCATION` 스키마는 DB 설계 v3.4 §4.16 참조. 주요 차이점:
> - 컬럼명: `latitude DECIMAL(10,8)` / `longitude DECIMAL(11,8)` (암호화는 애플리케이션 레벨 AES-256-GCM 적용 후 DECIMAL 컬럼에 저장)
> - `group_id` 미포함 (user_id 기반 관리)
> - `trip_id` 미포함 (v3.0에서 제거)
> - PostGIS `geom GEOGRAPHY(POINT,4326)` 추가
> - 이동 세션(`movement_session_id`), Activity Recognition(`activity_type`) 컬럼 추가

```sql
-- 월별 범위 파티셔닝 (TB_LOCATION 기준)
CREATE TABLE tb_location (
    location_id      UUID DEFAULT gen_random_uuid(),
    user_id          VARCHAR(128) NOT NULL,
    latitude         DECIMAL(10, 8) NOT NULL,  -- AES-256-GCM 암호화 후 저장 (앱 레벨)
    longitude        DECIMAL(11, 8) NOT NULL,
    geom             GEOGRAPHY(POINT, 4326),   -- PostGIS 공간 인덱스용
    accuracy         DECIMAL(6, 2),
    speed            DECIMAL(6, 2),
    heading          DECIMAL(5, 2),
    altitude         DECIMAL(8, 2),
    battery_level    INTEGER,
    tracking_mode    VARCHAR(20) DEFAULT 'normal',  -- normal|power_saving|minimal|sos
    movement_session_id UUID,
    activity_type    VARCHAR(20),
    i_idx            BIGINT,
    recorded_at      TIMESTAMPTZ NOT NULL DEFAULT (now() AT TIME ZONE 'UTC'),
    PRIMARY KEY (location_id, recorded_at)
) PARTITION BY RANGE (recorded_at);

-- 파티션 자동 생성 (월별)
CREATE TABLE tb_location_2026_03 PARTITION OF tb_location
    FOR VALUES FROM ('2026-03-01') TO ('2026-04-01');
CREATE TABLE tb_location_2026_04 PARTITION OF tb_location
    FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');
-- ... 3개월 선행 생성, 배치로 자동 관리
```

### 5.2 목적 외 활용 금지 항목

다음 목적으로의 위치 데이터 활용은 명시적으로 금지한다:

- 광고 타겟팅 또는 마케팅 프로파일링
- 제3자 데이터 브로커에게 위치 데이터 판매/제공

- 법원 영장 없는 수사기관 제공 (단, 위치정보법 §29 긴급 상황 제외)
- AI 모델 학습 데이터로의 활용 (SafeTrip AI 기능 원칙 §2.3 준수)

### 6.2 여행 기간 상한과 보존 기간 계산

비즈니스 원칙 v5.1 §02.3에 따라, **여행 최대 기간은 15일**로 제한된다. 이 제한은 데이터 보존 기간 계산 및 수집 지속 기간에 직접적인 영향을 미친다.

```
여행 기간 15일 제한이 데이터 생명주기에 미치는 영향:

1. 수집 지속 기간 상한:
   ├── 단일 여행에서 위치 수집이 연속 진행되는 최대 기간 = 15일
   ├── 15일 초과 시 여행 자동 종료 → 위치 수집 자동 중단
   └── 연장 불가: 새 여행 생성 필요 (데이터 별도 관리)

2. 보존 기간 계산 예시 (일반 사용자):
   ├── 여행 시작: D+0
   ├── 여행 최대 종료: D+15
   ├── 보존 기간 만료: D+15 + 90일 = D+105
   ├── 유예 기간 만료: D+105 + 30일 = D+135
   ├── 소프트 삭제 → 익명화: D+135 + 30일 = D+165
   └── 물리적 삭제 완료: D+165 + 30일 = D+195 (최대)

3. 보존 기간 계산 예시 (미성년자):
   ├── 여행 종료: D+15
   ├── 보존 기간 만료: D+15 + 30일 = D+45
   └── 물리적 삭제 완료: D+45 + 60일(유예+익명화) = D+105 (최대)

4. 배치 스케줄링 영향:
   ├── retention_scan (JOB-03): 15일 상한을 trip 종료 판정에 활용
   ├── 여행 미종료 판정: ended_at IS NULL AND started_at + 15일 < NOW()
   │   → 자동 abandoned 처리 → 보존 기간 기산 시작
   └── 단일 여행 데이터 최대 크기 예측:
       └── safety_first × 15일 = ~43,200건 (30초 간격 × 15일)
       └── standard × 15일 = ~21,600건 (1분 간격 × 15일)
       └── privacy_first × 15일 = ~4,320건 (5분 간격 × 15일)
```

### 6.3 보존 기간 기산점

```
보존 기간 기산 규칙:
├── 일반 데이터: TB_TRIP.ended_at (여행 종료 시점)
│   └── 여행 기간이 15일을 초과하면 시스템이 자동 종료 처리
├── SOS 데이터: TB_SOS_EVENT.resolved_at (SOS 해소 시점)
│   └── 미해소 SOS: 발생 시점 + 3년 (최장 보존)
├── 계정 삭제 시: 삭제 요청 시점 (별도 절차, §11 참조)
├── 여행 미종료 (abandoned): 마지막 활동 + 30일 또는 started_at + 15일 중 빠른 시점을 종료로 간주
└── 기산점 미확정 시: 데이터 생성 시점(created_at) 기준
```

### 6.4 보존 기간 만료 후 처리

```
보존 기간 만료 감지 (일일 배치, 매일 03:00 UTC)
    │
    ├── 만료 대상 데이터 식별
    │   └── SELECT WHERE (trip_ended_at + retention_period) < NOW()
    │
    ├── 유예 기간 (30일)
    │   ├── 데이터 상태를 "pending_deletion"으로 변경
    │   ├── 사용자에게 알림: "이동기록이 30일 후 삭제됩니다"
    │   └── 사용자 다운로드 기회 제공 (GPX 내보내기)
    │
    ├── 유예 기간 만료 → Stage 2 (소프트 삭제) 진행
    └── 사용자가 유예 기간 중 보존 연장 요청 → 미지원 (법적 보존 기간 준수)
```

## 7. 삭제 단계: 3단계 데이터 파기 절차

### 7.1 3단계 파기 프로세스

위치기반서비스 이용약관에 정의된 3단계 파기 절차의 기술 실행 계획이다.

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Stage 1: 만료 감지 (Detection)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  실행 주기: 매일 03:00 UTC (서버 부하 최저 시간)
  실행 주체: data_lifecycle_scheduler (Node.js 크론 잡)

  처리:
    1. 보존 기간 만료 데이터 스캔
    2. 만료 대상 목록 생성 (deletion_batch_id 발행)
    3. 대상 데이터에 pending_deletion 플래그 설정
    4. 관리자 대시보드에 삭제 예정 리포트 게시
    5. 감사 로그 기록: {batch_id, target_count, data_types, scheduled_at}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Stage 2: 소프트 삭제 + 익명화 (Soft Delete & Anonymize)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  실행 시점: Stage 1 완료 + 30일 유예 기간 경과 후
  실행 주체: data_anonymizer (Node.js 크론 잡)

  처리:
    1. 위경도 데이터 → 지오해시 정밀도 3 수준으로 라운딩
       (정밀도 3 ≒ 156km 해상도, 개인 식별 불가)
    2. user_id → SHA-256 해시값으로 치환 (비가역)
    3. 부가 데이터 (battery, speed, accuracy) → NULL 처리
    4. deleted_at 타임스탬프 설정
    5. is_anonymized = true 플래그 설정
    6. 감사 로그 기록: {batch_id, anonymized_count, method}

  익명화 보존 기간: 추가 30일 (통계 목적, 개인 식별 불가)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Stage 3: 물리적 삭제 (Hard Delete)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  실행 시점: Stage 2 완료 + 30일 경과 후
  실행 주체: data_purger (Node.js 크론 잡)

  처리:
    1. DELETE FROM tb_location WHERE deletion_batch_id = {batch_id}  -- (v3.1~: TB_LOCATION_LOG → TB_LOCATION)
    2. DELETE FROM tb_stay_point WHERE deletion_batch_id = {batch_id}
    3. DELETE FROM tb_heartbeat WHERE deletion_batch_id = {batch_id}
    4. 파티션 단위 삭제: 해당 월 파티션이 전부 만료 시 DROP PARTITION
    5. VACUUM FULL 실행 (디스크 공간 회수, 데이터 복구 불가)
    6. Firebase RTDB 해당 경로 삭제 확인
    7. 백업에서도 해당 데이터 제거 (백업 보존 기간: 30일)
    8. 감사 로그 기록: {batch_id, deleted_count, freed_space_mb, completed_at}
    9. 삭제 완료 증빙 리포트 생성 (법적 요구 시 제출용)
```

### 7.2 SOS 데이터 특별 파기 절차

SOS 관련 위치 데이터(L4)는 법적 분쟁 가능성이 있으므로 별도 파기 절차를 적용한다.

```
SOS 데이터 파기:
├── 보존 기간: 여행 종료 + 3년 (일반 데이터의 12배)
├── 소프트 삭제 불가: SOS 이벤트는 감사 로그 성격이므로 소프트 삭제 미적용
├── 물리적 삭제 전 검증:
│   ├── 해당 SOS에 연계된 법적 분쟁 존재 여부 확인
│   ├── 수사기관 보전 요청 존재 여부 확인
│   └── 확인 완료 시에만 삭제 진행
├── 삭제 승인: 시스템 자동 삭제가 아닌 관리자 승인 후 삭제
└── 감사 증적: 삭제 전 데이터 해시값 기록 (삭제 사실 증명용)
```

```
  ├── 비공유 시간에는 수집 자체를 하지 않음
  ├── GPS 센서 비활성화 (배터리 절약 효과)
```

### 9.2 가디언 과금 등급별 위치 데이터 접근 차이

비즈니스 원칙 v5.1 §03.1에 따라, 가디언은 **무료 가디언 (최대 2명)**과 **유료 가디언 (최대 3명 추가)**으로 구분된다. 과금 등급에 따라 위치 데이터 접근 범위가 차등 적용된다.

```
가디언 과금 등급별 위치 데이터 접근 차이:

무료 가디언 (Free, 최대 2명):
├── 실시간 위치 조회: ✅ (공유 시간 중 연결 멤버)
├── 이동기록 조회: ❌ (비활성)
├── Heartbeat 상태: ✅ (연결 멤버)
├── SOS 알림 수신: ✅
├── 위치 접근 빈도: 표준 (지도 열람 시)
├── 위치 갱신 주기: 일반 (프라이버시 등급 기본값)
└── 위치 알림: SOS, 지오펜스 이탈 알림만

유료 가디언 (Paid, 최대 3명 추가):
├── 실시간 위치 조회: ✅ (공유 시간 + 비공유 시간 스냅샷)
├── 이동기록 조회: ✅ (당일 이동 경로 요약)
├── Heartbeat 상태: ✅ (연결 멤버 + 상세 상태)
├── SOS 알림 수신: ✅
├── 위치 접근 빈도: 향상 (폴링 간격 단축)
├── 위치 갱신 주기: 향상 (유료 등급은 수집 빈도 1단계 상향)
├── 위치 알림: SOS, 지오펜스 이탈, 일정 지연, 통신 두절 경고
└── 일일 안전 리포트: ✅ (이동 요약 + 체류 지점 목록)

데이터 처리 차이:
├── 저장: 무료/유료 가디언 모두 접근 로그는 동일하게 기록
├── 보존: 과금 등급에 의한 보존 기간 차이 없음 (보존 정책은 데이터 주체 기준)
├── 삭제: 가디언 해지 시 해당 가디언의 접근 로그만 보존 (6개월)
└── 과금 전환: 유료 → 무료 전환 시 확장 접근 권한 즉시 회수
```

> **참고**: 가디언의 위치 데이터 접근은 연결된 멤버의 프라이버시 등급에도 영향받는다. 예를 들어, 유료 가디언이라도 프라이버시 우선 등급의 멤버 비공유 시간대 데이터에는 접근할 수 없다 (§8.2 참조). 단, **안전 최우선 등급의 미성년자**에 대해서는 무료/유료 구분 없이 전 시간 접근이 가능하다 (§10.3 참조).

### 9.3 접근 로그 기록 (위치정보법 §16)

모든 위치 데이터 접근은 TB_LOCATION_ACCESS_LOG에 기록된다.

> ⚠️ **v3.4 실제 구현 반영**: DB 설계 v3.4 §4.32 기준. 이전 명칭 `TB_DATA_ACCESS_LOG`는 `TB_LOCATION_ACCESS_LOG`로 변경되었다.

```sql
CREATE TABLE tb_location_access_log (
    log_id               BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
    user_id              VARCHAR(128) NOT NULL,     -- 위치정보 주체 (보호 대상)
    accessed_by_user_id  VARCHAR(128),              -- 열람자 user_id (NULL = 시스템)
    access_type          VARCHAR(30) NOT NULL,
        -- realtime_view | history_view | sos_broadcast |
        -- geofence_alert | guardian_snapshot | guardian_request |
        -- attendance_check | ai_analysis | safety_guide
    trip_id              UUID REFERENCES tb_trip(trip_id) ON DELETE SET NULL,
    location_data        JSONB,                     -- 암호화된 좌표 데이터
    access_purpose       VARCHAR(200),
    created_at           TIMESTAMPTZ DEFAULT NOW(),
    expired_at           TIMESTAMPTZ                -- created_at + 6개월
);

CREATE INDEX idx_loc_access_user    ON tb_location_access_log(user_id, created_at DESC);
CREATE INDEX idx_loc_access_type    ON tb_location_access_log(access_type);
CREATE INDEX idx_loc_access_expired ON tb_location_access_log(expired_at);
```

보존 기간: 6개월 (위치정보법 §16 요구사항)

## 10. 특수 상황 데이터 처리

### 10.1 SOS 발동 시

SOS는 모든 프라이버시 설정을 무시하는 최상위 안전 기능이다. (SOS 원칙 v1.1 §1.1)

```
SOS 발동 시 데이터 처리:
├── 위치 강제 수집: GPS 고정밀 모드 즉시 활성화
├── 수집 빈도: 10초 간격 (최대 빈도)
├── 서버 전송: 즉시 (오프라인 시 큐잉 후 복구 시 즉시)
├── 접근 권한: 그룹 전체 + 가디언 전원에게 실시간 공유
├── 프라이버시 무시: 비공유 시간이라도 위치 공개
├── 가디언 일시 중지 무시: 일시 중지 중이라도 가디언에게 전달
├── 강제 공유 지속: 캡틴 수동 해제 또는 24시간 후 자동 해제
└── 보존 기간: 여행 종료 + 3년 (L4 적용)
```

### 10.2 Heartbeat 타임아웃 시

```
Heartbeat 타임아웃 시 데이터 처리:
├── 마지막 알려진 위치: 서버에 보존 (에스컬레이션 판단용)
├── 의심 점수 계산: 마지막 위치, 배터리, 모션 상태 활용
├── Level 3 도달 시: 마지막 위치 기반 반경 표시 (시간 경과에 따라 확대)
├── Auto-SOS 발동 시: 마지막 위치를 SOS 위치로 기록 (L4 보존 적용)
└── 전원 복구 시: 복구 위치와 오프라인 기간 동안의 궤적 기록
```

### 10.3 미성년자 데이터 처리

미성년자(14세 미만)의 위치 데이터는 강화된 보호 정책을 적용한다.

```
미성년자 위치 데이터 특별 규칙:
├── 수집: 법정대리인(가디언) 동의 필수
├── 보존 기간: 여행 종료 + 30일 (일반 90일보다 단축)
├── 접근 제한: AI 개인 분석 차단 (Intelligence AI 미적용)

├── 내보내기: GPX 등 외부 반출 시 가디언 승인 필요
└── 삭제: 법정대리인 또는 본인(14세 이상) 요청 시 즉시 삭제
```

## 13. 데이터 암호화 및 보안 기술 사양

### 13.1 암호화 계층

| 계층 | 방식 | 적용 범위 | 키 관리 |
|------|------|---------|--------|
| **전송 암호화 (In-Transit)** | TLS 1.3 | 모든 클라이언트-서버 통신 | 서버 인증서 (Let's Encrypt) |
| **저장 암호화 (At-Rest) — DB 레벨** | 디스크 암호화 (LUKS/dm-crypt) | PostgreSQL 데이터 디렉토리 전체 | OS 레벨 키 관리 |
| **저장 암호화 (At-Rest) — 컬럼 레벨** | AES-256-GCM | 위경도 필드 (latitude, longitude) | 환경변수 + KMS |
| **로컬 저장 암호화** | SQLCipher (AES-256-CBC) | 클라이언트 SQLite DB | 기기 Keystore 파생 키 |
| **백업 암호화** | AES-256-GCM | PostgreSQL 백업 파일 | 별도 백업 키 (KMS) |

### 13.2 키 로테이션 절차

```
키 로테이션 주기: 90일

1. 새 키 생성 (KMS에서 자동 생성)
2. 새 키로 새 데이터 암호화 시작
3. 배경 마이그레이션: 이전 키로 암호화된 데이터를 새 키로 재암호화
   └── 일일 배치 (서버 부하 최저 시간, 03:00 UTC)
   └── 처리량: 100만 레코드/배치
4. 마이그레이션 완료 후 이전 키 비활성화
5. 이전 키는 1세대 보존 후 폐기
```

## 14. 데이터 주체 권리 이행 절차

위치정보법 §24(위치정보 주체의 권리) 및 개인정보보호법 제35~37조에 따른 데이터 주체의 권리 이행:

### 14.1 열람 청구 (Right of Access)

```
열람 청구 처리:
├── 청구 경로: 설정 > 내 데이터 관리 > 위치 데이터 열람
├── 제공 형태: 이동기록 화면 (앱 내) + GPX 파일 다운로드
├── 처리 기한: 10일 이내 (개인정보보호법 §35④)
├── 제공 범위: 청구자 본인의 전체 위치 데이터
└── 접근 로그: 열람 청구 사실을 TB_LOCATION_ACCESS_LOG에 기록
```

### 14.2 동의 철회 (Right to Withdraw Consent)

```
동의 철회 처리:
├── 철회 경로: 설정 > 위치 설정 > 위치 수집 동의 철회
├── 즉시 효과:
│   ├── 위치 수집 즉시 중단
│   ├── Firebase RTDB 위치 데이터 삭제
│   ├── 지도 마커 "위치 비공유" 상태로 전환
│   └── 이동기록 저장 중단
├── 기존 데이터: 보존 기간까지 보존 (별도 삭제 청구 필요)
├── 서비스 제한 안내:
│   "위치 수집을 중단하면 실시간 위치 공유, 지오펜스, SOS 위치 전송이
│    비활성화됩니다. 계속하시겠습니까?"
└── SOS 제한 경고:
    "⚠️ SOS 발송 시 위치가 포함되지 않습니다."
```

### 14.3 삭제 청구 (Right to Erasure)

```
삭제 청구 처리:
├── 청구 경로: 설정 > 내 데이터 관리 > 위치 데이터 삭제
├── 삭제 범위 선택:
│   ├── 특정 여행의 이동기록 삭제
│   ├── 특정 날짜의 이동기록 삭제
│   └── 전체 위치 데이터 삭제
├── 처리 기한: 10일 이내 (개인정보보호법 §36②)
├── 삭제 불가 예외:
│   ├── 진행 중인 SOS 관련 데이터
│   ├── 법적 보존 의무 데이터 (접근 로그 6개월)
│   └── 수사기관 보전 요청 데이터
├── 처리 완료 통보: 앱 내 알림 + SMS
└── 감사 로그: 삭제 청구 및 처리 결과 기록
```

### 14.4 일시 중지 (Temporary Suspension)

```
위치 공유 일시 중지:
├── 중지 경로: 설정 > 위치 설정 > 위치 공유 일시 중지
├── 중지 범위: 서버 전송 중단 (로컬 수집은 등급에 따라 차등)
├── 중지 기간: 사용자 지정 (1시간~24시간) 또는 수동 해제
├── 가디언 통보: 일시 중지 시작/해제 시 가디언에게 알림
├── 안전 최우선 등급 제한:
│   └── 미성년자 대상: 가디언 일시 중지 차단 (SOS 원칙 §11)
└── SOS 예외: 일시 중지 중에도 SOS 발동 시 위치 강제 공유
```

### 15.2 관련 테이블: TB_DATA_DELETION_LOG (실제 구현) / TB_DATA_DELETION_BATCH (설계)

> ⚠️ **v3.4 실제 구현 반영**: DB 설계 v3.4 §4.34에 정의된 실제 테이블은 `TB_DATA_DELETION_LOG`이며,
> 아래 `TB_DATA_DELETION_BATCH` 설계와 목적이 상이하다.
> - `TB_DATA_DELETION_LOG`: 개별 삭제 이력 기록 (account_soft_delete | account_hard_delete | location_batch_delete)
> - `TB_DATA_DELETION_BATCH`: 배치 단계 추적 (DETECTED→SOFT_DELETED→ANONYMIZED→HARD_DELETED) — 미구현, 데이터 생명주기 시스템 구축 시 추가 필요

**▼ DB v3.4 §4.34 실제 구현 (TB_DATA_DELETION_LOG)**
```sql
CREATE TABLE tb_data_deletion_log (
    deletion_id    BIGINT PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
    user_id        VARCHAR(128) NOT NULL,            -- FK 아님 (삭제 후에도 기록 보존)
    deletion_type  VARCHAR(30) NOT NULL,
        -- account_soft_delete | account_hard_delete |
        -- location_batch_delete | trip_data_delete | consent_withdrawal
    affected_tables TEXT[],
    record_count   INTEGER,
    requested_by   VARCHAR(20),                      -- user | system | admin
    executed_at    TIMESTAMPTZ DEFAULT NOW(),
    notes          TEXT
);
```

**▼ 데이터 생명주기 배치 추적 설계 (TB_DATA_DELETION_BATCH — 미구현)**
```sql
CREATE TABLE tb_data_deletion_batch (
    batch_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    stage            VARCHAR(20) NOT NULL,   -- DETECTED / SOFT_DELETED / ANONYMIZED / HARD_DELETED
    data_type        VARCHAR(20) NOT NULL,   -- LOCATION / STAY_POINT / HEARTBEAT / SOS_EVENT
    target_trip_id   UUID,                   -- 특정 여행 대상 (NULL = 전체)
    target_user_id   VARCHAR(128),           -- 특정 유저 대상 (NULL = 전체)
    target_count     INTEGER NOT NULL,       -- 대상 레코드 수
    retention_policy VARCHAR(50),            -- 적용된 보존 정책
    detected_at      TIMESTAMPTZ,
    soft_deleted_at  TIMESTAMPTZ,
    anonymized_at    TIMESTAMPTZ,
    hard_deleted_at  TIMESTAMPTZ,
    freed_space_mb   REAL,
    created_at       TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_deletion_batch_stage ON tb_data_deletion_batch(stage, created_at);
```

### 17.1 모니터링 지표

| 지표 | 경고 임계치 | 위험 임계치 | 대응 |

| DB 파티션 여유 | 1개월 미만 | 파티션 없음 | 파티션 자동 생성 점검 |

## 18. 에러 및 엣지케이스 처리

### 18.1 수집 단계 에러

| 에러 상황 | 처리 방법 |
|----------|----------|
| GPS 신호 미수신 | 네트워크 위치 폴백 (accuracy 표시), 3분간 미수신 시 "GPS 불가" 이벤트 기록 |
| 위치 권한 런타임 취소 | 수집 즉시 중단, "위치 권한 필요" 배너 표시, 권한 복구 시 자동 재개 |
| 배터리 최적화에 의한 수집 중단 | Foreground Service 유지, WorkManager 백업, 사용자에게 배터리 최적화 예외 안내 |

### 18.2 전송 단계 에러

| 에러 상황 | 처리 방법 |
|----------|----------|
| API 서버 응답 없음 (5xx) | 지수 백오프 재시도 (30초, 1분, 2분, 5분), 최대 10회 |
| 인증 토큰 만료 | Firebase SDK 자동 갱신 → 재전송, 갱신 실패 시 로컬 큐잉 |
| 데이터 형식 오류 (4xx) | 오류 데이터 격리 로깅, 다음 데이터 전송 계속 |
| Firebase RTDB 쓰기 거부 | Security Rules 위반 로깅, 관리자 알림 |

### 18.3 저장 단계 에러

| 에러 상황 | 처리 방법 |
|----------|----------|
| PostgreSQL 연결 실패 | 커넥션 풀 재설정, 3회 재시도, 실패 시 임시 파일 저장 |
| 암호화 키 접근 불가 | 서비스 중단 알림 (위치 저장 불가), 이전 키 폴백 시도 |
| 파티션 미존재 | 자동 파티션 생성 트리거, 실패 시 기본 파티션에 저장 |
| 디스크 공간 부족 | 긴급 삭제 배치 실행, 관리자 알림 |

### 18.4 삭제 단계 에러

| 에러 상황 | 처리 방법 |
|----------|----------|
| 삭제 배치 중 서버 크래시 | 트랜잭션 롤백 → 다음 배치에서 재시도 |
| VACUUM 실패 | pg_repack 대안 사용, 수동 점검 스케줄링 |
| 백업에서 삭제 실패 | 관리자 알림, 수동 백업 정리 프로세스 실행 |
| 법적 보전 요청과 삭제 충돌 | 삭제 일시 중단, 법무팀 확인 후 재개 |
