## 1. 세이프트립 마스터 거버넌스 개요

### 1.1 SafeTrip의 핵심 가치 선언

SafeTrip은 **여행자의 안전을 기술로 보장하는 위치 기반 안전 플랫폼**이다. 일반 여행 앱이 편의(convenience)를 최우선으로 추구하는 반면, SafeTrip은 다음 세 가지 가치를 동시에 달성해야 한다.

| 핵심 가치 | 정의 | 위험 실패 시나리오 |
|-----------|------|-------------------|
| **생명 안전 (Life Safety)** | 긴급 상황에서 여행자의 생명을 보호하기 위한 SOS, 위치 추적, 긴급 연락이 100% 신뢰할 수 있어야 한다 | SOS 미발송, 위치 오차로 구조 실패, 긴급 연락처 부재 |
| **위치 데이터 보호 (Location Privacy)** | 실시간 위치라는 민감한 개인정보를 법적·윤리적 기준에 따라 수집·저장·공유·삭제해야 한다 | 위치 데이터 유출, 동의 없는 추적, 불법 데이터 보관 |
| **정보 정확성 (Information Accuracy)** | 안전 경보, 외교부 여행경보, 의료·입국 정보 등 안전에 직결되는 정보가 정확하고 최신이어야 한다 | 해제된 경보 지속 표시, 잘못된 긴급 전화번호, 구버전 입국 정보 |

### 1.2 원칙 문서 거버넌스의 필요성

SafeTrip은 현재 **24개의 원칙/기준 문서**를 보유하고 있으며, 서비스 런칭까지 추가로 **11개 이상의 필수 문서**가 필요하다. 이 문서들은 상호 의존성이 높아, 하나의 변경이 다수의 문서에 연쇄적 영향을 미친다.

**거버넌스 없이 발생하는 문제:**

- 역할 권한이 비즈니스 원칙과 멤버 탭 원칙에서 상충하는 정의를 가질 수 있음
- 법적 필수 문서(개인정보처리방침, LBS 이용약관)가 없는 상태로 기능 개발이 진행될 수 있음
- SOS 원칙 변경 시 영향 받는 6개 이상의 문서가 동기화되지 않을 수 있음
- **가디언 유/무료 경계 변경이 과금·DB·UI 문서에 일관되게 반영되지 않을 수 있음**
- **여행 기간 정책(15일 제한)이 과금·일정·온보딩 문서에서 불일치할 수 있음**

### 1.3 문서 계층 분류 체계

SafeTrip의 전체 원칙 문서를 **5개 계층(Tier)**으로 분류한다. 상위 계층의 문서가 하위 계층의 기준이 되며, 상충 시 상위 문서가 우선한다.

```
Tier 0 ── 법적 기반 (Legal Foundation)
│         서비스 런칭의 법적 전제 조건. 이 문서 없이는 앱 출시 불가.
│
Tier 1 ── 비즈니스 근간 (Business Core)
│         SafeTrip의 비즈니스 모델, 역할 체계, 데이터 구조를 정의하는 최상위 설계 문서.
│         모든 하위 문서의 기준점.
│
Tier 2 ── 시스템 설계 (System Design)
│         아키텍처, 화면 구성, 공통 컴포넌트 등 시스템 전반의 설계 원칙.
│
Tier 3 ── 기능 원칙 (Feature Principles)
│         개별 탭, 화면, 기능의 상세 설계 원칙.
│
Tier 4 ── 운영 및 품질 (Operations & Quality)
          서비스 운영, 테스트, 성능, 고객 지원에 관한 원칙.
```

### 4.2 핵심 의존성 체인 (Critical Dependencies)

SafeTrip에서 가장 중요한 **네 가지** 의존성 체인을 다음과 같이 정의한다.

**체인 A: 위치 추적 체인 (Location Chain)**
```
개인정보처리방침
  └→ LBS 이용약관
       └→ 위치 데이터 수집·저장·삭제 정책
            ├→ 비즈니스 원칙 v5.1 (프라이버시 등급, 위치 공유 모드)
            │    ├→ 설정 메뉴 (위치 공유 설정 UI)
            │    ├→ 지도 원칙 (레이어 시스템, 마커 표시)
            │    ├→ 멤버탭 (멤버 위치 상태)
            │    └→ 이동기록 (과거 경로, 프라이버시 마스킹)
            └→ 국제 데이터 처리 (GDPR, 국외 이전)
```

**체인 B: 긴급 안전 체인 (Safety Chain)**
```
서비스 이용약관 (면책 범위)
  └→ SOS 원칙 (6가지 시나리오)
       ├→ 긴급 구조기관 연동 원칙 (실제 구조 연결)
       │    └→ 안전가이드 (긴급연락 탭)
       ├→ 사고 대응·면책 원칙 (사후 대응)
       ├→ 알림 원칙 (긴급 알림 전달)
       └→ 앱 성능 원칙 (SOS SLA 99.9%)
```

**체인 C: 미성년자 보호 체인 (Minor Protection Chain)**
```
개인정보처리방침 (아동 조항)
  └→ 미성년자 보호 원칙
       ├→ 온보딩 시나리오 (학생 참여 B, 가디언 참여 C)
       ├→ 비즈니스 원칙 v5.1 (가디언 역할, 안전 최우선 등급)
       │    ├→ 멤버탭 (가디언 카드, 미성년자 표시)
       │    └→ 설정 메뉴 (가디언 일시 중지 제한)
       └→ 위치 데이터 정책 (미성년자 데이터 보유 기간 30일)
```

**체인 D: 가디언 과금 체인 (Guardian Pricing Chain)** ← v2.0 신규
```
비즈니스 원칙 v5.1 §03.1 (무료 2명 + 유료 3명 구분)
  ├→ 서비스 이용약관 (가디언 추가 과금 고지)
  ├→ 설정 메뉴 (가디언 관리 UI + 과금 안내 모달)
  ├→ 멤버탭 (가디언 무료/유료 표시)
  ├→ 온보딩 시나리오 (가디언 연결 시 과금 분기)
  ├→ DB 설계 (tb_guardian_link.is_paid, payment_id)
  └→ B2B 프레임워크 §12.4 (기업 가디언 일괄 연결)
       └→ 초대코드 원칙 (B2B CSV 일괄 생성)
```

| 기존 기능의 상세 추가 | 마이너 (v1.0 → v1.1) | 검증 항목 추가 | 관련 팀 고지 |
| 기능 추가/삭제, 규칙 변경 | 메이저 (v1.x → v2.0) | 역할 체계 변경 | 전파 대상 문서 전체 검토 |
| 상위 문서 메이저 변경 반영 | 마이너 이상 | 비즈니스 원칙 v5.1 반영 | 해당 하위 문서 일괄 갱신 |
| **인라인 변경 표기 정책** | — | — | **본문에는 최종 확정 내용만 기재, 변경 이력은 부록에 일괄 정리** (v5.1 방식 준용) |

### 5.2 문서 품질 체크리스트

모든 원칙 문서는 작성 완료 시 다음 항목을 충족해야 한다.

| # | 체크 항목 |
|---|----------|
| 1 | 문서 목적과 적용 범위가 명시되어 있다 |
| 2 | 기준 문서(상위 문서)가 명시되어 있다 — **비즈니스 원칙은 v5.1을 참조해야 한다** |
| 3 | 역할별(캡틴/크루장/크루/가디언) 접근 권한이 정의되어 있다 |
| 4 | 프라이버시 등급별(안전 최우선/표준/프라이버시 우선) 동작 차이가 정의되어 있다 |
| 5 | 에러 및 엣지케이스 처리가 포함되어 있다 |
| 6 | 검증 체크리스트가 포함되어 있다 |
| 7 | 기존 문서 대비 변경/확장 사항이 명시되어 있다 |
| 8 | DB 스키마가 필요한 경우 테이블 구조가 포함되어 있다 — **v5.1 스키마 변경 반영 여부 확인** |
| 9 | 구현 우선순위(P0~P3)와 Phase 배치가 포함되어 있다 |
| 10 | 오프라인 동작이 해당되는 경우 대응 방안이 포함되어 있다 |
| 11 | **가디언 과금(무료/유료)이 해당되는 경우 과금 분기가 포함되어 있다** ← v2.0 추가 |
| 12 | **여행 기간 제한(15일)이 해당되는 경우 제한 처리가 포함되어 있다** ← v2.0 추가 |

### 5.3 정기 검토 주기

| 문서 계층 | 검토 주기 | 트리거 |
|----------|----------|--------|
| Tier 0 (법적 기반) | 6개월 또는 법률 변경 시 | 개인정보보호법, 위치정보법 개정 |
| Tier 1 (비즈니스 근간) | 분기 또는 비즈니스 모델 변경 시 | 과금 구조, 역할 체계, B2B 계약 변경, **가디언 과금 정책 변경** |
| Tier 2 (시스템 설계) | 분기 또는 아키텍처 변경 시 | 서버 마이그레이션, 신규 API 도입 |
| Tier 3 (기능 원칙) | 해당 기능 메이저 업데이트 시 | 기능 추가/삭제, UX 리디자인, **상위 문서 메이저 버전 변경** |
| Tier 4 (운영 및 품질) | 분기 또는 인시던트 발생 시 | SLA 미달, 사고 대응 후 회고 |

### 5.4 상위 문서 메이저 변경 시 하위 문서 갱신 프로세스

비즈니스 원칙 v5.1 변경과 같은 상위 문서 메이저 변경 발생 시, 다음 프로세스를 따른다.

```
1. 변경 영향 분석
   ├── §4.4 변경 전파 규칙에 따라 영향 받는 문서 목록 식별
   └── §4.5 갱신 매트릭스에서 구체적 갱신 내용 확인

2. 갱신 우선순위 결정
   ├── 🔴 즉시: DB 스키마, 서비스 이용약관 등 법적·기술적 필수 변경
   ├── 🟠 2주 내: 사용자 대면 UI 관련 문서 (설정 메뉴, 멤버탭, 온보딩)
   └── 🟡 4주 내: 나머지 참조 문서 마이너 업데이트

3. 갱신 실행
   ├── 각 문서의 기준 문서 버전 표기 갱신
   ├── 변경 사항 반영 (테이블, 규칙, 검증 항목)
   └── 품질 체크리스트(§5.2) 재검증

4. 갱신 완료 확인
   ├── 문서 버전 마이너 이상 증가
   └── 변경 이력 기록 (각 문서 부록)
```

> **v3.0 실행 결과**: 비즈니스 원칙 v5.1 기준으로 전체 38개 하위 문서의 정합성 검증을 완료하였다. 모든 하위 문서가 v5.1 참조를 확인하였으며, §4.5 갱신 매트릭스에 반영 완료 상태를 기록하였다.
