## 1. 문서 목적 및 적용 범위

### 1.1 목적

본 문서는 SafeTrip의 SOS 긴급 안전 시스템 전체를 정의한다. SOS는 SafeTrip의 가장 중요한 핵심 기능으로, 어떤 설정이나 상태보다 우선한다.

### 1.2 적용 범위

- SafeTrip 앱의 SOS 버튼, Watchdog 자동 감지, 수신자 알림 전체
- 역할: 캡틴/크루장/크루(발송 가능), 가디언(수신 전용)
- 여행 상태: `active` 상태에서만 SOS 발송 가능

## 2. SOS 최상위 원칙

> **SOS는 모든 프라이버시 설정을 무시하는 최상위 안전 기능이다.** (비즈니스 원칙 §05.1)

| 원칙 | 내용 |
|------|------|
| **프라이버시 무시** | SOS 발동 시 프라이버시 등급·위치 공유 설정에 무관하게 정확한 위치 강제 포함 |
| **법적 근거** | 위치정보법 제29조 (긴급구조), 개인정보보호법 §15①③ (급박한 위험 방지) |
| **안전 보조 수단** | SafeTrip SOS는 보조 수단이며, 119/112 등 공식 구조기관 직접 연락이 최우선 |
| **전달 보장** | SOS 알림은 FCM 최고 우선순위(PRIORITY_HIGH)로 발송, 오프라인 시 SMS 폴백 |
| **해제 통제** | SOS 발송자 본인 또는 캡틴만 해제 가능 |

## 3. SOS 발동 방법

### 3.1 수동 발동 (사용자 직접)

| 발동 방법 | 조건 | 설명 |
|----------|------|------|
| **SOS 버튼 3초 장누르기** | active 여행 + 멤버 역할 | 오발송 방지를 위한 3초 홀드 |
| **카운트다운 취소** | 3초 미만 해제 | 카운트다운 중 손 떼면 취소 |

### 3.2 자동 발동 (Watchdog)

| 발동 조건 | 임계값 | 설명 |
|----------|--------|------|
| **Heartbeat 미수신** | 설정된 주기 × 3 누락 | 사용자 무응답 감지 |
| **지오펜스 이탈 + 무응답** | 이탈 후 5분 무응답 | 위험 지역 이탈 자동 감지 |
| **극단적 이동 감지** | 비정상 이동 패턴 | Watchdog 알고리즘 |

### 3.3 발동 시 확인 UI

```
[SOS 버튼 장누르기]
    ↓
카운트다운 오버레이 표시:
┌─────────────────────────┐
│  3초 후 SOS가 발송됩니다  │
│  ████████░░░░░  2초     │
│  [취소]                  │
└─────────────────────────┘
    ↓ (3초 경과)
SOS 발동
```

## 4. 6가지 SOS 시나리오

### 시나리오 A — 사용자 직접 SOS 발송 (일반)

| 단계 | 동작 |
|------|------|
| 1 | SOS 버튼 3초 장누르기 → 카운트다운 표시 |
| 2 | SOS 확정 → 위치 수집 (프라이버시 무관, 정확한 GPS) |
| 3 | API POST `/api/v1/trips/{tripId}/sos` → 서버 처리 |
| 4 | FCM PRIORITY_HIGH → 전체 멤버 + 연결 가디언 전원 |
| 5 | 채팅탭에 SOS 시스템 메시지 자동 삽입 |
| 6 | 긴급 구조기관 연동 화면 표시 (119/112/대사관) |

### 시나리오 B — 자동 Watchdog SOS

| 단계 | 동작 |
|------|------|
| 1 | Heartbeat 3주기 연속 누락 감지 |
| 2 | 사용자 기기에 "괜찮으세요?" 확인 알림 발송 |
| 3 | 2분 내 응답 없으면 → 자동 SOS 발동 |
| 4 | 이후 시나리오 A 3~6 단계와 동일 |
| 5 | SOS 메시지에 "자동 감지 SOS" 표시 |

### 시나리오 C — 지오펜스 이탈 SOS

| 단계 | 동작 |
|------|------|
| 1 | 지오펜스 이탈 감지 → 캡틴/크루장에게 이탈 알림 |
| 2 | 이탈 후 5분 내 사용자 무응답 → 자동 SOS 발동 |
| 3 | SOS 메시지에 "지오펜스 이탈 후 미응답" 표시 |
| 4 | 이후 시나리오 A 3~6 단계와 동일 |

### 시나리오 D — 가디언 긴급 알림 (별도 이벤트)

> ⚠️ 가디언 긴급 알림은 SOS가 아닌 별도 이벤트(`GUARDIAN_ALERT`)이다.

| 단계 | 동작 |
|------|------|
| 1 | 가디언이 연결 멤버에게 긴급 알림 발송 |
| 2 | 이벤트 타입: `GUARDIAN_ALERT` (SOS와 구분) |
| 3 | 수신 대상: 연결된 멤버 + 해당 멤버의 캡틴/크루장 |
| 4 | FCM 강도: SOS와 동일한 PRIORITY_HIGH |
| 5 | 멤버 앱에 풀스크린 알림 표시 |

### 시나리오 E — 미성년자 SOS (§11 특칙 적용)

| 단계 | 동작 |
|------|------|
| 1 | 미성년자 멤버가 SOS 발동 |
| 2 | 위치 정보 즉시 수집 (safety_first 등급 강제이므로 항상 허용) |
| 3 | 가디언(부모/학부모)에게 우선 알림 발송 |
| 4 | 대사관 연결 안내 화면 자동 표시 |
| 5 | SOS 해제는 캡틴만 가능 (미성년자 본인 해제 불가) |

### 시나리오 F — 오프라인 SOS

| 단계 | 동작 |
|------|------|
| 1 | 네트워크 없는 상태에서 SOS 발동 |
| 2 | SMS 발송: 긴급 연락처 + 연결된 가디언 전화번호 |
| 3 | 로컬 알람 울리기 (기기 최대 볼륨) |
| 4 | SQLite 큐잉 → 네트워크 복귀 시 API 재전송 (Priority 1) |
| 5 | 화면에 긴급 연락처 번호 크게 표시 |

## 5. Heartbeat/Watchdog 시스템

### 5.1 Heartbeat 주기

| 상황 | Heartbeat 주기 |
|------|:-------------:|
| 온라인 + 여행 active | 30초 |
| 오프라인 모드 | 120초 (배터리 절약) |
| SOS 발동 중 | 10초 (위치 추적 강화) |
| 배터리 < 20% | 60초 (절전 모드) |

### 5.2 Watchdog 감지 로직

```
서버 Watchdog 프로세스 (60초 주기 실행):
    foreach active_trip:
        foreach active_member:
            last_heartbeat = select max(recorded_at) from tb_location where user_id = member
            threshold = now() - (heartbeat_interval * 3)
            if last_heartbeat < threshold:
                → "괜찮으세요?" 확인 알림 발송
                → 2분 대기 후 재확인
                → 여전히 무응답: 자동 SOS 발동
```

### 5.3 Heartbeat 저장

```sql
-- tb_location 테이블의 최신 레코드로 Heartbeat 추적
-- 별도 테이블 없이 위치 업데이트 주기를 Heartbeat로 활용
SELECT MAX(recorded_at) AS last_heartbeat
FROM tb_location
WHERE user_id = :user_id
  AND trip_id = :trip_id;
```

## 6. SOS 알림 전달 체인

### 6.1 전달 순서

```
SOS 발동
    ↓
1단계: 서버 SOS 이벤트 생성 (tb_sos_event)
    ↓
2단계: FCM PRIORITY_HIGH 알림 발송
    ├── 그룹 전체 멤버 (캡틴 + 크루장 + 크루)
    └── 발송자에 연결된 가디언 전원 (무료/유료 모두)
    ↓
3단계: 채팅탭 SOS 시스템 메시지 자동 삽입
    메시지 형식: "[SOS] {사용자명}이 긴급 도움을 요청했습니다. 위치: {주소}"
    ↓
4단계: 앱 내 SOS 알림 배너 표시
    (수신자가 앱 열면 자동으로 발신자 위치로 지도 이동)
    ↓
5단계: (자동) 긴급 구조기관 연동 화면 표시 (발신자 기기)
```

### 6.2 알림 페이로드

```json
{
  "title": "🆘 SOS 긴급 알림",
  "body": "{사용자명}이 긴급 도움을 요청했습니다",
  "data": {
    "event_type": "SOS",
    "sos_event_id": "{id}",
    "trip_id": "{tripId}",
    "sender_user_id": "{userId}",
    "latitude": "{lat}",
    "longitude": "{lng}",
    "triggered_at": "{ISO8601}"
  },
  "android": { "priority": "high", "notification": { "sound": "sos_alert" } },
  "apns": { "headers": { "apns-priority": "10" }, "payload": { "aps": { "sound": "sos_alert.caf" } } }
}
```

### 6.3 가디언 SOS 수신 (비즈니스 원칙 §03.1 반영)

- **무료 가디언 (1~2번째)**: SOS 알림 수신 ✅
- **유료 가디언 (3번째~)**: SOS 알림 수신 ✅
- **가디언은 SOS 과금 여부와 무관하게 모두 수신한다.** SOS는 안전 기능이므로 과금 차등 없음.

## 7. SOS 해제 프로토콜

### 7.1 해제 권한

| 역할 | 해제 가능 |
|------|:--------:|
| SOS 발송자 본인 | ✅ |
| 캡틴 | ✅ |
| 크루장 | ❌ (발송자/캡틴만 가능) |
| 크루 | ❌ |
| 가디언 | ❌ |

> 미성년자 SOS의 경우 미성년자 본인도 해제 불가 — 캡틴만 해제 가능 (§11)

### 7.2 해제 절차

```
[SOS 해제 버튼 탭]
    ↓
확인 다이얼로그:
"SOS를 해제하시겠습니까?
실제 응급 상황이 아님을 확인하셨나요?"
[취소] [해제 확인]
    ↓ (해제 확인)
PATCH /api/v1/trips/{tripId}/sos/{sosEventId}/resolve
    ↓
FCM: 전체 수신자에게 "SOS 해제" 알림
    ↓
채팅탭에 "SOS 해제됨" 시스템 메시지 삽입
    ↓
바텀시트 잠금 해제, peek 상태로 복원
```

### 7.3 오발송(가짜 SOS) 처리

- 해제 시 "오발송이었나요?" 옵션 선택 가능 → `tb_sos_event.false_alarm = TRUE` 기록
- 오발송이더라도 수신자에게 "SOS 해제됨 (오발송 확인)" 알림 발송

## 8. SOS 시 자동 화면 전환

### 8.1 발송자 기기 화면 변화

```
SOS 발동
    ↓
1. 바텀시트 → collapsed 강제 전환 (fade 150ms) — 바텀시트 원칙 §10
    ↓
2. 지도 → 발송자 현재 위치로 자동 센터링
    ↓
3. SOS 오버레이 화면 표시 (전체 화면 최상단 z-index):
   ┌─────────────────────────┐
   │  🆘 SOS 발송됨           │
   │  위치: {주소}            │
   │  시각: {시간}            │
   │  알림 발송: 멤버 N명, 가디언 N명 │
   │  [긴급 구조기관 연결]    │
   │  [SOS 해제]              │
   └─────────────────────────┘
```

### 8.2 수신자 기기 화면 변화

```
SOS FCM 수신
    ↓
앱 열림 시: SOS 발신자 위치로 지도 자동 센터링
    ↓
멤버탭: SOS 발신자 카드 빨간색 강조 + "SOS 발송 중" 배지
    ↓
상단 배너: "[이름] SOS 발송 중 — {시간 경과}" 표시
```

## 10. 프라이버시 등급별 동작 차이

> SOS는 모든 프라이버시 설정을 무시하므로 등급별 차이가 없다.

| 등급 | SOS 발동 | SOS 위치 정보 | 비고 |
|:----:|:-------:|:------------:|------|
| 안전 최우선 | ✅ | 정확한 GPS 좌표 | 평상시도 공유 중 |
| 표준 | ✅ | **정확한 GPS 좌표** (평상시 위치 공유 설정 무시) | §05.1 프라이버시 무시 |
| 프라이버시 우선 | ✅ | **정확한 GPS 좌표** (평상시 위치 숨김 무시) | §05.1 프라이버시 무시 |

## 11. 미성년자 SOS 특칙

비즈니스 원칙 §10 및 미성년자 보호 원칙 (#05) 기반.

| 항목 | 규칙 |
|------|------|
| 미성년자 SOS 발동 | ✅ 가능 (성인과 동일) |
| SOS 시 가디언 알림 | 연결된 모든 가디언 (부모/학부모 포함) 우선 알림 |
| 대사관 연결 | 미성년자 SOS 시 대사관 연결 화면 자동 표시 |
| SOS 해제 | 캡틴만 가능 (미성년자 본인 해제 불가) |
| 여행 등급 | 미성년자 포함 여행은 safety_first 강제 → SOS 발동 시 항상 정확한 위치 |

## 12. 에러 및 엣지케이스 처리

| 케이스 | 처리 방법 |
|--------|----------|
| **SOS 발송 실패 (네트워크)** | 오프라인 시나리오 F 자동 전환 (SMS + 로컬 큐잉) |
| **FCM 알림 미전달** | 서버 재시도 (30초 간격 3회), 이후 SMS 폴백 |
| **SOS 중 앱 강제 종료** | 서버의 SOS 이벤트는 유지, 재진입 시 SOS 화면 표시 |
| **오발송 (즉시 취소)** | 3초 카운트다운 중 취소 → 발송 안 됨 |
| **동시 다중 SOS** | 각각 별도 이벤트로 처리, 화면에 발신자별 배너 |
| **SOS 발동 중 여행 종료** | SOS 이벤트 유지, 여행 종료 후에도 SOS 해제 전까지 알림 지속 |
| **가디언 연결 없는 멤버 SOS** | 그룹 멤버에게만 알림, 가디언 없음 표시 |
| **GPS 신호 없음 (SOS 발동)** | 마지막 알려진 위치 + "GPS 신호 없음" 메시지 포함 |

## 13. 오프라인 대응

시나리오 F 참조. 오프라인 동작 통합 원칙 (#16) §3.2 연동.

| 상황 | 동작 |
|------|------|
| 완전 오프라인 SOS | SMS + 로컬 알람 + SQLite 큐잉 |
| 불안정 네트워크 SOS | 재시도 3회 후 SMS 폴백 |
| 복귀 후 SOS 큐 전송 | Priority 1 — 가장 먼저 동기화 |

## 14. DB 스키마

### 14.1 SOS 이벤트 테이블 (DB 설계 v3.4 참조)

```sql
-- tb_sos_event (참조: DB 설계 v3.4 §4.x)
CREATE TABLE IF NOT EXISTS tb_sos_event (
    id              UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    trip_id         UUID         NOT NULL REFERENCES tb_trip(id) ON DELETE CASCADE,
    sender_id       UUID         NOT NULL REFERENCES tb_user(id),
    trigger_type    VARCHAR(20)  NOT NULL DEFAULT 'manual',
                    -- manual | watchdog | geofence_exit
    status          VARCHAR(20)  NOT NULL DEFAULT 'active',
                    -- active | resolved | false_alarm
    latitude        DECIMAL(10,8),
    longitude       DECIMAL(11,8),
    address         TEXT,
    false_alarm     BOOLEAN      DEFAULT FALSE,
    resolved_by     UUID         REFERENCES tb_user(id),
    resolved_at     TIMESTAMPTZ,
    sms_sent        BOOLEAN      DEFAULT FALSE,
    offline_queued  BOOLEAN      DEFAULT FALSE,
    created_at      TIMESTAMPTZ  DEFAULT now(),
    updated_at      TIMESTAMPTZ  DEFAULT now()
);

CREATE INDEX idx_sos_event_trip ON tb_sos_event(trip_id, created_at DESC);
CREATE INDEX idx_sos_event_sender ON tb_sos_event(sender_id, created_at DESC);
CREATE INDEX idx_sos_event_active ON tb_sos_event(trip_id) WHERE status = 'active';
```
