## 1. 문서 목적 및 적용 범위

### 1.1 목적

SafeTrip은 해외 여행 중 네트워크가 불안정한 환경에서도 핵심 안전 기능이 작동해야 한다. 본 문서는 **여러 기능 문서에 분산된 오프라인 대응을 통합**하여 일관된 오프라인 전략을 정의한다. 모든 기능 원칙 문서(Tier 3)는 본 문서의 오프라인 정책을 준수해야 한다.

### 1.2 적용 범위

- SafeTrip 모바일 앱(Flutter)의 모든 기능
- 네트워크 연결이 없거나 불안정한 환경
- 해외 여행 중 현지 네트워크 불안정 시나리오
- 비행기 탑승, 산악·도서 지역, 터널 등 일시적 오프라인 상황

### 1.3 핵심 원칙

| 원칙 | 설명 |
|------|------|
| **안전 우선** | SOS 기능은 오프라인에서도 반드시 동작해야 한다 |
| **데이터 손실 방지** | 오프라인 중 생성된 데이터는 로컬 큐잉 후 자동 동기화한다 |
| **사용자 인지** | 오프라인 상태를 사용자에게 명확히 알려야 한다 |
| **배터리 효율** | 오프라인 시 GPS 수집 주기를 최적화하여 배터리 소모를 줄인다 |

## 2. 오프라인 감지 기준

### 2.1 네트워크 상태 분류

| 상태 | 조건 | 대응 |
|------|------|------|
| **온라인 (Online)** | RSSI >= -80dBm, 패킷 손실 < 5% | 정상 동작 |
| **불안정 (Degraded)** | RSSI -80 ~ -95dBm, 패킷 손실 5~20%, 응답 > 3s | 재시도 로직 활성화, 오프라인 배너 표시 |
| **오프라인 (Offline)** | 네트워크 없음, 패킷 손실 > 20%, 응답 없음 | 오프라인 모드 전환 |

### 2.2 오프라인 감지 로직

```dart
// Flutter ConnectivityPlus + 실제 API 헬스체크 조합
class NetworkDetector {
  // 1단계: OS 수준 네트워크 감지
  ConnectivityResult connectivityResult;

  // 2단계: 실제 HTTP 헬스체크 (30초 주기)
  Future<bool> isActuallyOnline() async {
    try {
      final response = await http
        .get(Uri.parse('${API_BASE}/health'))
        .timeout(Duration(seconds: 5));
      return response.statusCode == 200;
    } catch (e) {
      return false;
    }
  }

  // 3단계: 3회 연속 실패 시 오프라인 상태로 전환
  int consecutiveFailures = 0;
  static const int OFFLINE_THRESHOLD = 3;
}
```

### 2.3 상태 전환 규칙

- **온라인 → 오프라인**: 헬스체크 3회 연속 실패 시
- **오프라인 → 온라인**: 헬스체크 1회 성공 시 (즉시 동기화 시작)
- **불안정 상태**: 2회 연속 실패 시 진입, 2회 연속 성공 시 온라인 복귀

### 3.2 SOS 오프라인 폴백 (가장 중요)

SOS는 네트워크 없이도 반드시 작동해야 한다.

```
SOS 발동
    ↓
네트워크 확인
    ├── 온라인: API POST /api/v1/sos → FCM 푸시 → 가디언 알림
    └── 오프라인:
            ├── 1순위: SMS 발송 (긴급 연락처 + 가디언 전화번호)
            │         메시지: "[SafeTrip SOS] {사용자명}이 긴급 도움을 요청합니다.
            │                  위치: {GPS좌표} | 여행: {여행명}"
            ├── 2순위: 로컬 알람 울리기 (기기 최대 볼륨)
            ├── 3순위: SQLite 큐잉 → 네트워크 복귀 시 재전송
            └── 4순위: 화면에 긴급 연락처 번호 크게 표시
```

> SMS 발송은 기기 SMS API(Flutter `telephony` 패키지)를 직접 사용한다. SMS 권한이 없는 경우 사용자에게 권한 요청 안내.

### 3.3 위치 데이터 큐잉

| 항목 | 값 |
|------|-----|
| **최대 큐잉 시간** | 72시간 |
| **최대 큐잉 건수** | 8,640건 (5분 주기 × 72시간 × 1회) |
| **큐 가득 찼을 때** | 가장 오래된 데이터부터 삭제 |
| **업로드 우선순위** | 최신 → 과거 순 (최근 위치가 중요) |
| **배치 업로드** | 복귀 시 100건씩 배치 전송 |

### 3.4 채팅 큐잉

| 항목 | 값 |
|------|-----|
| **최대 큐잉 건수** | 100건 |
| **큐 가득 찼을 때** | 더 이상 발송 불가 안내 |
| **메시지 표시 상태** | "전송 대기 중" 아이콘 표시 |
| **발송 순서** | FIFO (먼저 보낸 것 먼저 전송) |

## 4. 복귀 시 동기화 우선순위

비즈니스 원칙 §11.2에 따른 동기화 순서:

```
네트워크 복귀 감지
    ↓
[Priority 1] SOS 이벤트 큐 → API 재전송 (즉시)
    ↓
[Priority 2] 가디언 긴급 알림/요청 큐
    ↓
[Priority 3] 위치 데이터 배치 업로드 (100건씩)
    ↓
[Priority 4] 일정 변경 동기화 (충돌 감지 포함)
    ↓
[Priority 5] 채팅 메시지 발송 큐
    ↓
[Priority 6] 일반 이벤트/로그 업로드
    ↓
[완료] 오프라인 배너 제거, 정상 상태 복원
```

### 4.1 동기화 트랜잭션 관리

- 각 Priority 단계는 독립 트랜잭션으로 처리한다.
- 단계 실패 시 해당 단계만 재시도하며, 다음 단계는 계속 진행한다.
- 동기화 상태는 `tb_sync_queue_offline` 테이블(SQLite)에 기록한다.

### 4.2 동기화 완료 알림

동기화 완료 후 사용자에게 토스트 알림:
- 성공: "오프라인 중 {N}건의 데이터가 동기화되었습니다."
- 부분 실패: "일부 데이터 동기화 실패. 재시도 중..."

## 5. SQLite 로컬 DB 스키마

오프라인 중 로컬 저장을 위한 SQLite 테이블 정의.

### 5.1 위치 큐 테이블

```sql
CREATE TABLE IF NOT EXISTS local_location_queue (
    id              INTEGER  PRIMARY KEY AUTOINCREMENT,
    trip_id         TEXT     NOT NULL,
    user_id         TEXT     NOT NULL,
    latitude        REAL     NOT NULL,
    longitude       REAL     NOT NULL,
    accuracy        REAL,
    speed           REAL,
    recorded_at     TEXT     NOT NULL,  -- ISO 8601
    is_synced       INTEGER  DEFAULT 0, -- 0: 미동기화, 1: 동기화완료
    sync_attempted  INTEGER  DEFAULT 0,
    created_at      TEXT     DEFAULT (datetime('now'))
);
CREATE INDEX idx_llq_synced ON local_location_queue(is_synced, recorded_at);
```

### 5.2 SOS 큐 테이블

```sql
CREATE TABLE IF NOT EXISTS local_sos_queue (
    id              INTEGER  PRIMARY KEY AUTOINCREMENT,
    trip_id         TEXT     NOT NULL,
    user_id         TEXT     NOT NULL,
    latitude        REAL,
    longitude       REAL,
    sos_type        TEXT     DEFAULT 'manual',  -- manual, watchdog, geofence
    triggered_at    TEXT     NOT NULL,
    sms_sent        INTEGER  DEFAULT 0,  -- SMS 폴백 발송 여부
    is_synced       INTEGER  DEFAULT 0,
    retry_count     INTEGER  DEFAULT 0,
    created_at      TEXT     DEFAULT (datetime('now'))
);
```

### 5.3 채팅 큐 테이블

```sql
CREATE TABLE IF NOT EXISTS local_chat_queue (
    id              INTEGER  PRIMARY KEY AUTOINCREMENT,
    trip_id         TEXT     NOT NULL,
    sender_id       TEXT     NOT NULL,
    message_type    TEXT     DEFAULT 'text',  -- text, system, ai
    content         TEXT     NOT NULL,
    local_id        TEXT     NOT NULL UNIQUE, -- 로컬 임시 ID
    is_synced       INTEGER  DEFAULT 0,
    retry_count     INTEGER  DEFAULT 0,
    created_at      TEXT     DEFAULT (datetime('now'))
);
```

### 5.4 일정 임시 편집 테이블

```sql
CREATE TABLE IF NOT EXISTS local_schedule_draft (
    id              INTEGER  PRIMARY KEY AUTOINCREMENT,
    schedule_id     TEXT,    -- NULL이면 신규 일정
    trip_id         TEXT     NOT NULL,
    action          TEXT     NOT NULL,  -- create, update, delete
    payload         TEXT     NOT NULL,  -- JSON
    is_synced       INTEGER  DEFAULT 0,
    conflict_status TEXT     DEFAULT 'pending',  -- pending, resolved, conflict
    created_at      TEXT     DEFAULT (datetime('now'))
);
```

### 5.5 캐시 메타데이터 테이블

```sql
CREATE TABLE IF NOT EXISTS local_cache_meta (
    cache_key       TEXT     PRIMARY KEY,  -- 'safety_guide_{country_code}', 'members_{trip_id}'
    data            TEXT     NOT NULL,     -- JSON 직렬화
    cached_at       TEXT     NOT NULL,
    expires_at      TEXT,                  -- NULL이면 만료 없음
    version         INTEGER  DEFAULT 1
);
```

## 6. 충돌 해소 전략

오프라인 중 로컬 편집과 서버 최신 데이터 간 충돌 시:

### 6.1 충돌 유형별 해소 방식

| 데이터 유형 | 충돌 감지 | 해소 전략 |
|----------|---------|---------|
| **일정 (Schedule)** | `updated_at` 타임스탬프 비교 | **서버 우선 (Server-wins)** + 로컬 변경 사항 사용자 알림 |
| **위치 데이터** | 타임스탬프 기반 병합 | **시계열 병합** (타임스탬프 순으로 삽입) |
| **채팅 메시지** | 순서 충돌 | **서버 타임스탬프 기준 정렬**, 로컬 메시지는 실제 전송 시간 반영 |
| **SOS 이벤트** | 중복 방지 | **로컬 unique ID** 기반 중복 제거 |
| **프로필/설정** | 버전 번호 비교 | **서버 우선** (분산 편집 불허) |

### 6.2 일정 충돌 사용자 안내

```
[충돌 감지] "오프라인 중 변경한 일정과 서버의 최신 일정이 다릅니다.
어떤 버전을 사용하시겠습니까?"
    ├── [내 변경 사항 유지] → 서버에 강제 업데이트
    └── [최신 서버 버전 사용] → 로컬 변경 폐기
```

> 충돌 판단 기준: 서버 `updated_at` > 로컬 편집 시작 시간이면 충돌로 간주

## 7. 배터리 절약 정책

### 7.1 네트워크 상태별 GPS 수집 주기

| 상태 | 정상 GPS 주기 | 오프라인 GPS 주기 | 이유 |
|------|:-----------:|:--------------:|------|
| **안전 최우선 등급** | 30초 | 5분 | 로컬 큐잉만 하므로 고주기 불필요 |
| **표준 등급** | 60초 | 5분 | — |
| **프라이버시 우선 등급** | 5분 | 10분 | — |
| **SOS 발동 중** | 10초 | 30초 | SOS 중에도 배터리 고려 |

### 7.2 오프라인 시 비활성화 항목

- 실시간 위치 업로드 (로컬 큐잉으로 대체)
- AI 기능 API 호출 (완전 비활성화)
- MOFA API 갱신 요청 (캐시 데이터 사용)
- FCM 연결 재시도 (OS 수준 위임)
- 헬스체크 API 호출 (30초 → 60초 주기 연장)

### 7.3 배터리 임계값 처리

| 배터리 잔량 | 처리 |
|:---------:|------|
| < 20% | GPS 수집 주기를 표준의 2배로 줄임, 사용자 경고 배너 |
| < 10% | GPS 수집 주기를 표준의 4배로 줄임, 긴급 기능만 유지 |
| < 5% | SOS SMS 발송 대기 모드, 화면 밝기 최소화 안내 |

## 11. 에러 및 엣지케이스 처리

| 케이스 | 처리 방법 |
|--------|----------|
| **SQLite 저장 실패** | 내부 메모리 임시 저장 → 토스트 경고 |
| **SMS 발송 권한 없음 (SOS)** | 권한 요청 화면 표시, 임시로 화면에 긴급번호 표시 |
| **72시간 초과 위치 데이터** | 자동 삭제 (가장 오래된 것부터), 사용자 알림 |
| **큐 동기화 중 앱 종료** | 앱 재시작 시 자동 재시도 |
| **충돌 해소 대기 중 앱 종료** | 충돌 상태 SQLite에 저장, 재시작 시 충돌 해소 UI 표시 |
| **일정 편집 충돌 (양측 수정)** | 충돌 해소 다이얼로그 표시, 캡틴/크루장 우선 |
| **오프라인 중 여행 생성 시도** | 온라인 필요 안내, 진행 불가 |
| **오프라인 상태에서 채팅 100건 초과** | "전송 대기 한도(100건)에 도달했습니다" 안내 |
