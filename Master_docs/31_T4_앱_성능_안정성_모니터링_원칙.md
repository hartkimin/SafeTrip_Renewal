## 1. 문서 목적 및 적용 범위

### 1.1 목적

본 문서는 SafeTrip 서비스의 SLA(Service Level Agreement), 성능 기준, 모니터링 체계, 인시던트 대응 프로세스를 정의한다. 비즈니스 원칙 v5.1 §12.2의 B2B SLA 조항을 실행 수준으로 구체화하고, SOS 원칙(#13)의 안전 최우선 성능 요건을 운영 지침으로 통합하는 것이 주목적이다.

### 1.2 적용 범위

- **클라이언트**: SafeTrip Flutter 앱 (Android/iOS)
- **백엔드**: Node.js/TypeScript API 서버 (포트 3001), Firebase Realtime Database, Firebase Cloud Messaging
- **데이터베이스**: PostgreSQL (54개 테이블, DB 설계 v3.4 기준)
- **대상 사용자**: B2C 개인 사용자, B2B 기업 고객 (학교, 여행사)
- **역할 전체**: 캡틴, 크루장, 크루, 가디언

## 2. 설계 철학 (R1~R5)

| 원칙 | 코드 | 내용 |
|------|:----:|------|
| **Safety First SLA** | R1 | SOS 기능은 최고 등급 SLA를 보장한다. SOS는 모든 프라이버시 설정과 과금 상태를 무시하는 최상위 안전 기능이므로 (비즈니스 원칙 §05.1), SLA 역시 최고 수준(99.9%)으로 유지한다. |
| **차등 성능 기준** | R2 | 기능의 중요도에 따라 SLA를 차등 적용한다. SOS > 위치 추적 > 일반 기능 > 데모 투어 순으로 가용성 목표를 구분하여 운영 자원을 효율적으로 배분한다. |
| **선제적 모니터링** | R3 | 문제가 발생한 후 대응하는 것이 아닌, 임계값 도달 전 감지하고 예방적 조치를 취한다. 에러율·응답 시간·배터리 등 핵심 지표를 실시간으로 추적한다. |
| **투명한 상태 공개** | R4 | B2B 고객(학교, 여행사)에게 SLA 달성률 리포트를 월 1회 자동 발송한다. 장애 발생 시 고객에게 인지 후 24시간 이내 고지한다. |
| **무중단 개선** | R5 | 성능 개선 배포는 서비스 중단 없이 롤링 방식으로 진행한다. P1/P2 장애 해결 후 72시간 이내 회고(postmortem) 문서를 작성하여 재발 방지 사이클을 유지한다. |

## 3. SLA 정의

### 3.1 가용성 SLA 기준표

| 기능 영역 | 가용성 SLA | 월간 허용 다운타임 | 측정 기간 |
|----------|:----------:|:-----------------:|---------|
| SOS 기능 | **99.9%** | 월 ≤ 43분 | 월간 Rolling |
| 위치 추적 서비스 | **99.7%** | 월 ≤ 2시간 | 월간 Rolling |
| 일반 기능 (일정/채팅/지도) | **99.5%** | 월 ≤ 3.6시간 | 월간 Rolling |
| 데모 투어 모드 | **99.0%** | 월 ≤ 7.2시간 | 월간 Rolling |

> **SOS 기능 정의**: SOS 발송 API (`POST /api/v1/trips/{tripId}/sos`), FCM 알림 전달, 오프라인 SOS 큐잉까지 전체 체인 포함. (비즈니스 원칙 §05.1)

> **무료 구간 SLA 보장**: 안전 기능(SOS, 가디언 긴급 알림, 기본 위치 공유)은 과금 여부와 무관하게 유료 SLA와 동일하게 보장한다. (비즈니스 원칙 §09.5)

### 3.2 B2B SLA 계약 근거 (비즈니스 원칙 §12.2)

B2B 고객 유형별 차등 SLA를 적용한다.

| 고객 유형 | SOS + 위치 추적 SLA | 일반 기능 SLA | 적용 근거 |
|----------|:------------------:|:-----------:|---------|
| 교육 기관 (학교) | **99.9%** | **99.7%** | 미성년자 안전 책임 강화 |
| 여행사 | **99.7%** | **99.7%** | 전 기능 동일 수준 |
| 일반 B2C | 위 §3.1 기준 적용 | 위 §3.1 기준 적용 | — |

**B2B SLA 모니터링 리포트 제공 정책:**
- 발송 주기: 월 1회 자동 발송 (매월 1일)
- 발송 채널: 이메일 + B2B 관리자 포털 대시보드
- 리포트 내용: 월간 SLA 달성률, 가용성 분 단위 데이터, 장애 이력 요약

**SLA 위반 시 패널티:**
- 산정 기준: 위반 시간에 비례한 크레딧 환급
- 환급 한도: 최대 월 결제액의 30%
- 신청 방법: 위반 월 익월 15일 이내 B2B 관리자 포털에서 신청
- B2B 고객 전용 서비스 상태 페이지: 조직별 전용 URL 제공

## 4. 성능 기준 (KPI)

### 4.1 API 응답 시간

| 지표 | P50 | P95 | P99 | 비고 |
|-----|:---:|:---:|:---:|------|
| 전체 엔드포인트 응답 시간 | ≤ 200ms | ≤ 500ms | ≤ 1,000ms | — |
| SOS API 응답 시간 | ≤ 100ms | ≤ 300ms | ≤ 500ms | Safety 최우선 (비즈니스 원칙 §05.1) |
| 위치 업데이트 API | ≤ 150ms | ≤ 400ms | ≤ 800ms | — |
| 인증 API | ≤ 200ms | ≤ 500ms | ≤ 1,000ms | — |

### 4.2 클라이언트 성능

| 지표 | 기준값 | 비고 |
|-----|:------:|------|
| 앱 콜드 스타트 시간 | ≤ 3초 | 스플래시 화면 포함 |
| 화면 전환 시간 | ≤ 300ms | 모든 라우팅 전환 |
| FCM 알림 수신 지연 | ≤ 5초 | 서버 발송 → 기기 수신 기준 |
| 지도 타일 렌더링 | ≤ 1초 | 캐시 히트 시 |
| 위치 업데이트 지연 (실시간 모드) | ≤ 3초 | 포그라운드, 위치 변화 감지 후 |
| 위치 업데이트 지연 (배터리 절약 모드) | ≤ 30초 | 백그라운드 배터리 ≤ 20% |

### 4.3 SOS 전용 성능 기준

SOS는 모든 기능 중 최고 성능 기준을 적용한다.

```
SOS 발동 → API 수신: ≤ 500ms (P99)
API 처리 → FCM 발송: ≤ 200ms
FCM 발송 → 수신자 기기 수신: ≤ 5초
전체 체인 (발동 → 수신 확인): ≤ 6초 (P99)
```

## 5. 배터리 소모 기준

| 모드 | 소모 기준 | 전환 조건 |
|-----|:--------:|---------|
| 포그라운드 (전면 실행) | ≤ 5%/시간 | 앱 화면 활성 상태 |
| 백그라운드 위치 추적 | ≤ 2%/시간 | 앱 백그라운드 전환 시 자동 |
| 배터리 절약 모드 | ≤ 0.5%/시간 | 기기 배터리 ≤ 20% 시 자동 전환 |
| 완전 절약 모드 | 위치 추적 일시 중단 | 기기 배터리 ≤ 5% 시 (SOS 기능만 유지) |

**배터리 절약 모드 전환 시 동작:**
- 위치 업데이트 주기: 30초 → 60초로 자동 전환
- 지도 실시간 렌더링: 5초 → 15초로 자동 전환
- SOS 기능: 어떤 모드에서도 항상 활성 유지 (비즈니스 원칙 §05.1)
- 완전 절약 모드(≤ 5%)에서도 Heartbeat 60초 주기 유지 (SOS 원칙 §5.1 참조)

### 7.3 SOS 전용 모니터링

SOS 기능은 별도 모니터링 레이어를 운영한다.

```
SOS 모니터링 체크포인트:
1. SOS API 엔드포인트 가용성 (30초 주기 헬스체크)
2. SOS → FCM 전달 체인 latency (P50/P95/P99)
3. SOS 수신 확인율 (수신자 기기에서 확인 응답 비율)
4. 오프라인 SOS 큐 크기 및 처리 지연
5. 미해결 SOS 이벤트 현황 (active 상태 SOS 건수)
```

## 8. 알림 임계값 (Alerting)

| 조건 | 등급 | 알림 채널 | 대응 |
|-----|:----:|---------|------|
| API 에러율 > 1% (5분 지속) | Warning | 슬랙 #ops-warning | 담당자 확인 |
| API 에러율 > 5% (5분 지속) | Critical | 슬랙 #ops-critical + 온콜 전화 | 즉시 온콜 호출 |
| SOS 기능 장애 (헬스체크 실패) | Immediate | 전체 팀 알림 + SMS | 즉시 에스컬레이션 |
| API P99 응답 > 2,000ms | Warning | 슬랙 #ops-warning | 성능 분석 시작 |
| 위치 추적 지연 > 10초 | Warning | 슬랙 #ops-warning | 원인 파악 |
| 앱 크래시율 > 0.1%/DAU | Warning | 슬랙 #ops-warning | Crashlytics 분석 |
| DB 연결 풀 사용률 > 80% | Warning | 슬랙 #ops-warning | 커넥션 점검 |
| DB 연결 풀 사용률 > 95% | Critical | 슬랙 #ops-critical + 온콜 전화 | 즉시 스케일링 |
| FCM 전달률 < 95% | Warning | 슬랙 #ops-warning | FCM 토큰 점검 |

**SOS 임계값 특칙:**
- SOS API 헬스체크는 30초 주기로 수행한다.
- 연속 2회 헬스체크 실패(1분) 시 즉시 Immediate 등급으로 격상한다.
- SOS 알림을 위한 별도 PagerDuty 연동 또는 동급 온콜 도구를 운영한다.

## 9. 장애 등급 분류

| 등급 | 명칭 | 설명 | 예시 사례 | 초기 대응 시간 |
|:----:|------|------|---------|:------------:|
| **P1** | Critical | SOS 기능 장애 / 전체 서비스 다운 | SOS API 500 에러, FCM 알림 전달 불가, 서비스 전체 다운 | **15분 이내** |
| **P2** | High | 위치 추적 장애 / 인증 장애 | GPS 수신 불가, 로그인 불가, RTDB 연결 끊김 | **1시간 이내** |
| **P3** | Medium | 일정/채팅 기능 장애 | 채팅 메시지 지연, 일정 저장 실패, 지도 타일 미로딩 | **4시간 이내** |
| **P4** | Low | UI 결함 / 비핵심 기능 장애 | 화면 레이아웃 깨짐, 통계 데이터 오류, 데모 투어 오작동 | **24시간 이내** |

**SLA 위반 기준:**
- P1 장애 누적 시간이 SOS 기능 SLA(99.9%) 초과 시 → B2B 패널티 크레딧 자동 산정
- P2 장애 누적 시간이 위치 추적 SLA(99.7%) 초과 시 → B2B 패널티 크레딧 자동 산정

## 10. 인시던트 대응 프로세스 (5단계)

### 10.1 프로세스 개요

```
[1단계: 감지(Detect)]
    모니터링 알림 수신 (자동) 또는 유저 신고 (수동)
    ↓
[2단계: 분류(Classify)]
    P1~P4 등급 결정 — 5분 이내 완료
    장애 등급 채널에 "장애 시작" 선언 + 담당자 지정
    ↓
[3단계: 대응(Respond)]
    등급별 대응 시간 내 1차 조치 시작
    임시 완화 조치 (트래픽 차단, 롤백, 스케일링 등)
    ↓
[4단계: 해결(Resolve)]
    서비스 정상화 확인 (헬스체크 통과 + 에러율 정상)
    "장애 종료" 선언 + 영향 시간 기록
    ↓
[5단계: 회고(Postmortem)]
    P1/P2 장애: 장애 종료 후 72시간 이내 회고 문서 작성
    회고 문서 포함 항목: 타임라인, 근본 원인, 영향 범위, 재발 방지 조치
```

### 10.2 등급별 대응 체크리스트

**P1 (Critical) — 15분 이내 대응:**
- [ ] 슬랙 #ops-critical에 장애 선언 (감지 후 5분 이내)
- [ ] 온콜 담당자 호출 및 백업 담당자 대기
- [ ] SOS 기능 격리 가능 여부 확인 (다른 서비스와 분리)
- [ ] 임시 정적 페이지 또는 유지보수 모드 전환 검토
- [ ] 경영진 보고 (감지 후 15분 이내)

**P2 (High) — 1시간 이내 대응:**
- [ ] 슬랙 #ops-critical에 장애 선언
- [ ] 담당 엔지니어 확인 및 배정
- [ ] 영향 범위 파악 (영향 사용자 수, 여행 건수)
- [ ] 임시 완화 조치 적용

**P3/P4 — 일반 티켓으로 관리:**
- 슬랙 #ops-warning에 기록
- 정규 업무 시간 내 처리
- 버그 트래커에 이슈 등록

### 10.3 에스컬레이션 경로

```
1차: 자동 감지 (모니터링 시스템)
    ↓ P1/P2 감지 시
2차: 온콜 엔지니어 (15분 이내)
    ↓ 30분 이내 해결 불가 시
3차: 팀 리드 / 운영 팀장
    ↓ 1시간 이내 해결 불가 시 (P1)
4차: CTO / 경영진
```

비즈니스 원칙 §04의 프라이버시 등급에 따라 위치 업데이트 주기와 성능 동작이 달라진다.

| 프라이버시 등급 | 위치 업데이트 주기 | 가디언 공유 주기 | 성능 우선순위 | 근거 |

> - 스케줄 ON + 이동 중: 30초 간격
> - 스케줄 ON + 정지: 120초 간격
> - 스케줄 OFF: 수집하지 않음

## 13. 에러 및 엣지케이스 처리

| 케이스 | 처리 방법 |
|--------|----------|
| **SOS API 응답 실패** | 최대 3회 자동 재시도 (각 500ms 간격) → 실패 시 오프라인 큐잉 + SMS 폴백 (SOS 원칙 §12) |
| **위치 추적 중단 (GPS 손실)** | 30초 대기 → Wi-Fi/셀 위치로 폴백 → 정확도 저하 표시 |
| **Firebase RTDB 연결 끊김** | 로컬 버퍼링 (최대 10MB) → 재연결 시 일괄 전송 |
| **PostgreSQL 연결 풀 고갈** | 큐 대기 (최대 1,000ms) → 타임아웃 시 503 응답 + Warning 알림 |
| **크래시 루프 감지** | 5회 연속 크래시 시 자동 Crashlytics 리포트 + 마지막 정상 상태로 복귀 시도 |
| **FCM 알림 미전달** | 서버 재시도 (30초 간격 3회) → 이후 SMS 폴백 (SOS 원칙 §12) |
| **배터리 급감 (≤ 5%)** | 위치 추적 일시 중단 + 유저에게 알림 "위치 추적이 일시 중단되었습니다" + SOS 기능 유지 |
| **네트워크 간헐적 끊김** | 지수 백오프 재시도 (1초 → 2초 → 4초 → 최대 30초) |
| **다수 동시 SOS 이벤트** | 각각 독립 처리, 성능 저하 방지를 위한 큐 우선순위 적용 (SOS가 일반 요청보다 항상 우선) |
| **B2B 대규모 트래픽 (단체 여행)** | 멤버 50명 이상 여행: 위치 업데이트 배치 처리 (개별 요청 → 묶음 전송) |

## 14. DB 스키마

DB 설계 v3.4 (`07_T2_DB_설계_및_관계_v3_4.md`) 기준, `tb_performance_log` 및 `tb_incident` 테이블이 v3.4에 포함되지 않는 경우 Firebase Custom Metrics + BigQuery 연동으로 대체 운영한다.

### 14.1 성능 로그 테이블 (신규 제안)

```sql
-- tb_performance_log: API 응답시간, 에러율 기록
-- DB 설계 v3.4에 미포함 → 후속 DB 개정 시 추가 검토
CREATE TABLE IF NOT EXISTS tb_performance_log (
    id              UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    endpoint        VARCHAR(200) NOT NULL,      -- 엔드포인트 경로
    method          VARCHAR(10)  NOT NULL,       -- HTTP 메서드
    response_time   INTEGER      NOT NULL,       -- 응답 시간 (ms)
    status_code     SMALLINT     NOT NULL,       -- HTTP 상태 코드
    error_type      VARCHAR(50),                 -- 에러 유형 (4xx/5xx 시)
    user_id         UUID         REFERENCES tb_user(id),
    trip_id         UUID         REFERENCES tb_trip(id),
    recorded_at     TIMESTAMPTZ  DEFAULT now()
);

CREATE INDEX idx_perf_log_endpoint ON tb_performance_log(endpoint, recorded_at DESC);
CREATE INDEX idx_perf_log_status   ON tb_performance_log(status_code, recorded_at DESC);
-- 7일 이후 데이터는 BigQuery로 아카이브 후 삭제
```

### 14.2 인시던트 테이블 (신규 제안)

```sql
-- tb_incident: 장애 ID, 등급, 감지 시각, 해결 시각, 영향 범위, postmortem URL
-- DB 설계 v3.4에 미포함 → 후속 DB 개정 시 추가 검토
CREATE TABLE IF NOT EXISTS tb_incident (
    id               UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    incident_grade   VARCHAR(5)   NOT NULL CHECK (incident_grade IN ('P1','P2','P3','P4')),
    title            VARCHAR(200) NOT NULL,
    description      TEXT,
    detected_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),
    resolved_at      TIMESTAMPTZ,
    affected_scope   TEXT,        -- 영향 범위 (예: "SOS API, 전체 사용자")
    affected_users   INTEGER,     -- 영향 사용자 수 (추정)
    root_cause       TEXT,        -- 근본 원인
    postmortem_url   VARCHAR(500),-- 회고 문서 URL (Notion 등)
    created_by       UUID         REFERENCES tb_user(id),
    created_at       TIMESTAMPTZ  DEFAULT now(),
    updated_at       TIMESTAMPTZ  DEFAULT now()
);

CREATE INDEX idx_incident_grade      ON tb_incident(incident_grade, detected_at DESC);
CREATE INDEX idx_incident_unresolved ON tb_incident(detected_at) WHERE resolved_at IS NULL;
```

### 14.3 Firebase Custom Metrics 대체 방안

DB 테이블 미구현 시 Firebase Custom Metrics를 통해 다음 지표를 추적한다:

| 지표 | Firebase Event 이름 | 파라미터 |
|-----|-------------------|--------|
| SOS 전송 성공 | `sos_sent_success` | trip_id, latency_ms |
| SOS 전송 실패 | `sos_sent_failure` | trip_id, error_type |
| API 응답 시간 | `api_response_time` | endpoint, response_ms, status_code |
| 앱 시작 시간 | `app_cold_start` | startup_ms, device_model |

## 15. 오프라인 대응

비즈니스 원칙 §11 및 오프라인 동작 통합 원칙(#16) §3.2 연동.

### 15.1 모니터링 데이터 로컬 버퍼링

- **버퍼링 용량**: 최대 10MB
- **버퍼링 기간**: 최대 24시간
- **저장 형식**: SQLite (기기 내 로컬 DB)
- **네트워크 복귀 시**: 오래된 데이터부터 순서대로 일괄 전송

### 15.2 오프라인 중 우선순위

비즈니스 원칙 §11.2 동기화 우선순위 준수:

```
1순위: SOS 이벤트 (최우선 — 네트워크 복귀 즉시)
2순위: 가디언 긴급 알림/요청
3순위: 위치 데이터 업로드
4순위: 일정 변경 동기화
5순위: 채팅 메시지 발송
6순위: 성능 로그, 에러 리포트
```

### 15.3 오프라인 중 SOS 이벤트 처리

| 상황 | 처리 방법 |
|------|----------|
| 완전 오프라인 SOS 발동 | SMS 발송 + 기기 최대 볼륨 알람 + SQLite 큐잉 (SOS 원칙 시나리오 F) |
| 네트워크 복귀 | 큐잉된 SOS Priority 1로 즉시 전송 |
| 큐 전송 실패 | 30초 간격으로 최대 10회 재시도 |

### 15.4 모니터링 시스템 오프라인 내성

- 서버 모니터링 에이전트: 네트워크 단절 시 로컬 메모리 큐 (최대 5분 분량)
- 헬스체크 실패 감지: 30초 응답 없음 = 장애로 간주 → 자동 알림 발송
