## §1. 백오피스 존재 목적

### 1.1 정의

SafeTrip 백오피스는 **SafeTrip 모바일 앱 서비스의 운영 · 관리 · 분석을 위한 웹 기반 관리 플랫폼**이다. 서비스 운영진(내부 관리자), B2B 파트너 관리자, CS 담당자가 플랫폼 전체를 모니터링하고 신속하게 대응할 수 있는 **통합 관제 센터(Command Center)** 역할을 수행한다.

### 1.2 핵심 미션

| 미션 | 설명 |
|------|------|
| **🛡️ 안전 관제** | SOS 긴급 상황의 실시간 모니터링 및 신속 대응 |
| **👥 사용자 관리** | 회원 정보 조회, 활동 추적, 제재(Ban) 관리 |
| **✈️ 여행 운영** | 전체 여행의 생성/진행/종료 현황 파악 및 관리 |
| **💰 수익 관리** | 결제 내역, 구독 현황, 매출 분석 |
| **🏢 B2B 파트너** | 기업/학교/여행사 계약 관리 및 대시보드 제공 |
| **📊 데이터 분석** | 서비스 KPI, 사용자 행동 분석, 트렌드 파악 |
| **🔧 시스템 운영** | 감사 로그, 시스템 상태, 설정 관리 |

### 1.3 기존 백오피스의 한계 (v1.0 기준)

| 한계 | 상세 |
|------|------|
| **제한된 모듈** | Dashboard, SOS, Users, Trips, Finance, B2B, Audit, Settings 8개 모듈만 존재 |
| **통계 부재** | 대시보드에 기본 카운트만 표시, 시계열 차트·트렌드 분석 없음 |
| **실시간 모니터링 부재** | SOS Live Feed만 존재, 실시간 여행/위치 모니터링 없음 |
| **B2B 관리 미약** | 조직 조회 수준, 계약 관리·대시보드 커스터마이징 미구현 |
| **CS 도구 부재** | 고객 문의 추적, 이슈 에스컬레이션 시스템 없음 |
| **콘텐츠 관리 부재** | 안전가이드, 국가 정보, 공지사항 등 콘텐츠 관리 불가 |
| **RBAC 미구현** | 관리자 역할 구분 없이 단일 접근 권한 |
| **UI/UX 한계** | 각 페이지가 독립적 테이블 나열 수준, 워크플로우 연계 없음 |

## §2. 재개편 비전

### 2.1 비전 선언

> **"SafeTrip의 모든 운영 활동을 하나의 화면에서 — 안전 관제부터 비즈니스 분석까지"**

### 2.2 재개편 목표

| 목표 | KPI | 기준 |
|------|-----|------|
| **운영 효율성** | 관리자 1인당 처리 시간 50% 단축 | SOS 대응, 사용자 문의 처리 기준 |
| **실시간 가시성** | 전체 활성 여행/SOS의 실시간 모니터링 | 지연 ≤ 10초 |
| **데이터 기반 의사결정** | 핵심 KPI 대시보드 7종 이상 | DAU, MAU, 매출, 전환율 등 |
| **B2B 자율성** | B2B 파트너가 자체 대시보드에서 여행 관리 | 내부 CS 의존도 0% |
| **확장성** | 신규 기능 모듈 추가 시 3일 이내 | 모듈러 아키텍처 |

### 2.3 설계 원칙

| 원칙 | 설명 |
|------|------|
| **Safety First** | SOS 관제가 항상 최우선, 모든 화면에서 긴급 알림 접근 가능 |
| **Data-Driven** | 모든 의사결정에 데이터 시각화를 근거로 제공 |
| **Module-First** | 각 기능은 독립 모듈로 설계, 플러그인 방식 확장 |
| **Role-Aware** | 접근 권한은 관리자 역할(Super Admin/Ops Admin/CS Admin/B2B Admin/Analyst/Viewer 6종)에 따라 차등 |
| **Mobile-Consistent** | 디자인 언어는 SafeTrip 모바일 앱과 일관된 Ghibli High-Fidelity 스타일 유지 |
| **API-Backbone** | 모든 데이터는 NestJS 백엔드 API를 통해서만 접근 (직접 DB 접근 금지) |

## §4. 기술 스택 (v2.0)

### 4.1 프론트엔드

| 영역 | 기술 | 비고 |
|------|------|------|
| **프레임워크** | Next.js 16+ (App Router) | SSR + CSR 하이브리드 |
| **언어** | TypeScript | 타입 안전성 보장 |
| **스타일링** | CSS Variables + PostCSS | Ghibli 테마 시스템, 다크모드 지원 |
| **아이콘** | Lucide React | 일관된 아이콘 체계 |
| **차트** | Recharts 또는 Chart.js | 데이터 시각화 |
| **테이블** | TanStack Table (React Table v8) | 고급 정렬, 필터, 페이지네이션 |
| **날짜** | date-fns 또는 dayjs | 경량 날짜 라이브러리 |
| **HTTP 클라이언트** | Native Fetch API | apiClient 래퍼 유지 |
| **실시간 통신** | WebSocket / SSE | SOS 실시간 업데이트 |
| **상태 관리** | React Context + SWR or TanStack Query | 서버 상태 캐싱 |

### 4.2 배포/인프라

| 영역 | 기술 | 비고 |
|------|------|------|
| **컨테이너** | Docker (multi-stage build) | standalone 출력 최적화 |
| **포트 매핑** | Host `3006` → Container `3000` | 기존 유지 |
| **인증** | Firebase Auth (Production) / x-test-bypass (Dev) | 기존 메커니즘 유지 |
| **백엔드 API** | NestJS (port 3001) | Admin 엔드포인트 확장 |

## §7. 성공 기준

| 기준 | 측정 방법 |
|------|----------|
| SOS 대응 시간 ≤ 30초 | SOS 발생 ~ 관리자 확인까지 소요 시간 |
| 대시보드 로딩 ≤ 2초 | 첫 의미 있는 데이터 표시까지 |
| 관리자 만족도 ≥ 4.0/5.0 | 내부 설문 |
| B2B 파트너 자립도 ≥ 80% | CS 없이 자체 해결한 작업 비율 |
| 시스템 가용성 ≥ 99.5% | 월간 가동 시간 |
