## 공통 작성 규칙 (1/2 문서와 동일)

### 모든 문서에 반드시 포함해야 하는 구조

```
1. 문서 헤더 (메타데이터 테이블)
   - 문서 ID, 계층, 등급, 버전, 작성일
   - 기준 문서(상위 문서) — 반드시 v5.1 참조 명시
   - 관련 법령 (해당 시)
   - 후행 문서 목록
   - 검토 주기, 담당 부서
2. 문서 목적 및 적용 범위
3. 본문 (문서별 고유 내용 — 아래 개별 지침 참조)
4. 역할별 접근 권한 매트릭스 (캡틴/크루장/크루/가디언)
5. 프라이버시 등급별 동작 차이 (안전 최우선/표준/프라이버시 우선)
6. 에러 및 엣지케이스 처리
7. DB 스키마 (해당 시) — v5.1 부록 C 스키마 반영 필수
8. 오프라인 대응 (해당 시)
9. 구현 우선순위 (P0~P3) 및 Phase 배치
10. 검증 체크리스트 (법적 + 기능 + 거버넌스 12항목)
11. 기존 문서 대비 변경·확장 사항 (변경 이력)
12. 부록 (관련 문서 참조 테이블)
```

#### 작성 프롬프트

```
[역할] SafeTrip의 위치 추적 시스템 설계자 겸 프라이버시 정책 전문가

[목적] SafeTrip 멤버탭 내 개별 멤버의 이동기록 화면(타임라인·지도 이중 뷰,
체류 지점 분석, 프라이버시 마스킹, 인사이트)을 v5.1 기준으로 갱신하라.

[기준 문서] SafeTrip_비즈니스_원칙_v5.1

[갱신 필수 사항]
1. 기준 문서 참조: v3.0 → v5.1 갱신 (문서 전체에서 v3.0 참조를 v5.1로 변경)
2. 이동기록 설계 철학 5대 원칙(M1~M5) 유지 및 v5.1 정합성 확인
   - M1: 투명성 / M2: 이중 뷰(타임라인+지도) / M3: 시간순 / M4: 프라이버시 마스킹 / M5: 인사이트 확장
3. 이동기록 데이터 구조 (위치 포인트, 체류 지점, 경로 세그먼트)
   - 수집 주기, 배터리 최적화, 정확도 기준 유지
4. 타임라인 뷰 구성 원칙
   - 시간 슬라이더, 이벤트 마커(체류/이동/알림), 일정 연동 표시
   - 확대/축소 시 데이터 밀도 조절
5. 지도 뷰 구성 원칙
   - 경로 폴리라인, 체류 지점 마커, 히트맵 모드(Phase 2)
   - 지도 기본화면(#17) 레이어 시스템과의 연동 규칙
6. ★ 가디언 과금별 이동기록 접근 범위 차등 — §09.3
   - 무료 가디언: 당일 이동기록만 조회 가능 (24시간)
   - 유료 가디언: 전체 보존 기간 이동기록 조회 가능
   - 과금 전환 유도 UI (이동기록 상세 조회 시 유료 전환 안내)
7. ★ 이동기록 보존 기간 정책 — §13
   - 일반 멤버: 30일 보존 후 자동 삭제
   - 유료 가디언 연결 멤버: 90일 보존
   - 보존 만료 전 알림 및 수동 백업(내보내기) 옵션
8. 프라이버시 등급별 이동기록 표시 차이 — §04
   - 안전 최우선: 전체 경로 + 체류 지점 + 정확 주소
   - 표준: 경로 + 체류 지점(반경 표시), 정확 주소 비공개
   - 프라이버시 우선: 일정 연동 시간대만 경로 표시, 비연동 시간 마스킹
9. 프라이버시 마스킹 규칙 상세
   - 마스킹 범위(반경 500m 원), 해제 조건(SOS 시), 마스킹 로그 기록
   - 위치 데이터 정책(#09)과 마스킹 규칙 정합 확인
10. 역할별 접근 권한 매트릭스 — 4역할 × 이동기록 기능 전수 정의
    - 캡틴: 전 멤버 이동기록 조회
    - 크루장: 소속 조 멤버 이동기록 조회
    - 크루: 본인 이동기록만 조회
    - 가디언: 연결 멤버 이동기록 조회 (과금별 범위 차등)
11. 그룹 발자취(Group Footprint)와의 차이점 명확화
    - 개별 이동기록 vs. 그룹 통합 발자취 — 데이터 소스, 표시 방식, 접근 권한 차이
12. 체류 지점 분석 알고리즘 규칙 (체류 판정 기준: 반경/시간/클러스터링)
13. 이동기록 인사이트 기능 (Phase 2)
    - 일별 이동 거리, 체류 핫스팟, 가장 먼 멤버 등
14. 오프라인 대응: 오프라인 시 이동기록 로컬 저장 → 네트워크 복귀 시 동기화
15. DB 스키마: tb_location_history, tb_stay_point 참조 — v5.1 부록 C 반영
    - tb_guardian_link.is_paid 필드와 조회 범위 연동
16. 에러 및 엣지케이스 처리
    - GPS 신호 약함/없음, 기기 전원 꺼짐, 시간대 변경(해외), 위치 점프
17. 검증 체크리스트 유지 및 갱신
18. 기존 문서 대비 변경/확장 사항 테이블 갱신

[현재 문서 갭 분석]
- v3.0 참조 → v5.1 미반영 (전 섹션)
- 가디언 과금별 접근 범위 차등 미정의 (무료/유료 가디언 동일 취급 중)
- 보존 기간 30일/90일 이원화 미반영
- DB 스키마 섹션에서 tb_guardian_link.is_paid 연동 미반영
- 프라이버시 등급별 마스킹이 위치 데이터 정책(#09)과 일부 불일치
```

   - 안전가이드: 실시간 위험 정보, 맞춤 안전 팁
4. ★ AI 과금 체계 — §09
   - 무료 AI: Safety AI 전체 (안전은 무료 원칙), Convenience AI 기본 기능
   - 유료 AI (AI 구독): Intelligence AI 전체, Convenience AI 고급 기능
   - 과금 분기점 명확 정의 (기능별 무료/유료 구분 테이블)
   - AI 구독 결제 흐름 (인앱 결제, PG사 연동)
5. ★ 미성년자 AI 제한 — §10
   - 미성년자(크루) AI 챗봇 직접 대화 제한
   - Safety AI는 미성년자에게도 전체 제공 (보호 목적)
   - Convenience AI 중 위치 기반 추천 기능 제한
   - 미성년자 판별 기준 (프로필 생년월일, 캡틴 설정)
6. AI 데이터 처리 원칙
   - LLM 호출 시 개인정보 마스킹 (이름, 전화번호 등)
   - 위치 데이터의 AI 학습 미사용 원칙
   - AI 응답 캐싱 및 로그 보존 기간
   - 국제 데이터 처리(#08) 원칙과의 정합성
7. AI 모델 선택 기준 및 폴백 전략
   - 1차: 클라우드 LLM / 2차: 경량 온디바이스 모델 / 3차: 규칙 기반 대응
   - 오프라인 시 Safety AI 폴백 (규칙 기반 이탈 감지)
8. AI 응답 품질 기준
   - 할루시네이션 방지 (여행 안전 정보 정확도 기준)
   - 응답 시간 기준 (Safety AI ≤ 2초, Convenience AI ≤ 5초)
   - 다국어 지원 (한국어, 영어, 일본어)
9. 프라이버시 등급별 AI 동작 차이 — §04
   - 안전 최우선: 모든 AI 기능 활성화 (Safety + Convenience)
   - 표준: Safety AI 활성화, Convenience AI 일정 연동 시간대만
   - 프라이버시 우선: Safety AI만 활성화, 위치 기반 AI 비활성
10. Phase별 AI 기능 로드맵
    - Phase 1: Safety AI (이탈 감지, 위험 알림), 기본 Convenience (일정 추천)
    - Phase 2: AI 챗봇, 고급 Convenience (장소 추천, 번역)
    - Phase 3: Intelligence AI (패턴 분석, 예측)
11. AI 실패 시 에러 처리 UX
    - AI 응답 지연 시 로딩 상태 표시 + 수동 대안 제공
    - AI 장애 시 기능 저하(degradation) 전략
12. 역할별 AI 접근 권한 매트릭스 — 4역할 × AI 기능 전수 정의
    - 캡틴: 전체 AI 기능 접근
    - 크루장: 소속 조 범위 AI 기능 접근
    - 크루: 본인 관련 AI 기능만 (미성년자 제한 적용)
    - 가디언: Safety AI + 연결 멤버 인사이트 (과금별 차등)
13. DB 스키마: tb_ai_usage_log, tb_ai_subscription 참조 — v5.1 부록 C 반영

15. 기존 문서 대비 변경/확장 사항 테이블 갱신

3. ★ B2B SLA 계약 근거 — §12.2
   - 기업 고객(학교, 여행사)용 SLA 차등 정의
   - B2B SLA 모니터링 리포트 제공 주기
   - SLA 위반 시 패널티/보상 정책
4. 성능 기준 (KPI)
   - API 응답 시간: P50 ≤ 200ms, P95 ≤ 500ms, P99 ≤ 1000ms
   - 위치 업데이트 지연: ≤ 3초 (실시간), ≤ 30초 (배터리 절약)
   - 앱 시작 시간 (콜드 스타트): ≤ 3초
   - 화면 전환 시간: ≤ 300ms
   - FCM 알림 수신: ≤ 5초 (서버 발송 → 기기 수신)
5. 배터리 소모 기준
   - 전면 실행: ≤ 5%/시간
   - 백그라운드 위치 추적: ≤ 2%/시간
   - 배터리 절약 모드 자동 전환 임계값
6. 네트워크 사용량 기준
   - 위치 업데이트: ≤ 1KB/회
   - 지도 타일 캐시 전략
   - 월간 예상 데이터 사용량
7. 모니터링 대시보드
   - 헬스체크 (서버 상태, DB 연결, Firebase 상태)
   - 에러율 (API 에러, 클라이언트 크래시)
   - 위치 정확도 모니터링
   - 실시간 사용자 수 (DAU/MAU)
8. 알림 임계값 (Alerting)
   - 에러율 1% 초과: Warning
   - 에러율 5% 초과: Critical → 온콜 호출
   - SOS 기능 장애: Immediate → 에스컬레이션
9. 장애 등급 분류
   - P1 (Critical): SOS 기능 장애, 전체 서비스 다운
   - P2 (High): 위치 추적 장애, 인증 장애
   - P3 (Medium): 일정/채팅 기능 장애
   - P4 (Low): UI 결함, 비핵심 기능 장애
10. 인시던트 대응 프로세스
    - 감지 → 분류 → 대응 → 해결 → 회고(postmortem) 5단계
    - 각 장애 등급별 대응 시간 기준 (P1: 15분 내 대응, P2: 1시간 내 등)
11. 역할별 모니터링 접근 권한
    - 관리자 전용 대시보드 (B2B 고객 대시보드 별도)
12. 프라이버시 등급별 성능 차이
    - 위치 업데이트 주기 차이 (안전 최우선: 30초, 표준: 60초, 프라이버시 우선: 일정 시간만)
13. 오프라인 대응: 모니터링 데이터 로컬 버퍼링 → 네트워크 복귀 시 일괄 전송

15. 구현 우선순위: P0(SLA·알림), P1(대시보드), P2(B2B 리포트)
16. 검증 체크리스트 (거버넌스 12항목 전수)
