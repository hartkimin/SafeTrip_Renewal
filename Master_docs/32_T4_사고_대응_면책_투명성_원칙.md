## 1. 문서 목적 및 적용 범위

### 1.1 목적

본 문서는 SafeTrip 서비스 운영 중 발생 가능한 사고·장애·법적 이슈에 대한 대응 매뉴얼 및 면책·투명성 정책을 정의한다. 비즈니스 원칙 v5.1 §05.1(SOS 기능), §08(데이터 정합성), §12(B2B 프레임워크), §13(데이터 생명주기)를 실행 수준으로 구체화하고, SOS 원칙(#13) 및 앱 성능·안정성·모니터링 원칙(#31)과 통합하여 법적 리스크 관리 체계를 수립하는 것이 주목적이다.

### 1.2 적용 범위

- **서비스 전체**: SafeTrip Flutter 앱, Node.js/TypeScript 백엔드 API, Firebase RTDB, FCM
- **사고 유형**: SOS 장애(Type A), 위치 오차(Type B), 오발송(Type C), 시스템 장애(Type D), 데이터 유출(Type E)
- **대상 사용자**: B2C 개인 사용자, B2B 기업 고객 (학교, 여행사, 기업, 보험사)
- **역할 전체**: 캡틴, 크루장, 크루, 가디언 및 운영·법무팀

## 3. 사고 유형 분류 및 대응 매뉴얼

### 3.1 사고 유형 분류 (Type A~E)

| 유형 | 설명 | 심각도 | 예시 |
|------|------|--------|------|
| **Type A** | SOS 미발송 / 발송 지연 | Critical | SOS 버튼 눌렀으나 가디언·그룹 멤버 미수신; FCM 채널 장애로 SOS 알림 지연 |
| **Type B** | 위치 오차 / 위치 미수신 | High | GPS 오차 500m 이상; 위치 데이터 서버 미전송; RTDB 위치 채널 단절 |
| **Type C** | SOS 오발송 / 오알림 | Medium | 실수로 SOS 발송; 잘못된 수신자에게 푸시 알림 발송; 테스트 데이터 프로덕션 노출 |
| **Type D** | 시스템 장애 / 서비스 다운 | High/Medium | API 서버 다운; 앱 크래시 루프; PostgreSQL 연결 불가; Firebase RTDB 장애 |
| **Type E** | 데이터 유출 / 프라이버시 침해 | Critical/High | 위치 데이터 무단 접근; 타 유저 위치 정보 노출; `tb_location_history` 불법 조회 |

> **심각도 정의**: Critical — 인명 안전 직결 또는 법적 즉시 처리 의무 / High — 핵심 서비스 불능 / Medium — 서비스 일부 저하

### 3.2 유형별 대응 매뉴얼

**에스컬레이션 경로**

```
1차 (자동 감지·모니터링)
    ↓ 이상 감지
2차 (운영팀 — 즉시 대응)
    ↓ 해결 불가 또는 임계 초과
3차 (경영진 — 의사결정)
    ↓ 법적 이슈 또는 Type A/E
4차 (법무팀 — 법적 대응)
```

| 유형 | 인지 시간 | 1차 대응 | 에스컬레이션 | 사후 처리 |
|------|---------|---------|-----------|---------|
| **Type A** | 5분 내 (자동 감지) | 15분 내 SOS 채널 복구 시도; FCM 재발송 트리거 | 즉시 3차·4차 동시 에스컬레이션; 외부 긴급 연락처 SMS 발송 검토 | Postmortem 필수 (72시간 내 소집) |
| **Type B** | 10분 내 (모니터링 경보) | 30분 내 위치 채널 점검; RTDB 연결 상태 진단 | 2차 → 미해결 시 3차 1시간 내 보고 | Postmortem 권장; GPS 정확도 지표 재검토 |
| **Type C** | 15분 내 (유저 신고·로그 감지) | 1시간 내 오발송 사과 인앱 공지; 잘못된 알림 취소 처리 | 2차 운영팀 자체 대응 원칙; 반복 시 3차 | 재발 방지 조치; 테스트/프로덕션 격리 강화 |
| **Type D** | 자동 모니터링 감지 (P1 5분, P2 10분) | #31 §9 장애 등급 기준 인시던트 프로세스 준수 | #31 §10 인시던트 대응 5단계 절차 적용 | P1/P2: Postmortem 필수; P3/P4: 개선 티켓 생성 |
| **Type E** | 즉시 (보안 모니터링 경보 또는 외부 신고) | 즉시 해당 데이터 접근 경로 차단; 영향 범위 파악 시작 | 즉시 4차 법무 에스컬레이션; 3차 경영진 동시 보고 | 개인정보 침해 신고 (개인정보보호법 §34, 72시간 내); Postmortem 필수 |

> **Type D 인시던트 등급**: 앱 성능·안정성·모니터링 원칙(#31) §9의 P1~P4 장애 등급 분류 기준을 그대로 적용한다. Type D 장애는 #31의 인시던트 대응 프로세스가 기본 프레임이 되며, 본 문서는 법적·면책 관련 추가 조치를 보완한다.

### 3.3 면책 범위 명확화 (비즈니스 원칙 §08)

이용약관 §8과 면책 범위 1:1 대응 테이블:

| 구분 | 이용약관 §8 조항 대응 | 책임/면책 내용 | 예시 |
|------|---------------------|--------------|------|
| **SafeTrip 책임** | §8.1 서비스 장애 책임 | 서버 장애로 인한 SOS 미발송; SafeTrip 인프라 결함으로 인한 위치 추적 서비스 장애; SafeTrip 보안 취약점으로 인한 데이터 유출 | 서버 다운으로 SOS 전송 실패; DB 오류로 위치 미수신 |
| **SafeTrip 면책** | §8.2 이용자·외부 환경 원인 면책 | GPS 기기 고장; 통신사 네트워크 장애; 유저 앱 설정 오류(알림 비활성화); 가디언 앱 미설치 또는 미실행; 사용자 배터리 방전 | 기기 GPS 불량으로 위치 오차; 유저가 알림 차단 설정 후 SOS 미수신 |
| **기술적 한계 면책** | §8.3 기술적 불가피한 한계 면책 | 정상 운영 상태에서의 GPS 오차 (±100m 이내 허용 기준); 극지·심해·지하 등 GPS 신호 불가 환경; 실내 건물 내부 위치 정확도 저하 | 건물 내부 GPS 미작동; 산악 지역 신호 약화로 위치 오차 |
| **제3자 원인 면책** | §8.4 제3자 서비스 장애 면책 | Google Firebase 플랫폼 장애; Apple APNs 서비스 중단; 지도 서비스(Naver Maps, Google Maps) 장애; 외교부 API 다운 | Firebase RTDB 전체 장애로 위치 미수신 |

> **안전 기능 최저 보장**: 비즈니스 원칙 §09.5에 따라 SOS 발송·수신, 가디언 긴급 알림 등 안전 기능은 과금 상태와 무관하게 항상 동일 품질로 제공되어야 한다. 따라서 안전 기능 장애(Type A)에 대해서는 면책 범위를 엄격하게 제한한다.

### 3.4 유저 커뮤니케이션 계획

#### 고지 채널 및 시간 기준

| 채널 | 발송 시점 | 대상 | 비고 |
|------|---------|------|------|
| 인앱 알림 (FCM) | 사고 인지 즉시 (최대 1시간 내) | 영향받은 사용자 전원 | Type A: 5분 내 발송 목표 |
| 이메일 | 24시간 내 | 가입 이메일 보유 사용자 | 상세 내용·보상 안내 포함 |
| 앱 내 공지사항 | 48시간 내 | 전체 사용자 | 공식 공지 게시 |
| B2B 고객 전용 알림 | 12시간 내 | 계약 고객 담당자 | 계약서 SLA 기준 별도 적용 |

#### 고지 내용 필수 항목

1. 사고 발생 사실 (발생 시각, 영향받은 기능)
2. 영향 범위 (영향받은 사용자 수 또는 비율, 기간)
3. SafeTrip의 조치 사항 (완료된 조치 및 예정 조치)
4. 보상 여부 (SLA 위반 시 크레딧, 환불 기준)
5. 재발 방지 약속 (간략 기술)
6. 문의처 (고객센터 연락처)

#### 투명성 보고서

- **발행 주기**: 분기별 1회 (1·4·7·10월 발행)
- **포함 내용**:
  - 사고 통계 (유형별 발생 건수, 평균 대응 시간)
  - SLA 달성률 (비즈니스 원칙 §12.2 B2B SLA 기준 포함)
  - 개선 사항 (완료된 Postmortem 액션 아이템)
  - 데이터 요청 통계 (수사 협조 요청 건수, 제공 건수 — 법적 공개 가능 범위 내)

### 3.5 수사 협조 프로토콜

#### 일반 수사 협조 절차 (영장 기반)

```
1단계 — 영장 접수 및 확인
   - 압수수색 영장 원본 확인 (팩스·전자 영장 불가, 원본 필수)
   - 영장 범위(대상자, 기간, 데이터 유형) 검토

2단계 — 법무팀 검토
   - 24시간 내 법무팀 법적 유효성 검토
   - 필요 시 외부 법률 자문 요청

3단계 — 데이터 추출 및 제공
   - 지정 포맷(CSV/JSON)으로 데이터 추출
   - 암호화 매체 또는 보안 채널로 전달
   - 영장 범위 초과 데이터는 절대 제공 금지

4단계 — 제공 로그 기록
   - `tb_data_request_log`에 전체 이력 기록
   - 제공 데이터 해시값 보관

5단계 — 당사자 사후 통지
   - 수사가 법적으로 종결되거나 비밀 유지 의무 해제 시점에 당사자 통지
   - 통지 불가 사유(수사 진행 중 비밀유지 명령 등) 발생 시 법무팀 판단
```

#### 제공 가능 데이터 (영장 필수)

| 데이터 유형 | 보존 기간 (비즈니스 원칙 §13) | 비고 |
|-----------|--------------------------|------|
| 위치 이력 (`tb_location_history`) | 여행 종료 후 90일 (SOS 기록 3년) | 영장 범위 내 기간 한정 |
| SOS 기록 | 여행 종료 후 3년 | 법적 분쟁 대비 장기 보관 |
| 이벤트 로그 | 여행 종료 후 1년 | - |
| 채팅 메타데이터 | 여행 종료 후 90일 | 내용 포함 시 별도 영장 필요 |

#### 긴급 수사 협조 (인명 관련 영장 전 제공)

- **적용 조건**: 생명·신체에 즉각적인 위험이 있는 긴급 상황으로 수사기관이 서면 요청한 경우
- **법적 근거**:
  - 위치정보법 제29조 (생명·신체 위협 시 위치정보 제공)
  - 통신비밀보호법 제13조 (긴급 통신 사실 확인자료 제공)
- **제공 범위**: 현재 위치 또는 최근 위치 이력 최소 범위에 한정
- **사후 처리**: 72시간 내 영장 제출 요청; 미제출 시 이후 협조 거부 가능

#### 해외 수사기관 요청

- 한국 법원 영장 또는 사법공조 조약(MLAT) 기반 공식 요청서 필수
- 법무팀 검토 후 외교부 협조 또는 법률 자문 병행

### 3.6 사후 회고 (Postmortem) 프로세스

#### 소집 기준

| 소집 필수 | 소집 권장 |
|---------|---------|
| P1 장애 (#31 §9 기준) | P2 장애 (#31 §9 기준) |
| Type A 사고 (SOS 미발송) | Type B 사고 (위치 오차) |
| Type E 사고 (데이터 유출) | Type C 반복 발생 (3회 이상/월) |

- **소집 시점**: 사고 해결 후 72시간 내
- **참여 인원**: 운영팀장, 개발팀장, (법무팀장 — Type A/E 필수), (경영진 대표 — P1/Type A/E)

#### Postmortem 표준 양식

```markdown
## Postmortem: [사고 유형] [발생일시]

### 1. 타임라인
- HH:MM — 사건 발생 / 인지 시각
- HH:MM — 1차 대응 시작
- HH:MM — 에스컬레이션
- HH:MM — 해결 완료
- HH:MM — 유저 고지 발송

### 2. 원인 분석
- **근본 원인(Root Cause)**: [기술적/프로세스/외부 원인]
- **기여 요인(Contributing Factors)**: [복수 기재]

### 3. 영향 범위
- 영향받은 사용자 수: N명
- 영향 기간: HH:MM ~ HH:MM (총 N분)
- 영향받은 기능: [SOS / 위치 공유 / 기타]

### 4. 재발 방지 액션 아이템
| # | 조치 내용 | 담당자 | 기한 | 상태 |
|---|---------|-------|------|------|
| 1 | [조치 내용] | [담당] | YYYY-MM-DD | 예정 |

### 5. 문서 갱신 계획
- 갱신 대상 문서: [문서명]
- 갱신 사유: [사고로 인한 정책 변경 내용]
- 갱신 기한: 회고 후 30일 내
```

#### 결과 반영 규칙

- 회고 후 30일 내 관련 원칙 문서 갱신 필수
- 갱신된 문서는 문서 버전을 올리고 변경 이력에 Postmortem 참조 번호 기재
- 액션 아이템 완료율은 투명성 보고서에 분기별 공개

### 3.7 보험·배상 정책

#### 서비스 배상 기준

| 배상 유형 | 기준 | 최대 배상 한도 |
|---------|------|------------|
| SLA 위반 크레딧 | 월 가용성 미달 시 비례 크레딧 (#31 §3 기준) | 해당 월 결제액 30% |
| Type A 사고 보상 | SOS 장애로 실질 피해 발생 시 (법무 검토 후 개별 협의) | 결제액 100% 환불 |
| 데이터 유출 피해 보상 | 개인정보 유출로 명백한 손해 발생 시 (법무 결정) | 개별 협의 |

#### 보험 가입 의무

| 보험 유형 | 목적 | 갱신 주기 |
|---------|------|---------|
| 사이버 보험 | 데이터 유출, 랜섬웨어, 개인정보 침해 | 연간 |
| 위치 서비스 배상 책임 보험 | SOS 서비스 장애로 인한 배상 책임 | 연간 |

#### B2B 고객 별도 배상 정책

- 비즈니스 원칙 §12 B2B 프레임워크와 연계하여, 계약서에 SLA 및 배상 한도를 별도 명시한다.
- B2B 학교 계약(수학여행): SOS Type A 장애 발생 시 계약 금액 전액 환불 조항 포함 가능
- B2B 기업 출장 계약: 계약별 협의된 배상 한도 및 SLA 적용
- B2B 보험사 제휴: 별도 API SLA 계약 조건 우선 적용

## 5. 프라이버시 등급별 사고 대응 차이

비즈니스 원칙 v5.1 §02.7 프라이버시 등급(안전 최우선 / 표준 / 프라이버시 우선) 기준으로 사고 발생 시 데이터 제공 범위가 달라진다.

| 프라이버시 등급 | DB 값 | 사고 시 데이터 제공 범위 | 고지 대상 | 특이사항 |
|-------------|------|----------------------|---------|---------|
| **안전 최우선** | `safety_first` | 전체 데이터 즉시 제공 (캡틴·가디언 전원) | 전체 멤버 + 가디언 | SOS 사고 시 모든 프라이버시 설정 무시, 위치 전체 공개 (비즈니스 원칙 §05.1) |
| **표준** | `standard` | 사고 관련 데이터만 제공 (캡틴·가디언 — 스케줄 연동) | 관련 멤버 + 연결 가디언 | SOS 사고 시 프라이버시 등급 무시, 위치 강제 공개 |
| **프라이버시 우선** | `privacy_first` | 최소 데이터만 제공 + 당사자 동의 원칙 | 당사자 + 캡틴만 | 생명 위협 시 동의 없이도 위치 제공 가능 (위치정보법 §29); SOS는 등급 무관 항상 발송 |

> **공통 원칙**: 비즈니스 원칙 §04.8에 따라 SOS 발송 시에는 프라이버시 등급과 무관하게 발송 시점의 위치가 강제로 포함된다. 이 원칙은 모든 사고 대응 절차에서 예외 없이 적용된다.

## 6. 에러 및 엣지케이스

| 상황 | 처리 방식 |
|------|---------|
| SOS 미발송 + 오프라인 상태 | Type A 사고로 분류; 오프라인 로그를 로컬에 보존 (최대 7일, 30MB); 네트워크 복귀 시 자동 재전송 시도; 재전송 실패 시 Type A 인시던트 재처리 (비즈니스 원칙 §11) |
| 수사 협조 요청 중 해당 데이터 삭제 요청 | 법무팀 검토 후 보존 명령 우선 적용; 삭제 요청을 일시 보류하고 수사 종료 후 처리; `tb_data_request_log`에 보존 명령 상태 기록 |
| 데이터 유출 범위 불명확 | 최대 범위(모든 가능성)로 가정하고 즉시 72시간 내 개인정보보호위원회 신고; 범위 확정 후 추가 고지 발송 |
| B2B 고객사 사고 시 계약 조건 상충 | B2B 계약서 SLA 및 배상 조항 우선 적용 (비즈니스 원칙 §12); 계약서 미정의 사항은 본 문서 기준 적용 |
| Type A + Type E 동시 발생 (SOS 장애 + 데이터 유출) | 두 유형 동시 처리; Type A(인명 안전) 최우선; Type E 법무 에스컬레이션 병행 |
| Postmortem 중 신규 사고 발생 | 진행 중인 Postmortem 일시 중단; 신규 사고 대응 우선; Postmortem은 신규 사고 해결 후 재개 |
| 유저 오신고 (SOS 오발송 Type C) | 오발송 사용자에게 확인 알림 발송; 실제 위험 여부 확인 절차 진행; 오발송 확정 시 수신자에게 정정 알림 |
| 미성년자 포함 여행 사고 | Type A/B: 학부모 가디언에게 최우선 고지; 프라이버시 등급 `safety_first` 강제 적용; 비즈니스 원칙 §10 미성년자 보호 정책 병행 |

## 7. DB 스키마

본 섹션에서 정의하는 테이블은 DB 설계 v3.4(#07)에 포함되지 않은 신규 테이블이며, v3.4.1 이후 버전에 반영을 권장한다.

### 7.1 `tb_incident_log` — 사고 기록 테이블

```sql
CREATE TABLE tb_incident_log (
    incident_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    incident_type   VARCHAR(10) NOT NULL CHECK (incident_type IN ('A','B','C','D','E')),
    severity        VARCHAR(10) NOT NULL CHECK (severity IN ('critical','high','medium','low')),
    detected_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    resolved_at     TIMESTAMPTZ,
    affected_users  INTEGER NOT NULL DEFAULT 0,       -- 영향받은 사용자 수
    description     TEXT,                             -- 사고 개요
    postmortem_url  VARCHAR(500),                     -- Postmortem 문서 링크 (Notion 등)
    notified_at     TIMESTAMPTZ,                      -- 유저 고지 발송 시각
    created_by      UUID REFERENCES tb_user(user_id), -- 최초 기록자 (운영팀 계정)
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 인덱스
CREATE INDEX idx_incident_log_type ON tb_incident_log (incident_type, detected_at DESC);
CREATE INDEX idx_incident_log_severity ON tb_incident_log (severity, resolved_at);
```

### 7.2 `tb_data_request_log` — 수사 협조 데이터 제공 기록 테이블

```sql
CREATE TABLE tb_data_request_log (
    request_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    requesting_agency   VARCHAR(200) NOT NULL,       -- 요청 기관명 (경찰청, 검찰청 등)
    request_type        VARCHAR(50) NOT NULL CHECK (request_type IN (
                            'warrant',               -- 영장
                            'emergency_no_warrant',  -- 긴급 영장 전 제공
                            'mlat',                  -- 사법공조조약
                            'other'
                        )),
    data_scope          TEXT NOT NULL,               -- 제공 데이터 범위 상세 기술
    legal_basis         VARCHAR(200) NOT NULL,       -- 법적 근거 (위치정보법 §29 등)
    provided_at         TIMESTAMPTZ,                 -- 데이터 실제 제공 시각
    warrant_number      VARCHAR(100),                -- 영장 번호 (긴급 시 NULL)
    legal_reviewer      UUID REFERENCES tb_user(user_id), -- 법무팀 검토자
    data_hash           VARCHAR(64),                 -- 제공 데이터 SHA-256 해시
    subject_notified    BOOLEAN DEFAULT FALSE,       -- 당사자 사후 통지 여부
    notified_at         TIMESTAMPTZ,                 -- 당사자 통지 시각
    preservation_order  BOOLEAN DEFAULT FALSE,       -- 보존 명령 여부
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 인덱스
CREATE INDEX idx_data_request_agency ON tb_data_request_log (requesting_agency, created_at DESC);
CREATE INDEX idx_data_request_type ON tb_data_request_log (request_type, provided_at);
```

## 8. 오프라인 대응

비즈니스 원칙 §11 오프라인 기본 정책과 연계하여 사고 관련 오프라인 처리를 정의한다.

| 상황 | 처리 방식 | 보존 기준 |
|------|---------|---------|
| 오프라인 중 SOS 발송 실패 | 로컬 큐에 저장; 네트워크 복귀 시 자동 재전송 (비즈니스 원칙 §11.1) | 로컬 저장 최대 7일 또는 30MB |
| 오프라인 중 사고 로그 생성 | 기기 로컬에 암호화 저장 | 로컬 저장 최대 7일 또는 30MB |
| 네트워크 복귀 시 | 로그 자동 서버 전송 → 사고 분류 시작 → 필요 시 사후 Postmortem 소집 | 전송 후 로컬 삭제 |
| 오프라인 중 수사 협조 요청 수신 | 요청 로그 보관; 네트워크 복귀 후 72시간 내 법무팀 처리 | 요청 접수 기록 보존 의무 |

> **오프라인 SOS 우선순위**: 비즈니스 원칙 §11.2 오프라인 복귀 동기화 우선순위 1순위가 SOS 이벤트이다. Type A 사고 판단 시 오프라인 큐 SOS 재전송 성공 여부를 반드시 확인해야 한다.
