## §01 문서 목적 및 적용 범위

### 01.1 문서 목적

본 문서는 SafeTrip 앱에서 여행 그룹에 참여하는 유일한 진입 방식인 **초대코드 시스템**의 전체 규칙을 정의한다.

초대코드는 다음 목표를 동시에 달성하기 위해 설계된다:

1. **보안**: 인가된 사용자만 여행 그룹에 합류할 수 있도록 진입점을 단일화
2. **역할 내재**: 코드 자체에 참여 후 배정될 역할이 사전 정의됨
3. **추적 가능성**: 코드 생성-사용-만료 전 과정이 DB에 기록되어 감사 가능
4. **B2B 확장성**: 기업/학교 단위의 일괄 초대 및 가디언 자동 연결 지원

### 01.2 적용 범위

본 문서의 원칙은 다음 모든 상황에 적용된다:

- B2C 일반 사용자가 초대코드를 통해 여행에 합류하는 모든 플로우
- B2B 관리자가 CSV를 통해 초대코드를 일괄 생성하는 플로우
- 딥링크(`safetrip://invite/{code}`) 또는 QR코드를 통한 합류 플로우
- 가디언 전용 초대코드를 통한 가디언 등록 플로우
- 기업 가디언 모델 A (직접 가디언) / 모델 B (시스템 가디언) 플로우

### 01.3 적용 제외 범위

- 가디언과 멤버의 **직접 연결**(초대코드 없이 링크 생성): 가디언 원칙 문서 참조
- 여행 생성 시 캡틴 자동 배정: 비즈니스 원칙 §02.1 참조
- B2B 계약 자체의 관리 및 청구: 비즈니스 원칙 §12 참조

## §02 초대코드 설계 철학

### 02.1 단일 진입점 원칙

SafeTrip에서 여행 그룹 참여는 **초대코드를 통해서만** 가능하다.

- 사용자는 앱 내 검색으로 여행을 직접 찾을 수 없다.
- 여행 그룹의 존재 자체가 비공개이며, 코드 보유자만 합류 가능하다.
- 이는 여행 멤버의 프라이버시를 보호하고, 무단 합류를 원천 차단하기 위함이다.

### 02.2 핵심 설계 원칙 4가지

| 원칙 | 설명 |
|------|------|
| **역할 내재** | 초대코드에 `target_role`이 사전 지정되어 있으며, 코드 사용 즉시 해당 역할이 부여된다 |
| **시간 제한** | 모든 코드는 만료 시간(`expires_at`)이 존재하며, 무기한 유효 코드는 허용되지 않는다 |
| **추적 가능** | 코드 생성자(`created_by`), 사용자, 사용 시각이 전부 기록된다 |
| **사전 검증** | 코드 적용 전 8단계 서버측 검증을 반드시 통과해야 한다 |

### 02.3 model_type 구분

초대코드는 생성 주체에 따라 `model_type`이 구분된다:

| model_type | 설명 | 생성 주체 |
|------------|------|----------|
| `direct` | 일반 사용자(캡틴/크루장)가 앱 내에서 직접 생성하는 코드 | 캡틴, 크루장 |
| `system` | B2B 계약 하에 관리자 포털 또는 API Key를 통해 자동 생성되는 코드 | B2B 시스템, API |

- `system` 타입 코드는 `b2b_batch_id`가 반드시 연결되어 있어야 한다.
- `direct` 타입 코드는 `b2b_batch_id`가 NULL이다.

## §03 코드 생성 규칙

### 03.1 코드 형식

| 항목 | 규칙 |
|------|------|
| **길이** | 7자리 |
| **문자셋** | 대문자 알파벳 + 숫자 (A-Z, 0-9) |
| **혼동 문자 제외** | O (오), 0 (영), I (아이), l (소문자 L) — 총 4가지 제외 |
| **유효 문자 수** | 알파벳 22자 + 숫자 9자 = 31가지 문자 |
| **가능한 조합 수** | 31^7 = 약 270억 가지 |
| **충돌 방지** | 생성 시 DB에서 중복 여부 즉시 확인 (최대 5회 재시도 후 실패 처리) |

> 혼동 문자(O/0/I/l)를 제외하는 이유: 종이 또는 메시지로 코드를 전달했을 때 사용자가 육안으로 오독하는 것을 방지한다.

### 03.2 유효 기간

| 설정 | 기간 |
|------|------|
| **기본값** | 코드 생성 시점으로부터 **72시간** |
| **캡틴 설정 가능 범위** | 최소 1시간 ~ 최대 7일 (168시간) |
| **B2B 배치 코드** | 배치(batch) 단위 만료: 배치의 `expires_at`이 개별 코드에 우선 적용 |
| **만료 후** | 코드 재활성화 불가. 캡틴이 신규 코드를 발급해야 함 |

### 03.3 사용 횟수 옵션

| 옵션 | max_uses 값 | 설명 |
|------|------------|------|
| **단일 사용** | 1 (기본값) | 1명만 사용 가능. 사용 즉시 소진 |
| **다중 사용** | 2~100 | 캡틴이 직접 설정. B2B 배치에서 주로 활용 |
| **무제한** | NULL | B2B 특수 계약 시에만 허용. `direct` 타입 불가 |

### 03.4 역할별 코드 종류

| 코드 종류 | target_role 값 | 합류 후 배정 역할 |
|----------|---------------|-----------------|
| 크루장 초대 코드 | `crew_chief` | 크루장 |
| 크루 초대 코드 | `crew` | 크루 |
| 가디언 초대 코드 | `guardian` | 가디언 |

> 캡틴 역할은 초대코드로 배정되지 않는다. 캡틴은 여행 생성자에게만 자동 부여되며(비즈니스 원칙 §02.1), 리더 위임을 통해서만 이전 가능하다.

## §04 역할별 초대코드 발급 권한 매트릭스

### 04.1 권한 매트릭스

| 권한 항목 | 캡틴 | 크루장 | 크루 | 가디언 |
|----------|------|-------|------|--------|
| 크루장 코드 생성 | O | X | X | X |
| 크루 코드 생성 | O | O | X | X |
| 가디언 코드 생성 | O | X | X | X |
| 코드 삭제(비활성화) | O | 본인 생성 코드만 | X | X |
| 코드 갱신(재생성) | O | 본인 생성 코드만 | X | X |
| 코드 목록 조회 | O | 본인 생성 코드만 | X | X |
| 코드 사용(합류) | O(다른 여행) | O(다른 여행) | O | O |
| 다중 사용 횟수 설정 | O | X | X | X |
| 만료 기간 설정 | O | X | X | X |

### 04.2 역할별 발급 제약 상세

**캡틴**
- 모든 역할(크루장/크루/가디언)에 대한 초대코드 생성 가능
- 다중 사용, 만료 기간 등 고급 설정 가능
- 발급한 코드를 언제든지 비활성화하거나 삭제할 수 있음
- 동시 활성 코드 수 제한: 역할당 최대 10개

**크루장**
- 크루(crew) 역할 초대코드만 생성 가능
- 자신이 생성한 코드만 관리 가능
- 크루장 코드 또는 가디언 코드 생성 불가
- 동시 활성 코드 수 제한: 최대 5개

**크루**
- 초대코드 생성 권한 없음
- 캡틴/크루장으로부터 받은 코드를 사용하여 다른 여행에 합류하는 것만 가능

**가디언**
- 초대코드 생성 권한 없음
- 전용 가디언 초대코드를 받아 연결하는 것만 가능
- 가디언 코드는 멤버 코드와 분리 관리됨

## §05 8단계 코드 검증 프로세스

사용자가 코드를 입력하거나 딥링크로 합류를 시도할 때, 서버는 반드시 다음 8단계를 순서대로 검증해야 한다. 하나라도 실패하면 즉시 거부하고 이후 단계는 진행하지 않는다.

```
[1단계] 코드 존재 여부
         ↓ 존재하지 않으면 → ERR_CODE_NOT_FOUND
[2단계] 활성화 상태 (is_active = TRUE)
         ↓ 비활성화되었으면 → ERR_CODE_INACTIVE
[3단계] 만료 미도래 (NOW() < expires_at)
         ↓ 만료되었으면 → ERR_CODE_EXPIRED
[4단계] 사용 횟수 미초과 (current_uses < max_uses OR max_uses IS NULL)
         ↓ 소진되었으면 → ERR_CODE_EXHAUSTED
[5단계] 여행 유효성 (연결된 trip.status = 'scheduled' OR 'ongoing')
         ↓ 여행 종료/삭제되었으면 → ERR_TRIP_INVALID
[6단계] 중복 참여 여부 (해당 trip에 이미 group_member 레코드 없음)
         ↓ 이미 참여중이면 → ERR_ALREADY_MEMBER
[7단계] 정원 초과 (group.current_member_count < group.max_members)
         ↓ 정원 초과이면 → ERR_TRIP_FULL
[8단계] 역할 배정 가능 여부 (크루장 정원, 가디언 한도 등 확인)
         ↓ 역할 배정 불가이면 → ERR_ROLE_UNAVAILABLE
         ↓ 모든 검증 통과 → 합류 처리
```

### 05.1 검증 처리 원칙

- 검증은 **서버 측에서만** 수행한다. 클라이언트 측 검증은 UX 보조용이며 보안 목적이 아니다.
- 검증 실패 시 **실패 사유를 명확하게** 반환하되, 보안상 민감한 정보(예: 다른 멤버 목록)는 노출하지 않는다.
- 8단계 검증이 모두 통과된 경우에만 DB 트랜잭션으로 합류 처리를 진행한다.
- 트랜잭션 실패 시 자동 롤백하며, 클라이언트에 ERR_JOIN_FAILED를 반환한다.

## §06 초대 방법 3가지

### 06.1 코드 직접 입력

- 사용자가 앱 내 '코드 입력' 화면에서 7자리 코드를 직접 타이핑
- 대소문자 구분 없이 입력 허용 (서버에서 대문자로 정규화)
- 입력 시 혼동 방지를 위해 폰트가 구분이 명확한 모노스페이스 적용 권장
- 잘못된 문자(O, 0, I, l) 입력 시 자동 치환 또는 경고 표시

### 06.2 딥링크 처리

딥링크 URL 구조:
```
safetrip://invite/{code}
```

딥링크 처리 플로우:
1. **앱 설치된 경우**: 딥링크 직접 처리 → 스플래시 화면(#28)에서 파싱 → 웰컴 화면(#29)에서 자동 입력
2. **앱 미설치 경우**: App Store/Play Store 이동 → 설치 후 딥링크 재처리 (Deferred Deep Link)
3. **스플래시(#28) 처리 규칙**: 앱 시작 시 `intent.data` 또는 `getInitialUri()`로 딥링크 파싱. 코드가 있으면 웰컴 화면으로 코드를 전달
4. **웰컴(#29) 처리 규칙**: 전달받은 코드를 코드 입력 필드에 자동 입력. 사용자는 '참여하기' 버튼만 누르면 됨

웹 폴백 URL (유니버설 링크):
```
https://safetrip.app/invite/{code}
```

### 06.3 QR코드

- 캡틴/크루장이 초대코드 공유 화면에서 QR코드 이미지 생성 가능
- QR코드는 딥링크 URL을 인코딩한 형태
- QR코드 내용: `safetrip://invite/{code}`
- 인쇄 배포용 QR코드의 경우, QR코드 아래에 텍스트 코드를 함께 표시 권장

## §07 프라이버시 등급별 동작 차이

### 07.1 초대코드와 프라이버시 등급의 관계

**초대코드 시스템 자체는 프라이버시 등급에 무관하게 동일하게 동작한다.**

초대코드는 여행 합류를 위한 인증 수단이며, 합류 이후의 위치 공유 방식이 프라이버시 등급에 따라 달라진다. 초대코드 검증 로직, 코드 생성 규칙, 역할 배정은 안전최우선/표준/프라이버시우선 등급 모두 동일하게 적용된다.

### 07.2 등급별 선언

| 프라이버시 등급 | 초대코드 동작 |
|--------------|-------------|
| 안전최우선 (safety_first) | 초대코드 동작 동일. 합류 후 위치 공유가 강제 ON됨 |
| 표준 (standard) | 초대코드 동작 동일. 합류 후 위치 공유는 기본 설정 따름 |
| 프라이버시우선 (privacy_first) | 초대코드 동작 동일. 합류 후 위치 공유는 본인 제어 |

> 단, 가디언 초대코드 사용 시: 가디언은 프라이버시 등급 `safety_first` 여행에서는 자동으로 위치 열람 권한이 부여된다. 이는 초대코드 원칙이 아닌 가디언 원칙에서 별도 정의한다.

## §08 코드 수명주기 관리

### 08.1 상태 전이도

```
[생성됨]
  is_active = TRUE
  current_uses = 0
       |
       |--- 사용 시 current_uses++
       |
       +--→ [활성 / 일부 사용됨]
       |         is_active = TRUE
       |         current_uses < max_uses
       |
       +--→ [소진됨] ← current_uses >= max_uses
       |         is_active = TRUE
       |         (자동 상태, 사용 불가)
       |
       +--→ [만료됨] ← NOW() >= expires_at
       |         is_active = TRUE
       |         (자동 상태, 사용 불가)
       |
       +--→ [비활성화됨] ← 캡틴/크루장이 수동 비활성화
                 is_active = FALSE
                 (즉시 사용 불가)
```

### 08.2 상태 관리 원칙

- **소진/만료 상태**: is_active 값은 변경하지 않는다. 검증 로직에서 uses 및 expires_at으로 판단
- **비활성화**: 캡틴이 코드를 의도적으로 무효화할 경우 is_active = FALSE로 업데이트
- **재활성화**: 비활성화된 코드의 재활성화는 허용하지 않는다. 새 코드를 생성해야 함
- **만료 코드 보관**: 만료/소진 코드는 즉시 삭제하지 않고 여행 종료 후 90일간 보관 후 삭제

## §09 B2B 초대코드 — §12.3 기반

### 09.1 B2B CSV 일괄 코드 생성

비즈니스 원칙 §12.3에 따라 B2B 계약 고객은 관리자 포털에서 CSV 파일을 업로드하여 초대코드를 일괄 생성할 수 있다.

**CSV 파일 형식:**

```csv
이름,이메일,전화번호,역할,그룹명
홍길동,hong@corp.com,01012345678,crew,서울팀
김철수,kim@corp.com,01087654321,crew_chief,서울팀
박영희,park@corp.com,01011112222,guardian,서울팀
```

**CSV 컬럼 정의:**

| 컬럼 | 필수 여부 | 설명 |
|------|---------|------|
| 이름 | 필수 | 참여자 이름 |
| 이메일 | 선택 | 코드 발송 이메일 주소 |
| 전화번호 | 필수 | 코드 발송 SMS 번호 (01X-XXXX-XXXX 형식) |
| 역할 | 필수 | crew / crew_chief / guardian |
| 그룹명 | 선택 | 멀티 그룹 구성 시 그룹 구분 |

**일괄 생성 프로세스:**

```
관리자 포털: CSV 업로드
       ↓
서버: CSV 파싱 및 형식 검증
       ↓ 오류 있을 시 → ERR_CSV_FORMAT_INVALID (오류 행 목록 반환)
       ↓
서버: tb_b2b_invite_batch 레코드 생성 (batch_id 발급)
       ↓
서버: 각 행에 대해 7자리 코드 개별 생성 → tb_invite_code INSERT
       ↓ (model_type = 'system', b2b_batch_id = 발급된 batch_id)
       ↓
서버: 이메일/SMS 자동 발송 (각 코드를 해당 수신자에게)
       ↓
관리자: 코드 상태 대시보드에서 실시간 추적
```

**코드 상태 대시보드 추적 항목:**

| 상태 | 설명 |
|------|------|
| 발송됨 | 이메일/SMS 발송 완료, 미사용 상태 |
| 수신 확인 | 딥링크 클릭 또는 앱 열기 감지 |
| 사용됨 | 코드로 여행 합류 완료 |
| 만료됨 | expires_at 경과, 미사용 |

### 09.2 B2B 배치 만료 정책

- 배치 단위로 `expires_at`을 설정할 수 있다.
- 배치의 `expires_at`이 개별 코드의 `expires_at`보다 앞선 경우, 배치 만료를 우선 적용한다.
- 배치 만료 시 해당 배치의 모든 미사용 코드가 일괄 만료 처리된다.
- 관리자는 배치 만료 24시간 전에 알림을 받는다.

### 09.3 B2B 역할 자동 매핑 — §01.4 기반

B2B 코드 생성 시, 계약된 B2B 고객 유형에 따라 역할 표시명이 자동 변환된다.

| B2B 고객 유형 | crew_chief 표시명 | crew 표시명 | guardian 표시명 |
|-------------|-----------------|------------|----------------|
| 학교 (수학여행) | 인솔교사 | 학생 | 학부모/학년부장 |
| 여행사 (패키지) | 보조가이드 | 참가자 | 참가자 가족 |
| 기업 (출장) | 부팀장 | 팀원 | 팀원 가족/안전담당자 |
| 보험사 (제휴) | — | 피보험자 | 보험사 모니터링 |

- 표시명 변환은 앱 UI 레이어에서 처리한다. DB에는 항상 `crew`, `crew_chief`, `guardian` 등 내부 역할 코드를 저장한다.
- 초대코드 생성 시 `b2b_contract_id`를 참조하여 올바른 표시명 매핑을 적용한다.

## §10 기업 가디언 모델 — §12.4 기반

### 10.1 모델 A: 직접 가디언 (Direct Guardian)

- **정의**: 기업 안전담당자 또는 HR 담당자가 SafeTrip 앱을 직접 사용하여 가디언 역할을 수행하는 모델
- **초대코드 처리**: `target_role = 'guardian'`, `model_type = 'direct'`
- **가디언 초대 방식**: 캡틴(기업 담당자)이 가디언 초대코드 생성 → 가디언에게 전달 → 가디언이 코드 입력하여 연결
- **B2B CSV 지원**: 관리자 포털에서 팀원-가디언 매핑을 CSV로 일괄 업로드하여 guardian 코드 대량 생성
- **가디언 한도**: B2B 계약에 포함된 가디언 수는 별도 협의 (B2C 무료 2명 / 유료 3명 한도와 별개 적용 가능)

### 10.2 모델 B: 시스템 가디언 (System Guardian)

- **정의**: 기업 IT 시스템이 SafeTrip API를 통해 자동으로 모니터링하는 모델. 사람 가디언이 아닌 서버-투-서버 연동
- **초대코드 처리**: `target_role = 'guardian'`, `model_type = 'system'`
- **인증 방식**: API Key 기반 (초대코드 없이 직접 API 호출로 가디언 연결)
- **특이사항**: 앱 설치 불필요. 기업 내부 시스템(HR, 안전관리 SW)과 SafeTrip API 직접 연동
- **활용 예시**: 보험사 모니터링 시스템, 기업 ERP 연동 안전관리

### 10.3 두 모델 혼용 정책

- 동일 여행에서 일부 멤버는 모델 A, 일부는 모델 B 가디언을 가질 수 있다.
- 개인 가디언 상한(무료 2명 + 유료 3명)은 모델과 무관하게 동일하게 적용된다.
- 단, B2B 계약 조건에 의해 별도 한도를 협의할 수 있다.

## §11 참여 후 처리

코드 검증 8단계 통과 후 합류 처리 시 다음 작업이 트랜잭션 내에서 수행된다:

### 11.1 DB 처리

1. `tb_group_member` 레코드 INSERT (user_id, group_id, trip_id, member_role, is_admin 등)
2. `tb_invite_code.current_uses` +1
3. `max_uses`에 도달하면 해당 코드를 소진 상태로 표시 (추가 사용 방지)
4. B2B 코드인 경우: `tb_b2b_member_log` 레코드 INSERT (이력 추적)

### 11.2 클라이언트 처리

1. 멤버탭(#19) 화면 갱신: 새 멤버 목록 표시
2. 합류 시스템 메시지 발송: 채팅방에 "[이름]님이 여행에 참여했습니다" 메시지 자동 발송
3. 캡틴/크루장 푸시 알림: "새로운 멤버가 합류했습니다"
4. 가디언 합류의 경우: 해당 멤버에게 "가디언이 연결되었습니다" 알림

## §12 에러 및 엣지케이스 처리

### 12.1 일반 에러 12가지

| # | 에러 코드 | 상황 | 사용자 메시지 |
|---|---------|------|-------------|
| 1 | ERR_CODE_NOT_FOUND | 입력한 코드가 DB에 없음 | "유효하지 않은 초대코드입니다. 코드를 다시 확인해 주세요." |
| 2 | ERR_CODE_INACTIVE | is_active = FALSE (수동 비활성화됨) | "이 초대코드는 더 이상 유효하지 않습니다." |
| 3 | ERR_CODE_EXPIRED | NOW() >= expires_at | "초대코드 유효기간이 만료되었습니다. 새 코드를 요청해 주세요." |
| 4 | ERR_CODE_EXHAUSTED | current_uses >= max_uses | "초대코드 사용 횟수가 모두 소진되었습니다. 새 코드를 요청해 주세요." |
| 5 | ERR_TRIP_INVALID | 여행 상태가 scheduled/ongoing이 아님 | "해당 여행은 현재 참여가 불가합니다." |
| 6 | ERR_ALREADY_MEMBER | 해당 여행의 group_member가 이미 존재 | "이미 참여 중인 여행입니다." |
| 7 | ERR_TRIP_FULL | current_member_count >= max_members | "여행 정원이 가득 찼습니다." |
| 8 | ERR_ROLE_UNAVAILABLE | 역할 배정 불가 (크루장 정원 초과 등) | "해당 역할로는 현재 참여할 수 없습니다." |
| 9 | ERR_USER_NOT_REGISTERED | 비회원이 코드를 입력 | "SafeTrip 회원가입 후 이용해 주세요." |
| 10 | ERR_REJOIN_COOLDOWN | 탈퇴 후 재참여 제한 시간 미경과 | "해당 여행에서 탈퇴한 후 일정 시간이 지나야 재참여 가능합니다." |
| 11 | ERR_GUARDIAN_MEMBER_OVERLAP | 동일 여행 내 멤버이면서 가디언 시도 | "여행 멤버는 동일 여행의 가디언이 될 수 없습니다." |
| 12 | ERR_JOIN_FAILED | DB 트랜잭션 실패 (일시적 오류) | "참여 처리 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요." |

### 12.2 B2B 추가 에러 3가지

| # | 에러 코드 | 상황 | 처리 방법 |
|---|---------|------|---------|
| 13 | ERR_CSV_FORMAT_INVALID | CSV 파일 형식 오류 (필수 컬럼 누락, 잘못된 역할 값 등) | 오류 행 번호와 사유를 목록으로 반환. 관리자가 수정 후 재업로드 |
| 14 | ERR_B2B_CONTRACT_EXPIRED | B2B 계약 만료 또는 max_trips 초과 | 관리자에게 계약 갱신 안내. 코드 생성 거부 |
| 15 | ERR_GUARDIAN_QUOTA_EXCEEDED | 기업 가디언 코드 생성 한도 초과 | B2B 계약에 명시된 가디언 수 초과. 계약 업그레이드 필요 |

### 12.3 보안 관련 에러 처리

| 상황 | 처리 방법 |
|------|---------|
| **브루트포스 시도** | 동일 IP에서 1분 내 10회 이상 실패 → 5분 차단 (Rate Limiting) |
| **코드 노출 최소화** | 코드 목록 API는 캡틴/해당 크루장만 조회 가능. 코드 자체는 생성 즉시만 전체 표시 |
| **전송 암호화** | 모든 코드 전송은 HTTPS/TLS 1.2 이상 필수 |
| **로그 기록** | 코드 사용 시도(성공/실패 모두) tb_event_log에 기록 |

## §13 DB 스키마

### 13.1 TB_INVITE_CODE (역할별 초대코드)

> 기준: DB 설계 v3.4 §4.6 + 비즈니스 원칙 v5.1 B2B 반영

```sql
CREATE TABLE tb_invite_code (
    invite_code_id  UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    group_id        UUID         REFERENCES tb_group(group_id),
    trip_id         UUID         REFERENCES tb_trip(trip_id),
    code            VARCHAR(7)   UNIQUE NOT NULL,
    target_role     VARCHAR(30)  NOT NULL,           -- crew_chief | crew | guardian
    max_uses        INTEGER      DEFAULT 1,           -- NULL = 무제한 (B2B 특수 계약만)
    current_uses    INTEGER      DEFAULT 0,
    expires_at      TIMESTAMPTZ  NOT NULL,            -- 필수. 기본값 = 생성 시점 + 72h
    created_by      VARCHAR(128) REFERENCES tb_user(user_id),
    created_at      TIMESTAMPTZ  DEFAULT NOW(),
    is_active       BOOLEAN      DEFAULT TRUE,
    -- ★ v5.1 신규 컬럼: B2B 가디언 모델 구분
    model_type      VARCHAR(20)  DEFAULT 'direct',    -- direct | system
    -- ▼ B2B 일괄 초대코드 배치 참조
    b2b_batch_id    UUID         REFERENCES tb_b2b_invite_batch(batch_id)
);

-- 인덱스
CREATE INDEX idx_invite_code_group ON tb_invite_code(group_id);
CREATE INDEX idx_invite_code_code  ON tb_invite_code(code) WHERE is_active = TRUE;
CREATE INDEX idx_invite_code_batch ON tb_invite_code(b2b_batch_id) WHERE b2b_batch_id IS NOT NULL;
```

**컬럼 상세 설명:**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| invite_code_id | UUID | PK, 자동 생성 |
| group_id | UUID | 연결된 그룹. 여행 생성 후 코드 발급 |
| trip_id | UUID | 연결된 여행 |
| code | VARCHAR(7) | 7자리 영숫자 초대코드 (혼동 문자 제외) |
| target_role | VARCHAR(30) | crew_chief / crew / guardian |
| max_uses | INTEGER | 최대 사용 가능 횟수. NULL = 무제한 |
| current_uses | INTEGER | 누적 사용 횟수 |
| expires_at | TIMESTAMPTZ | 만료 시각 (필수) |
| created_by | VARCHAR(128) | 코드 생성자 user_id |
| created_at | TIMESTAMPTZ | 코드 생성 시각 |
| is_active | BOOLEAN | TRUE = 활성, FALSE = 수동 비활성화 |
| model_type | VARCHAR(20) | **★ v5.1 신규**: direct / system |
| b2b_batch_id | UUID | B2B 배치 참조. B2C이면 NULL |

### 13.2 TB_B2B_INVITE_BATCH (B2B 일괄 초대)

> 기준: DB 설계 v3.4 §4.45

```sql
CREATE TABLE tb_b2b_invite_batch (

    batch_name    VARCHAR(200),                    -- 예: "2026 수학여행 3학년 2반"

    sent_count    INTEGER      DEFAULT 0,

    expires_at    TIMESTAMPTZ,                     -- 배치 단위 만료 (개별 코드 우선 적용)

    updated_at    TIMESTAMPTZ
);

CREATE INDEX idx_b2b_invite_batch_contract ON tb_b2b_invite_batch(contract_id);
```

## §15 오프라인 대응

### 15.1 오프라인 시 초대코드 사용 불가 정책

**초대코드 사용(합류 처리)은 반드시 서버 검증이 필요하므로, 오프라인 상태에서는 코드 사용이 불가하다.**

이는 다음 이유에 기반한다:
- 8단계 검증 중 DB 조회(코드 존재, 활성 여부, 사용 횟수, 중복 참여 등)가 필수
- 오프라인 캐시만으로는 검증 신뢰성 보장 불가
- 중복 합류 등 데이터 무결성 문제 발생 가능

### 15.2 오프라인 UX 처리

| 상황 | 처리 방법 |
|------|---------|
| 코드 입력 화면 접근 | 접근 허용. 코드 입력은 가능 |
| '참여하기' 버튼 클릭 (오프라인) | "인터넷 연결이 필요합니다. 연결 후 다시 시도해 주세요." 안내 표시 |
| 딥링크로 앱 진입 (오프라인) | 코드를 로컬에 임시 저장. 인터넷 복구 후 자동 재시도 |
| 인터넷 복구 | 임시 저장된 코드로 자동 검증 재시도 (사용자 재입력 불필요) |

### 15.3 자동 재시도 정책

- 오프라인 중 딥링크로 접근한 코드는 앱 내 메모리에 임시 보관
- 인터넷 연결 감지 시 자동으로 /api/v1/invite-codes/validate 재호출
- 최대 재시도 횟수: 3회 (이후 사용자에게 수동 재시도 안내)
- 임시 보관 최대 시간: 1시간 (이후 자동 삭제)

## §17 검증 체크리스트

다음 12항목은 기능 구현 후 반드시 검증해야 하는 항목이다.

| # | 검증 항목 | 검증 방법 | 합격 기준 |
|---|---------|---------|---------|
| 1 | 7자리 코드 형식 준수 | 코드 생성 1000회 샘플링 | O/0/I/l 포함된 코드 0건, 길이 7자리 100% |
| 2 | 코드 중복 생성 방지 | 동시 코드 생성 요청 100건 | DB UNIQUE 제약으로 중복 코드 0건 |
| 3 | 8단계 검증 각 단계 정상 작동 | 각 단계별 실패 케이스 단위 테스트 | 각 에러 코드가 정확히 반환됨 |
| 4 | 트랜잭션 롤백 처리 | 합류 중 강제 오류 주입 | current_uses 증가 없이 롤백 확인 |
| 5 | 만료 기간 정확도 | 경계값 테스트 (만료 1초 전/후) | 정확한 시각 기준으로 통과/거부 |
| 6 | 역할 배정 권한 제어 | 크루장이 크루장 코드 생성 시도 | ERR_ROLE_UNAVAILABLE 반환 |
| 7 | 딥링크 파싱 정확도 | 유효/만료/없는 코드 딥링크 각각 테스트 | 각각 정상/에러 화면으로 정확히 이동 |
| 8 | 오프라인 대응 | 비행기 모드 상태에서 코드 사용 시도 | 오프라인 안내 메시지 표시, 복구 후 재시도 |
| 9 | B2B CSV 형식 검증 | 형식 오류 CSV 업로드 | 오류 행 목록과 함께 ERR_CSV_FORMAT_INVALID 반환 |
| 10 | model_type 저장 검증 | direct 코드 생성, system 코드 생성 각각 | DB에 올바른 model_type 저장 확인 |
| 11 | 브루트포스 방지 | 동일 IP에서 10회 이상 빠른 실패 요청 | 11번째 요청부터 5분 차단 확인 |
| 12 | 합류 후 처리 완결성 | 정상 합류 후 멤버탭, 채팅, 알림 확인 | 모든 후처리 항목 3초 이내 완료 |
