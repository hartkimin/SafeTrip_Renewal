## 1. 문서 목적 및 적용 범위

### 1.1 목적

본 문서는 SafeTrip 앱의 프로필화면(편집 모드·조회 모드·온보딩 모드) 전체 설계 원칙을 정의한다. 프로필화면은 사용자 신원 정보를 관리하고, 역할과 프라이버시 등급에 따라 타인에게 적절히 공개하는 핵심 개인정보 처리 접점이다. 비즈니스 원칙 v5.1의 가디언 유/무료 구분(§03.1), 위치 공유 프라이버시 등급(§04), 미성년자 보호(§10), 계정 삭제 및 데이터 처리(§14)를 UI 레벨에서 구현하는 규칙을 명세한다.

### 1.2 적용 범위

| 범위 | 포함 여부 |
|------|:--------:|
| 본인 프로필 편집 (닉네임, 프로필 사진, 긴급 연락처) | ✅ |

## 2. 프로필 설계 철학 4원칙

### 2.1 최소 정보 원칙 (Minimum Data Principle)

- 서비스 운영에 **필수적인 정보만** 수집하고 저장한다.
- 온보딩 필수 입력 항목: 닉네임(필수), 프로필 사진(선택), 긴급 연락처(미성년자 필수 / 성인 권장).
- 전화번호·이메일 등 민감 식별 정보는 인증 완료 후 마스킹 처리하여 표시한다.
- 불필요한 SNS 연동 정보, 세밀한 위치 이력은 프로필에 저장하지 않는다.

### 2.2 본인 편집 전용 원칙 (Self-Edit Only)

- 프로필 정보(닉네임, 사진, 긴급 연락처)는 **본인만 수정 가능**하다.
- 캡틴을 포함한 어떤 역할도 타인의 프로필을 직접 수정할 수 없다.
- 단, 캡틴은 미성년자 크루의 긴급 연락처 입력을 온보딩에서 안내할 수 있으나, 직접 입력하지 않는다.

### 2.3 역할 기반 표시 원칙 (Role-Based Display)

- 프로필 조회 시 **조회자의 역할**에 따라 표시되는 정보가 달라진다.
- 같은 프로필이라도 캡틴에게 보이는 정보와 일반 크루에게 보이는 정보는 상이하다.
- 가디언은 연결된 멤버의 프로필만 접근할 수 있으며, 비연결 멤버는 기본 정보(닉네임+사진)만 조회된다.

### 2.4 여행 맥락 확장 원칙 (Travel Context Extension)

- 여행 참여 중에는 기본 프로필에 **여행 맥락 정보**(현재 상태, 마지막 위치, 배정된 조)가 추가 표시된다.
- 여행 외 상황(앱 일반 모드)에서는 기본 프로필 정보만 표시된다.
- 여행 맥락 정보는 해당 여행 기간 동안만 유효하며, 여행 종료 후 자동으로 숨겨진다.

## 3. 프로필 화면 3가지 유형

### 3.1 편집 모드 (Edit Mode) — 본인 프로필 수정

본인이 자신의 프로필을 조회·수정하는 화면이다.

#### 편집 가능 항목

| 항목 | 필수 여부 | 제약 조건 |
|------|:--------:|----------|
| 닉네임 | ✅ 필수 | 2~20자, 특수문자 제한, 중복 검사 |
| 프로필 사진 | 선택 | 최대 5MB, JPG/PNG/GIF, AI 자동 검수 |
| 긴급 연락처 이름 | 미성년자 필수 / 성인 권장 | 1~30자 |
| 긴급 연락처 전화번호 | 미성년자 필수 / 성인 권장 | 국제 전화번호 형식 |

#### 편집 화면 구성

```
┌─────────────────────────────────┐
│  [프로필 사진]  [가디언 배지]        │
│  닉네임                           │
│  전화번호 (마스킹: ***-****-1234)   │
│                                 │
│  ─── 긴급 연락처 ───              │
│  이름:  [                    ]   │
│  전화:  [                    ]   │
│                                 │
│  ─── 계정 관리 ───               │
│  [계정 삭제]                      │
└─────────────────────────────────┘
```

#### 화면 진입 경로

- 설정 메뉴(#15) → "프로필 편집" 탭
- 앱 최초 실행 → 온보딩 완료 후 자동 진입 (온보딩 모드)

### 3.2 조회 모드 (View Mode) — 타인 프로필 조회

타인의 프로필을 조회하는 화면이다. 조회자의 역할과 프라이버시 등급에 따라 표시 정보가 달라진다 (§5, §6 참조).

#### 조회 화면 구성 (공통 레이아웃)

```
┌─────────────────────────────────┐
│  [프로필 사진]  [역할 배지]  [배지] │
│  닉네임                           │
│  여행 상태 (여행 중 / 미참여)        │
│                                 │
│  ─── 여행 맥락 (여행 중에만) ───    │
│  현재 상태: [이동 중 / 정지 / SOS]  │
│  마지막 위치: (권한에 따라 표시)      │
│  배정 조: (크루장·캡틴만 표시)       │
│                                 │
│  ─── 긴급 연락처 ───              │
│  (캡틴만 표시, 타 역할 비공개)       │
└─────────────────────────────────┘
```

#### 화면 진입 경로

- 멤버탭(#19) → 멤버 카드 탭 → 프로필 조회 하프시트 또는 전체 화면
- 지도탭(#17) → 멤버 마커 탭 → 프로필 요약 카드

### 3.3 온보딩 모드 (Onboarding Mode) — 최초 프로필 설정

회원 가입 완료 직후 최초 1회만 표시되는 프로필 설정 화면이다.

#### 온보딩 단계별 흐름

| 단계 | 화면 | 입력 항목 | 건너뛰기 허용 |
|:----:|------|----------|:----------:|
| 1 | 닉네임 설정 | 닉네임 (필수) | ❌ |
| 2 | 프로필 사진 | 사진 업로드 / 아바타 선택 | ✅ (기본 아바타 적용) |
| 3 | 긴급 연락처 | 이름, 전화번호 | 미성년자 ❌ / 성인 ✅ |
| 4 | 프라이버시 등급 선택 | safety_first / standard / privacy_first | ❌ |

#### 온보딩 필수 원칙

- 필수 단계(닉네임, 프라이버시 등급)는 건너뛸 수 없다.
- 미성년자 계정: 온보딩 3단계(긴급 연락처)가 필수이며, 건너뛰기 버튼이 표시되지 않는다.
- 온보딩 완료 전에는 여행 생성·참여를 차단한다.

## 4. 가디언 유/무료 배지 시스템

### 4.1 배지 정의 (§03.1 기반)

비즈니스 원칙 §03.1에 따라 멤버당 가디언은 무료 2명, 유료 최대 3명(총 5명)으로 구분된다. 프로필화면에서 가디언 본인의 과금 상태를 배지로 시각화한다.

| 과금 상태 | 배지 텍스트 | 배지 색상 | 설명 |
|----------|-----------|----------|------|
| 무료 가디언 | `가디언` | 기본 브랜드 컬러 (회색 계열) | 멤버당 무료 2명 한도 내 연결 |
| 유료 가디언 | `가디언+` | 강조 컬러 (골드 / 프리미엄) | 3번째~5번째 가디언, 추가 과금 |

### 4.2 배지 표시 위치

- **프로필 사진 우하단**: 원형 배지 아이콘 (지름 20dp 이하)
- **닉네임 옆**: 텍스트 배지 태그 형태 (닉네임 오른쪽에 인라인 배치)

두 위치 중 구현 우선순위에 따라 하나를 채택하되, 프로필 사진 우하단을 기본으로 한다.

### 4.3 배지 표시 대상 및 범위

| 화면 유형 | 가디언 배지 표시 여부 | 조건 |
|----------|:------------------:|------|
| 편집 모드 (본인) | ✅ | 본인이 가디언 역할일 경우 |
| 조회 모드 (타인 → 가디언) | ✅ | 조회 대상이 가디언이며, 조회자와 연결된 경우 |
| 조회 모드 (비연결 가디언) | ❌ | 배지 미표시 (닉네임+사진만 공개) |
| 멤버탭 멤버 카드 (#19) | ✅ | 멤버탭 원칙과 동일 규칙 적용 |

배지는 **가디언 본인과 연결된 멤버에게만** 표시된다. 다른 크루나 비연결 가디언에게는 과금 상태가 노출되지 않는다.

### 4.4 배지 실시간 반영

- 가디언 과금 상태 변경 시(결제 완료 / 해제 / 환불) **FCM 푸시 기반**으로 실시간 배지 갱신.
- FCM 이벤트 타입: `GUARDIAN_BADGE_UPDATED`
- 클라이언트는 이벤트 수신 즉시 프로필 캐시를 무효화하고 서버에서 최신 배지 상태를 재조회한다.

## 5. 역할별 프로필 정보 필터링 매트릭스

여행 참여 중 타인의 프로필 조회 시, 조회자의 역할에 따라 표시 항목이 다르다.

### 5.1 필터링 매트릭스

| 조회자 역할 | 조회 대상 | 닉네임 | 프로필 사진 | 여행 상태 | 긴급 연락처 | 마지막 위치 | 배정된 조 |
|-----------|---------|:-----:|:----------:|:--------:|:---------:|:---------:|:-------:|
| 캡틴 | 크루 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 캡틴 | 크루장 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 캡틴 | 연결 가디언 | ✅ | ✅ | ✅ | ❌ | ✅ | — |
| 크루장 | 소속 조원(크루) | ✅ | ✅ | ✅ | ❌ | ✅ | ✅ |
| 크루장 | 타 조 크루 | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ |
| 크루 | 크루 | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ |
| 크루 | 크루장 | ✅ | ✅ | ✅ | ❌ | ❌ | ✅ |
| 크루 | 캡틴 | ✅ | ✅ | ✅ | ❌ | ❌ | — |
| 연결 가디언 | 연결 멤버 | ✅ | ✅ | ✅ | ❌ | ✅ | ❌ |
| 비연결 가디언 | 타 멤버 | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |

> 범례: ✅ 표시됨 / ❌ 비공개 / — 해당 없음

### 5.2 필터링 적용 원칙

- **서버 사이드 필터링**: 역할별 필터링은 반드시 서버 API에서 처리하며, 클라이언트에서 임의로 필드를 숨기는 방식에 의존하지 않는다.
- **최소 공개 기본값**: 필터링 규칙이 불명확한 경우, 닉네임+프로필 사진만 표시(최소 공개)를 기본값으로 적용한다.
- **여행 외 상황**: 여행에 참여하지 않은 상태에서 타인 프로필 조회 시, 닉네임+프로필 사진만 표시한다.

## 6. 프라이버시 등급별 프로필 노출 차이

비즈니스 원칙 §04의 프라이버시 등급에 따라 프로필 내 위치 관련 정보 노출 범위가 달라진다.

### 6.1 등급별 노출 정책

| 프라이버시 등급 | 닉네임·사진 | 여행 상태 | 실시간 위치 | 마지막 위치 | 긴급 연락처 (캡틴만) |
|-------------|:----------:|:--------:|:---------:|:---------:|:-----------------:|
| 안전최우선 (safety_first) | ✅ | ✅ | ✅ (캡틴·가디언·크루장) | ✅ | ✅ |
| 표준 (standard) | ✅ | ✅ | 일정 공유 시간대만 | 공유 시간대 이후 | ✅ |
| 프라이버시우선 (privacy_first) | ✅ | ✅ | ❌ | ❌ | ✅ |

### 6.2 등급 변경 UI

- 프로필 편집 모드에서 직접 프라이버시 등급을 변경할 수 없다.
- 프라이버시 등급 변경은 **설정 메뉴(#15) → 위치 공유 설정** 경로를 통해서만 가능하다.
- 프로필 편집 화면에는 현재 프라이버시 등급을 읽기 전용으로 표시하고, 변경 경로 링크를 제공한다.

### 6.3 미성년자 강제 등급

비즈니스 원칙 §10.2에 따라 미성년자가 포함된 여행은 `safety_first` 등급이 강제 적용된다.

- 미성년자 크루의 프로필 조회 시, 실시간 위치가 캡틴·크루장·연결 가디언에게 항상 표시된다.
- 미성년자 본인이 등급을 변경하려고 시도해도 서버에서 차단하며, "현재 여행에서는 변경이 불가합니다" 안내를 표시한다.

## 7. 프로필 사진 규칙

### 7.1 업로드 제한

| 항목 | 규격 |
|------|------|
| 최대 파일 크기 | 5MB |
| 허용 포맷 | JPG, PNG, GIF (애니메이션 GIF 허용, 단 정지 이미지로 처리) |
| 최소 해상도 | 100 × 100px |
| 권장 해상도 | 400 × 400px (정방형) |
| 저장 포맷 | 서버 저장 시 WebP 변환 (원본 삭제) |

### 7.2 부적절 이미지 필터링

- 업로드 즉시 AI 자동 검수를 수행한다 (폭력, 성인물, 개인정보 노출 이미지 탐지).
- 검수 통과 전까지 기존 사진 또는 기본 아바타를 임시 표시한다.
- 검수 실패 시 "이미지를 사용할 수 없습니다" 안내와 함께 업로드를 취소한다.
- 검수 소요 시간: 최대 10초 (10초 초과 시 수동 검토 대기열로 이관).

### 7.3 기본 아바타 시스템

- 프로필 사진 미등록 또는 삭제 시 기본 아바타가 자동 적용된다.
- 아바타 테마: SafeTrip 여행 테마 아바타 (비행기, 캠핑, 산, 도시 등 10종 이상 제공).
- 사용자는 편집 모드에서 아바타를 자유롭게 선택·변경할 수 있다.
- 아바타 선택 시 별도 파일 업로드 없이 서버에 아바타 ID만 저장한다 (`avatar_id`).

### 7.4 미성년자 프로필 사진 제한

비즈니스 원칙 §10 권장 사항에 따라:

- 미성년자 계정의 실제 사진 등록을 시스템 레벨에서 강제 차단하지 않으나, 온보딩 시 아바타 사용을 **강력 권장**한다.
- 온보딩 화면에서 미성년자 계정은 "개인 안전을 위해 실제 사진 대신 아바타를 권장합니다" 안내 메시지를 표시한다.
- 아바타 선택 화면이 기본으로 표시되고, 실제 사진 업로드는 하단 옵션으로 배치한다.

## 8. 긴급 연락처 관리

### 8.1 접근 권한

| 역할 | 조회 권한 | 편집 권한 |
|------|:--------:|:--------:|
| 본인 (편집 모드) | ✅ | ✅ |
| 캡틴 (조회 모드) | ✅ | ❌ |
| 크루장 | ❌ | ❌ |
| 크루 | ❌ | ❌ |
| 가디언 | ❌ | ❌ |

긴급 연락처는 **본인 편집 모드와 캡틴 조회 모드에서만** 접근 가능하며, 그 외 역할에게는 완전히 비공개이다.

### 8.2 긴급 연락처와 SOS 연동

- SOS 발송 시 등록된 긴급 연락처에 자동으로 SMS/알림이 전송된다 (SOS 원칙 #13 참조).
- 긴급 연락처가 미등록된 상태에서 SOS를 발송하면, "긴급 연락처가 없습니다. 등록하시겠습니까?" 안내 팝업을 표시한다.
- 긴급 연락처는 최대 2명까지 등록 가능하다 (기본 1명, 추가 1명).

### 8.3 미성년자 긴급 연락처 필수 정책

- 미성년자 계정은 온보딩 3단계에서 긴급 연락처 입력이 **필수**이다.
- 온보딩 건너뛰기 버튼 미표시, 빈칸 제출 시 "미성년자는 긴급 연락처 등록이 필요합니다" 오류 표시.
- 이미 온보딩을 완료한 미성년자 계정이 긴급 연락처를 삭제하려고 시도할 경우, 서버에서 차단하고 "미성년자 계정의 긴급 연락처는 삭제할 수 없습니다" 안내를 반환한다.

## 9. 계정 삭제 흐름

비즈니스 원칙 §14.4 기준으로 계정 삭제 흐름을 정의한다. 본 문서는 비즈니스 원칙 §14.4의 **7일 유예 기간** 정책을 준수한다.

### 9.1 계정 삭제 진입 경로

설정 메뉴(#15) → 계정 관리 → 계정 삭제 또는 프로필 편집 모드 → 계정 관리 섹션 → 계정 삭제

### 9.2 삭제 단계별 흐름

#### 1단계: 진행 중인 여행 확인

- 서버에서 해당 계정이 참여 중인 `status = active` 여행을 조회한다.
- **여행 참여 중인 경우**: "현재 참여 중인 여행이 있습니다. 여행 종료 또는 탈퇴 후 계정 삭제가 가능합니다" 안내 표시, 삭제 프로세스 차단.
  - 캡틴인 경우: "리더십을 위임하거나 여행을 종료한 후 삭제해 주세요" 추가 안내.
- **여행 미참여 상태**: 2단계로 진행.

#### 2단계: 삭제 확인 (2중 인증)

```
[삭제 확인 화면]

"정말로 계정을 삭제하시겠습니까?"
"삭제 후 7일 유예 기간이 주어지며, 유예 기간 내 재로그인 시 삭제가 취소됩니다."

삭제 사유 선택 (필수):
  ○ 앱을 더 이상 사용하지 않음
  ○ 개인정보 보호 우려
  ○ 서비스 불만족
  ○ 다른 계정으로 전환
  ○ 기타 (직접 입력)

비밀번호 재확인: [________]

[취소]   [계정 삭제 요청]
```

#### 3단계: 유예 기간 시작

- 삭제 요청 확인 후 `tb_user.deletion_requested_at` 에 현재 시각을 기록한다.
- **7일 유예 기간** 동안 계정은 정상 로그인 가능 상태를 유지한다.
- 유예 기간 내 재로그인 시 "계정 삭제 취소 여부" 확인 팝업 표시 → 취소 선택 시 `deletion_requested_at = NULL`로 초기화.
- 매일 1회 이메일/앱 알림으로 "D-N일 후 계정이 삭제됩니다" 알림 발송.

#### 4단계: 유예 기간 만료 후 데이터 처리

`deletion_requested_at` 기준 7일 경과 후 서버 스케줄러가 자동 처리:

| 데이터 유형 | 처리 방식 | 근거 |
|-----------|----------|------|
| 프로필 정보 (이름, 이미지) | 즉시 물리 삭제 | §14.4 — 즉시 삭제 |
| 기기 토큰 (FCM) | 즉시 물리 삭제 | §14.4 — 즉시 삭제 |
| 긴급 연락처 | 즉시 물리 삭제 | §14.4 — 즉시 삭제 |
| 채팅 메시지 닉네임 | "탈퇴한 사용자"로 익명화 | §14.4 — 익명화 후 보관 |
| 위치 데이터·이동 기록 | 익명화 후 보관 (법적 보관 의무 기간) | §14.4 — 익명화 후 보관 |
| 이벤트 로그 | 익명화 후 보관 | §14.4 — 익명화 후 보관 |
| SOS 발송/수신 기록 | 영구 보관 (익명화) | §13 데이터 생명주기 |
| 결제·과금 기록 | 5년 보존 | 전자상거래법 제6조 |
| 약관 동의 이력 | 5년 보존 | 전자상거래법 |

#### 5단계: 삭제 완료 알림

- 계정 삭제 완료 후 등록된 이메일로 "SafeTrip 계정 삭제가 완료되었습니다" 이메일 발송.
- 이미 계정이 삭제된 이후에는 앱 재로그인 시 "존재하지 않는 계정입니다" 오류를 반환하고 로그인을 차단한다.

## 10. 에러 및 엣지케이스 처리

| 상황 | 처리 규칙 |

| AI 검수 실패 (부적절 이미지) | "이미지를 사용할 수 없습니다. 다른 이미지를 선택해 주세요" |
| AI 검수 타임아웃 (10초 초과) | 수동 검토 대기열 이관 + "검토 중입니다. 잠시 후 반영됩니다" 안내 |
| 지원하지 않는 포맷 선택 | "JPG, PNG, GIF 형식만 지원합니다" |

### 10.2 닉네임 관련

| 상황 | 처리 규칙 |
|------|----------|

| 닉네임 길이 위반 | "닉네임은 2자 이상 20자 이하로 입력해 주세요" |
| 금지어 포함 | "사용할 수 없는 단어가 포함되어 있습니다" |
| 특수문자 포함 | "특수문자는 사용할 수 없습니다 (밑줄·점 제외)" |

### 10.3 계정 삭제 관련

| 상황 | 처리 규칙 |
|------|----------|
| 여행 참여 중 삭제 시도 | 삭제 차단 + "여행 종료 또는 탈퇴 후 가능합니다" 안내 |
| 캡틴 상태에서 삭제 시도 | "리더십 위임 후 삭제 가능합니다" 안내 + 위임 화면 이동 버튼 |
| 비밀번호 재확인 실패 | "비밀번호가 일치하지 않습니다" (3회 연속 실패 시 10분 대기) |
| 유예 기간 중 재로그인 | 삭제 취소 확인 팝업 표시 |
| 이미 삭제된 계정으로 로그인 시도 | "존재하지 않는 계정입니다" HTTP 404 반환 |

### 10.4 프로필 조회 관련

| 상황 | 처리 규칙 |
|------|----------|
| 존재하지 않는 사용자 프로필 조회 | "존재하지 않는 사용자입니다" HTTP 404 반환 |
| 탈퇴한 사용자 프로필 조회 | "탈퇴한 사용자입니다" 표시 (닉네임: "탈퇴한 사용자", 사진: 기본 아바타) |
| 프로필 로딩 실패 | 스켈레톤 UI 유지 + "프로필을 불러올 수 없습니다" 안내 |
| 비연결 가디언이 상세 정보 접근 시도 | 서버에서 필터링, 기본 정보만 반환 (HTTP 200 + 제한된 필드) |

### 10.5 긴급 연락처 관련

| 상황 | 처리 규칙 |
|------|----------|
| 잘못된 전화번호 형식 | "올바른 전화번호 형식을 입력해 주세요 (예: +82-10-1234-5678)" |
| 미성년자 계정 긴급 연락처 삭제 시도 | 서버 차단 + "미성년자 계정의 긴급 연락처는 삭제할 수 없습니다" |
| 긴급 연락처 최대 2명 초과 | "긴급 연락처는 최대 2명까지 등록 가능합니다" |

## 11. DB 스키마

### 11.1 tb_user_profile (프로필 정보)

```sql
CREATE TABLE tb_user_profile (
  profile_id        BIGSERIAL PRIMARY KEY,
  user_id           BIGINT NOT NULL UNIQUE REFERENCES tb_user(user_id),

  -- 기본 프로필
  nickname          VARCHAR(20) NOT NULL,         -- 닉네임 (2~20자)
  profile_image_url TEXT,                          -- 프로필 사진 URL (WebP)
  avatar_id         VARCHAR(30),                   -- 기본 아바타 ID (사진 미등록 시)

  -- 개인정보
  birth_date        DATE,                          -- 생년월일 (미성년자 판별용)
  is_minor          BOOLEAN NOT NULL DEFAULT FALSE, -- 미성년자 여부

  -- 프라이버시
  privacy_level     VARCHAR(20) NOT NULL           -- 'safety_first' | 'standard' | 'privacy_first'
                    DEFAULT 'standard'
                    CHECK (privacy_level IN ('safety_first', 'standard', 'privacy_first')),

  -- 이미지 검수 상태
  image_review_status VARCHAR(20) DEFAULT 'none'  -- 'none' | 'pending' | 'approved' | 'rejected'
                      CHECK (image_review_status IN ('none', 'pending', 'approved', 'rejected')),

  -- 온보딩
  onboarding_completed BOOLEAN NOT NULL DEFAULT FALSE, -- 온보딩 완료 여부

  -- 공통 컬럼
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at        TIMESTAMPTZ                    -- 소프트 삭제
);

-- 인덱스
CREATE UNIQUE INDEX idx_profile_nickname ON tb_user_profile(nickname) WHERE deleted_at IS NULL;
CREATE INDEX idx_profile_user_id ON tb_user_profile(user_id);
```

### 11.2 tb_emergency_contact (긴급 연락처)

```sql
CREATE TABLE tb_emergency_contact (
  contact_id        BIGSERIAL PRIMARY KEY,
  user_id           BIGINT NOT NULL REFERENCES tb_user(user_id),

  -- 연락처 정보
  contact_name      VARCHAR(30) NOT NULL,          -- 긴급 연락처 이름
  contact_phone     VARCHAR(20) NOT NULL,          -- 전화번호 (국제 형식)
  contact_order     SMALLINT NOT NULL DEFAULT 1    -- 순서 (1: 주, 2: 부)
                    CHECK (contact_order IN (1, 2)),
  relationship      VARCHAR(20),                   -- 관계 (부모, 배우자, 친구 등, 선택)

  -- 공통 컬럼
  created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at        TIMESTAMPTZ                    -- 소프트 삭제
);

-- 인덱스
CREATE INDEX idx_emergency_contact_user_id ON tb_emergency_contact(user_id) WHERE deleted_at IS NULL;

-- 제약: 사용자당 최대 2개 (애플리케이션 레벨 검증, DB UNIQUE 인덱스 보조)
CREATE UNIQUE INDEX idx_emergency_contact_user_order
  ON tb_emergency_contact(user_id, contact_order) WHERE deleted_at IS NULL;
```

## 12. 오프라인 대응

### 12.1 프로필 로컬 캐시

| 데이터 | 캐시 방식 | 유효 기간 |
|--------|----------|----------|
| 본인 프로필 | 로컬 디바이스 저장소 (암호화) | 무기한 (서버 동기화 시 갱신) |
| 타인 프로필 (조회 이력) | 메모리 캐시 | 마지막 조회 기준 1시간 |
| 가디언 배지 상태 | 로컬 디바이스 저장소 | FCM 수신 시 갱신 |

오프라인 상태에서 타인 프로필 조회 시:
- 캐시된 프로필이 있으면 캐시 표시 + "마지막 업데이트: N분 전" 표시.
- 캐시된 프로필이 없으면 스켈레톤 UI + "오프라인 상태에서는 프로필을 불러올 수 없습니다" 안내.

### 12.2 오프라인 상태의 편집 처리

오프라인 중 프로필 편집을 시도할 경우:

```
[오프라인 편집 흐름]

사용자가 닉네임/긴급 연락처 변경 입력
    ↓
"현재 오프라인 상태입니다. 변경 사항을 임시 저장합니다." 안내 표시
    ↓
로컬 임시 저장 (draft 상태)
    ↓
네트워크 복귀 감지 (ConnectivityStream)
    ↓
"저장되지 않은 변경 사항이 있습니다. 지금 동기화하시겠습니까?" 팝업
    ↓
사용자 확인 → 서버 동기화 → 성공 시 draft 삭제
```

### 12.3 오프라인 제한 항목

오프라인 상태에서 허용되지 않는 프로필 작업:

| 작업 | 오프라인 허용 | 비고 |
|------|:-----------:|------|
| 닉네임 변경 | 임시 저장만 허용 | 네트워크 복귀 후 동기화 |
| 프로필 사진 업로드 | ❌ | AI 검수 서버 필요 |
| 긴급 연락처 수정 | 임시 저장만 허용 | 네트워크 복귀 후 동기화 |
| 계정 삭제 요청 | ❌ | 서버 상태 확인 필수 |
| 타인 프로필 조회 | 캐시만 표시 | — |
