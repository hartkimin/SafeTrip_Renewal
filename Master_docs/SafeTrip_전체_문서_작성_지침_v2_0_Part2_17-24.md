# SafeTrip 전체 원칙 문서 작성 지침서 v2.0 (2/2)

> **문서 목적**: SafeTrip 프로젝트의 35개 원칙 문서 중 #17~#24(Tier 3 기능 원칙)를 최신화(v5.1 기준)하기 위한 상세 작성 프롬프트, 참조 관계, 필수 포함 항목, 정합성 검증 방법을 정의한다. 본 문서는 1/2 문서의 공통 규칙을 포함하며, #17~#24 문서에 대해서는 #1~#16 수준의 상세 프롬프트를 제공한다.
>
> **기준 문서**: SafeTrip_마스터_원칙_거버넌스_v2.0, SafeTrip_비즈니스_원칙_v5.1
>
> **관련 문서**: SafeTrip_전체_문서_작성_지침_v2.0 (1/2) — 공통 규칙 + #1~#16 상세 프롬프트
>
> **v1.0 → v2.0 변경 사유**: 비즈니스 원칙 v5.0→v5.1 반영, 기존 문서 현황 기반 갭 분석 반영, 정합성 검증 강화

## 공통 작성 규칙 (1/2 문서와 동일)

### 모든 문서에 반드시 포함해야 하는 구조

```
1. 문서 헤더 (메타데이터 테이블)
   - 문서 ID, 계층, 등급, 버전, 작성일
   - 기준 문서(상위 문서) — 반드시 v5.1 참조 명시
   - 관련 법령 (해당 시)
   - 후행 문서 목록
   - 검토 주기, 담당 부서
2. 문서 목적 및 적용 범위
3. 본문 (문서별 고유 내용 — 아래 개별 지침 참조)
4. 역할별 접근 권한 매트릭스 (캡틴/크루장/크루/가디언)
5. 프라이버시 등급별 동작 차이 (안전 최우선/표준/프라이버시 우선)
6. 에러 및 엣지케이스 처리
7. DB 스키마 (해당 시) — v5.1 부록 C 스키마 반영 필수
8. 오프라인 대응 (해당 시)
9. 구현 우선순위 (P0~P3) 및 Phase 배치
10. 검증 체크리스트 (법적 + 기능 + 거버넌스 12항목)
11. 기존 문서 대비 변경·확장 사항 (변경 이력)
12. 부록 (관련 문서 참조 테이블)
```

| 항목 | 내용 |
|------|------|
| **계층** | Tier 3 (기능 원칙) |

#### 작성 프롬프트

```
[역할] SafeTrip의 지도 UI/UX 전문가 겸 위치 기반 안전 시스템 설계자

[목적] SafeTrip 메인 화면의 지도 영역이 제공하는 고유 기능 원칙
(7단계 레이어 시스템, 마커 표시, 자동 전환, 추가 기능 제안)을
v5.1 기준으로 갱신하라.

[기준 문서] SafeTrip_비즈니스_원칙_v5.1

[갱신 필수 사항]
1. 기준 문서 참조: v3.0 → v5.1 갱신 (문서 전체에서 v3.0 참조를 v5.1로 변경)
2. 지도 설계 철학 5대 원칙(G1~G5) 유지 및 v5.1 정합성 확인
   - G1: 지도 항상 갱신 / G2: 레이어 분리 / G3: 시간축 / G4: 역할별 정보 밀도 / G5: SOS 0초 진입
3. 7단계 레이어 시스템 (Layer 0~6) 정의 유지
   - Layer 6(긴급 오버레이) ~ Layer 0(지도 타일)
   - 레이어 토글 UI 확장 구성 확인
4. 지도 상태 자동 전환 규칙 5가지 (SOS → 지오펜스 → 멤버 오프라인 → 일정 → 앱 복귀)
   - 충돌 해결: 우선순위 P0~P3 기반
5. 지도 카메라 기본 위치 규칙 (여행 상태별: active/planning/미참여)
   - 가디언 화면: 연결 멤버만 포함하는 바운딩 박스 — §03 가디언 시스템 참조
6. 마커 클러스터링 규칙 (줌 레벨 15이상/12~14/11이하)
7. 터치 인터랙션 우선순위 (SOS > 멤버 > 이벤트 > 일정 > 지오펜스 > 빈영역)
8. ★ 프라이버시 등급별 마커 표시 차이 — §04 반영
   - 안전 최우선: 전체 마커 항상 표시
   - 표준: 24시간 공유 기반 마커 표시
   - 프라이버시 우선: 일정 연동 시간에만 마커 표시, 비공유 시간 마커 숨김
9. 추가 기능 7종의 Phase 배치 유지
   - Phase 1: 일정 미리보기, 안전시설 오버레이, 오프라인 지도
   - Phase 2: 멤버 분산도, 발자취 타임라인, 음성메모 핀
   - Phase 3: 안전 히트맵
10. 오프라인 지도 사전 다운로드 규칙 (다운로드 범위, 줌 레벨, 용량)
11. ★ 가디언 무료/유료에 따른 지도 접근 범위 차이 확인 — §03.1, §09.3
    - 무료 가디언: 연결 멤버 위치 마커만 (제한적 뷰)
    - 유료 가디언: 연결 멤버 + 그룹 전체 일정 마커 (확장 뷰) — 과금 분기 명시
12. 검증 체크리스트 4개 카테고리(Critical/High) 유지
    - 고유 원칙 5건, 일정 미리보기 5건, 레이어 시스템 4건, 자동 전환 3건
13. 기존 문서 대비 변경/확장 사항 테이블 갱신
14. DB 스키마 필요 시 tb_geofence, tb_location 참조 — v5.1 부록 C 반영
15. 역할별 접근 권한 매트릭스: 기능 7종 × 4역할 전수 정의

[현재 문서 갭 분석]
- v3.0 참조 → v5.1 미반영 (전 섹션)
- 프라이버시 등급별 마커 표시가 화면 구성 원칙에만 정의, 지도 문서에 미포함
- 가디언 뷰(무료/유료 차이) 미정의
- DB 스키마 섹션 없음 → 지오펜스, 마커 관련 DB 참조 추가 필요
- 역할별 접근 매트릭스가 기능 제안별로 분산 → 통합 매트릭스 필요
```

#### 작성 프롬프트

```
[역할] SafeTrip의 여행 일정 시스템 설계자 겸 UI/UX 전문가

[목적] SafeTrip 메인 화면 바텀시트의 일정 탭(설계 원칙, 화면 구성, 역할별 기능,
데이터 규칙, 추가 제안 기능)을 v5.1 기준으로 갱신하라.

[기준 문서] SafeTrip_비즈니스_원칙_v5.1

[갱신 필수 사항]
1. 기준 문서 참조: v3.0 → v5.1 갱신 (문서 전체에서 v3.0 참조를 v5.1로 변경)
2. 일정 탭 설계 철학 유지 (시간 중심, 위치 공유 일체화, 역할 기반, 지도 연결, 오프라인)
3. 일정 탭의 이중 역할 유지
   - 역할 A: 여행 스케줄 관리 (CRUD)
   - 역할 B: 위치 공유 스케줄 시각화 (schedule_linked)
4. ★ 여행 기간 15일 제한 반영 — §02.3
   - 날짜 타임라인에 최대 15일까지만 표시
   - 일정 추가 시 여행 기간(15일) 초과 날짜 선택 차단
   - 기존 여행 기간 변경 시 15일 초과 경고 및 차단
   - DB 스키마: tb_trip CHECK (end_date - start_date <= 15)
5. 프라이버시 등급별 일정 탭 동작 차이 3등급 전수 정의
   - 안전 최우선/표준: 공유 관련 UI 미표시
   - 프라이버시 우선: 일정 연동 배너, 공유 시간, 타임라인 바 표시
6. 바텀시트 상태별(hidden/mini/half/expanded/fullscreen) 일정 탭 표시 규칙
7. 화면 구성 상세 5개 영역 (A: 연동 배너, B: 날짜 타임라인, C: 공유 타임라인 바, D: 카드 목록, E: 추가 버튼)
8. 역할별 기능 권한 매트릭스 (15개 기능 × 4역할)
   - 가디언: 상태 탭에서 간접 열람만
   - 크루: 편집 불가, 열람/지도 이동/내보내기만
```

```
    - 초대코드에 모델 유형 태그 (model_type: 'direct' / 'system')
11. ★ B2B 역할 매핑 자동 적용 — §01.4
    - 코드 생성 시 B2B 고객 유형에 따라 역할명 자동 변환

    - 추가: CSV 형식 오류, B2B 계약 만료, 기업 가디언 권한 초과
15. 보안 원칙 (브루트포스 방지, 코드 노출 최소화, 전송 암호화)

18. DB 스키마: TB_INVITE_CODE + ★ model_type 컬럼 추가 (v5.1)

[현재 문서 갭 분석]
```

## 부록 C: 프롬프트 사용법

### 각 문서 업데이트 시 프롬프트 사용 순서

```
1. 해당 문서의 "작성 프롬프트" 섹션을 복사한다.
2. 기존 문서 원본을 함께 첨부한다.
3. 다음 형식으로 AI에게 요청한다:

   "다음 기준에 따라 [문서명]을 v{X.Y} → v{X.Y+1}로 갱신하라.

   [작성 프롬프트 붙여넣기]

   [기존 문서 원본 첨부]

   갱신 시 주의사항:
   - 기준 문서를 v5.1로 갱신
   - 공통 갱신 항목(여행 기간, 가디언 과금, DB 스키마) 확인
   - 거버넌스 12항목 체크리스트 충족 확인
   - 정합성 검증 방법에 따라 자가 검증 수행
   - 변경 이력을 부록에 기록"

4. 갱신 완료 후 "정합성 검증 방법" 테이블로 교차 검증한다.
5. 교차 참조 문서가 있는 경우 해당 문서도 함께 확인한다.
```
